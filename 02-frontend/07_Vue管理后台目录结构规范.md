# Vue 管理后台目录结构规范

## 目标

- 规范 Vue 管理后台项目的目录组织方式、模块边界和文件放置原则。
- 降低页面、组件、composables、请求层、权限逻辑和领域模块长期混杂的风险。
- 让新项目能够快速建立清晰结构，让已有项目能够判断哪些目录应继续沿用、哪些应调整。

## 适用范围

- 适用于 Vue 管理后台项目的目录设计、模块拆分和结构优化。
- 涉及技术选型时，应结合 [06_Vue管理后台技术选型规范](06_Vue管理后台技术选型规范.md) 使用。
- 涉及页面生成、组件使用和交互要求时，应结合 [04_AntDesignVue管理后台前端规范](04_AntDesignVue管理后台前端规范.md) 或 [05_ElementPlus管理后台前端规范](05_ElementPlus管理后台前端规范.md) 使用。

## 核心原则

- 目录按职责和业务边界组织，不按个人习惯或临时需求随意堆放。
- 路由级页面、页面私有实现、共享组件、领域模块、数据访问、权限逻辑、工具函数和类型定义应清晰分层。
- 页面私有实现优先就近放置，只有在跨页面复用后才上提到公共目录。
- 同一类基础设施只保留一套主入口，例如请求层、权限组件、表格封装、表单封装和主题配置。
- 结构设计优先服务长期维护，不为了短期开发方便把多类职责压进同一目录。

## 新项目目录设计要求

- 新项目开始前，应先确定目录组织策略，再进入大规模页面开发。
- 如果用户未指定目录结构，应先给出候选方案并让用户确认，不替用户拍板唯一模板。
- 新项目应至少明确以下边界：路由入口、页面目录、共享组件目录、领域模块目录、请求层目录、权限能力目录、样式与主题目录。
- 目录层级应控制在可读范围内，避免刚起步就形成过深嵌套。

## 已有项目结构判断要求

- 已有项目优先检查当前目录结构、路由组织方式、共享组件放置方式、请求层入口和权限能力位置。
- 如当前结构已能支撑新增页面开发，应优先沿用现状，不因理想模板强行整体搬迁。
- 只有在目录职责明显混杂、同类能力重复分散、结构已影响维护效率时，才讨论调整。
- 讨论调整时，应优先做局部收敛和渐进迁移，而不是一次性整体翻目录。

## 推荐目录边界

### 路由与应用壳层

- 路由入口、全局布局、菜单注册、权限路由守卫和应用初始化逻辑应集中在应用壳层目录。
- 这部分目录负责应用启动、页面挂载和全局能力编排，不承载具体业务页面实现。

### 页面目录

- 页面目录按业务域、菜单域或路由域组织，不按“组件类型”拆成难以理解的碎片目录。
- 每个路由级页面只承载页面编排、页面级状态和页面交互，不直接堆积大量通用逻辑。
- 页面私有组件、页面私有 composables、页面局部类型和页面局部样式应优先与页面同目录放置。

### 共享组件目录

- 跨页面复用的表格封装、查询栏、弹窗表单、状态标签、权限按钮等组件应集中沉淀到共享组件目录。
- 共享组件应以职责命名，不按作者、项目阶段或临时用途命名。
- 共享组件目录只放可复用能力，不放某个单页面专用的临时实现。

### 领域模块目录

- 与具体业务域绑定的数据访问、业务规则、数据转换、常量枚举和复用逻辑，应优先沉淀到对应领域模块目录。
- 同一业务域的接口定义、数据映射和领域级 composables 应尽量就近组织，减少跨目录跳转成本。
- 领域模块目录用于承载“业务共性”，不用于替代页面目录。

### 请求层与数据访问目录

- 请求客户端、鉴权处理、错误拦截、统一响应转换、上传下载能力和通用数据访问封装应集中在请求层目录。
- 不同页面不应各自维护一套独立请求工具或重复封装同类接口调用逻辑。
- 当存在领域级数据访问封装时，应明确区分“底层请求能力”和“业务数据访问能力”的边界。

### 权限能力目录

- 路由准入、菜单权限、按钮权限、权限组件和权限判断工具应集中在权限能力目录或统一模块中。
- 不应在页面目录中散落大量权限工具副本或重复权限码判断逻辑。

### composables、utils、types、constants 目录

- 仅跨页面复用的 composables、工具函数、类型定义和常量配置，才放入公共目录。
- 页面或领域私有的 composables、types、constants 应优先就近放置，避免所有内容都堆进公共目录。
- 公共目录内的文件命名和职责要稳定，避免成为“杂物间”。

## 推荐目录模板

- 可采用“应用壳层 + 页面目录 + 共享组件 + 领域模块 + 请求层 + 权限能力 + 公共基础能力”的思路组织。
- 一个可接受的模板通常包含以下层次：应用入口与路由、业务页面、共享组件、领域模块、数据访问、权限能力、公共 composables、公共 utils、公共 types、主题样式。
- 如果项目规模较小，可以适当合并部分目录，但仍应保持职责边界清晰。
- 如果项目规模较大，可以在业务域下继续细分页面、领域逻辑和私有组件，但不要无限制纵深嵌套。

## 结构失衡信号

- 一个目录同时混放页面、组件、接口、权限、工具和测试数据。
- 同类 composables、类型、常量和接口定义分散在多个不一致位置。
- 多个页面各自复制一套请求层、表格封装或权限判断工具。
- 页面文件过大，内部既有请求调用、又有数据转换、又有大量业务规则和子组件实现。
- 公共目录中堆满只被单个页面使用的私有实现。
- 临时代码、旧版本页面、废弃目录长期留在正式结构中。

## 调整策略

- 优先先收敛重复基础设施，再处理页面私有实现迁移。
- 优先局部移动和逐步重命名，不一次性大范围搬迁所有目录。
- 每次调整都应同步更新引用和说明，避免旧目录名和新目录名并存。
- 如结构调整涉及多人协作区域，应先明确迁移范围和落地顺序。

## 提交前自检

- 是否明确区分了页面目录、共享组件目录、领域模块目录和请求层目录的职责。
- 是否存在页面私有实现被过早放入公共目录，或公共能力仍滞留在单页面目录中的情况。
- 是否存在重复请求层、重复权限工具、重复表格封装或重复表单封装。
- 是否有目录命名模糊、用途不清、历史遗留未清理的问题。
- 是否在已有项目中做了不必要的大范围结构迁移。
