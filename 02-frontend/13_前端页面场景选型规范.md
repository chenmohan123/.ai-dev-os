# 前端页面场景选型规范

## 目标

- 为前端页面先做场景分流，明确什么时候按管理后台理解，什么时候按 AI 应用前端理解，什么时候按官网或营销落地页理解，什么时候按移动 Web 页面理解。
- 避免页面还没定性，就直接套管理后台模板、聊天模板或官网区块模板。
- 让后续进入具体技术栈规范、页面模板规范和目录规范时有统一前置判断。

## 适用范围

- 适用于页面需求分析、页面模板选择、前端方案确认和实现前澄清。
- 适用于新项目场景判断，也适用于已有项目中的新页面归类。
- 不替代具体技术栈规范、组件规范或页面模板规范；只负责先判断页面属于哪一类。

## 使用原则

- 先判断页面的主要目标，再判断页面的典型结构，最后才选择组件库、模板和目录落点。
- 一个页面只应有一个主场景归类；如确实跨场景，应先确定主场景，再把次级能力作为附属区域嵌入。
- 场景分流优先看页面主要任务，不只看是否使用了某个组件库或某种布局。

## 四类主场景

### 管理后台页面

- 主要目标：查询、录入、编辑、审批、配置、监控、运营、权限控制。
- 典型结构：搜索区、表格区、表单区、详情区、指标区、弹窗或抽屉工作流。
- 典型特征：字段多、状态多、操作多、信息密度高、强调效率和可维护性。
- 进入路径：先读 [02_管理后台页面通用规范](02_管理后台页面通用规范.md)，再按需读 [03_管理后台页面模板规范](03_管理后台页面模板规范.md) 与对应技术栈分册。

### AI 应用前端

- 主要目标：对话、提问、协作生成、Agent 执行、工具调用、消息流反馈、长任务执行。
- 典型结构：会话区、输入区、执行反馈区、上下文区、结果区、会话列表区。
- 典型特征：多轮交互、流式输出、任务执行状态、引用内容展示、工具反馈链路明显。
- 进入路径：先读 [11_AntDesignX智能应用前端规范](11_AntDesignX智能应用前端规范.md)，如需页面骨架再读 [14_智能应用与官网页面模板规范](14_智能应用与官网页面模板规范.md)。

### 官网或营销落地页

- 主要目标：品牌表达、产品介绍、能力展示、活动承接、转化引导。
- 典型结构：首屏、卖点区、能力区、背书区、案例区、FAQ、CTA 区、页脚。
- 典型特征：内容叙事明显、转化路径明确、视觉表达强于后台式信息密度。
- 进入路径：先读 [12_AntDesignLanding官网与落地页规范](12_AntDesignLanding官网与落地页规范.md)，如需页面骨架再读 [14_智能应用与官网页面模板规范](14_智能应用与官网页面模板规范.md)。

### 移动 Web 页面

- 主要目标：在手机浏览器或 WebView 中完成业务办理、待办处理、结果查看、扫码承接或轻量 AI 交互。
- 典型结构：标题区、卡片区、表单区、结果区、固定底部操作区或底部导航区。
- 典型特征：单手操作明显、一次性信息量受控、弱化复杂表格、强调任务闭环和反馈速度，通常以 H5 或 WebView 页面交付。
- 进入路径：先读 [15_移动端页面场景选型规范](15_移动端页面场景选型规范.md)，再按需读 [16_AntDesignMobile前端规范](16_AntDesignMobile前端规范.md) 和 [17_移动端页面模板规范](17_移动端页面模板规范.md)。

## 快速判断问题

- 这个页面的首要目标是让用户高效处理业务数据、与 AI 协作、理解产品并转化，还是在手机浏览器或 WebView 中快速完成一段业务链路。
- 页面主体是表格和表单、消息流和执行状态、内容区块和行动引导，还是卡片与固定底部操作。
- 用户在这个页面里主要是“做业务操作”“发起智能交互”“浏览内容并决定下一步行动”还是“在移动 Web 中快速完成任务”。
- 页面是否需要高密度字段、流式输出与消息反馈、内容叙事与品牌表达，还是需要单手操作、短链路和移动容器兼容。

## 常见混合场景处理

### 后台中的 AI 助手

- 主场景仍是管理后台页面。
- AI 助手应作为侧边面板、抽屉、辅助工作区或独立工作台嵌入，而不是把整个后台页面改造成聊天页。
- 进入路径：先读后台规范，再补读 [11_AntDesignX智能应用前端规范](11_AntDesignX智能应用前端规范.md)。

### 官网中的智能演示区

- 主场景仍是官网或营销落地页。
- 智能演示区只作为能力展示模块，不应让整页信息结构被聊天交互主导。
- 进入路径：先读官网规范，再按需补读 [11_AntDesignX智能应用前端规范](11_AntDesignX智能应用前端规范.md)。

### 后台能力的移动 Web 延伸页

- 主场景通常是移动 Web 页面，而不是管理后台页面。
- 不应把桌面后台布局整体压缩到手机端，应按移动任务链路重组信息和操作。
- 进入路径：先读 [15_移动端页面场景选型规范](15_移动端页面场景选型规范.md) 和 [16_AntDesignMobile前端规范](16_AntDesignMobile前端规范.md)，必要时补读后台规范。

### 平台型工作台

- 如果主体是运营、配置、审批、监控和数据操作，归为管理后台。
- 如果主体是多轮交互、任务编排、执行状态追踪和生成式工作流，归为 AI 应用前端。
- 如果主体是移动 Web 中的快速办理、待办处理、结果查看或扫码承接，归为移动 Web 页面。
- 不要因为页面里同时有侧栏、表单和消息区，就简单定义为两者各一半；必须先定主场景。

## 不建议的做法

- 看见 React + Ant Design 就默认按管理后台页面处理。
- 看见聊天框就把所有页面都按 AI 应用前端处理。
- 看见大横幅和卡片区就默认套官网模板，而不先判断转化目标和页面任务。
- 看见手机尺寸或 WebView 就直接套移动端组件库，而不先判断页面任务和交付形态。
- 一个页面同时套用后台模板、聊天模板和官网区块模板，导致结构失真。
- 把桌面后台、官网和移动 Web 页面做成同一套无主次的响应式拼装页。

## 输出要求

- 给出页面方案时，应明确写出主场景归类、判断依据、推荐进入的下一篇规范。
- 如果页面存在混合场景，应明确主场景是什么，次级场景以什么形式附着。
- 如果需求还不足以判断场景，应先向用户补问页面目标、主要用户动作和主体结构，而不是直接开始出页面代码。

## 提交前自检

- 是否先判断了页面主目标，而不是先选组件库。
- 是否明确了这是管理后台、AI 应用前端、官网落地页还是移动 Web 页面。
- 是否给出了后续应进入的模板规范和技术栈规范。
- 是否避免把混合场景页面做成无主次的拼装页面。
