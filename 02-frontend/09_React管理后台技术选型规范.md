# React 管理后台技术选型规范

## 目标

- 规范 React 管理后台在新项目和已有项目中的关键技术决策方式。
- 明确哪些技术点可以作为候选项让用户选择，哪些技术点应优先复用项目现状。
- 避免在页面生成或规范落地时擅自拍板请求层、状态管理、表单方案和表格体系。

## 适用范围

- 适用于 React 管理后台项目的技术选型、方案确认和实现前决策。
- 涉及 React + Ant Design 的页面与工程约束时，应结合 [08_AntDesignReact管理后台前端规范](08_AntDesignReact管理后台前端规范.md) 一并使用。

## 总体原则

- 新项目如未明确指定框架、组件库、请求层、缓存方案、状态管理或表单方案，应先列出候选并让用户确认，不替用户拍板。
- 已有项目如已存在稳定技术栈、目录结构、组件封装、请求层和权限体系，应优先复用项目现状，不为追求理想方案强行重构。
- 选型应优先保证团队可维护性、项目一致性和长期演进成本，而不是只追求单点最优。
- 同一类能力只保留一套主方案，避免请求层、状态管理、表格或表单体系并存。
- 新项目在确认技术栈时，还应明确是否采用最近验证通过的稳定版本基线，而不是默认沿用过时脚手架版本。

## 新项目决策要求

- 在进入实现前，至少确认以下内容：框架、组件库、路由方案、请求层、缓存方案、全局状态管理、表单方案、表格策略、权限组织方式。
- 如项目要作为模板仓库、脚手架或演示仓库长期复用，应额外确认关键依赖的推荐稳定版本和升级策略。
- 当存在多个合理候选时，应用简洁方式给出候选方案、适用场景和推荐理由，由用户做最终确认。
- 如用户未做选择，不应默认引入新的重量级依赖或复杂基础设施。
- 新项目一旦确认方案，应在初始化阶段一次性固定，避免开发过程中频繁切换。

## 已有项目决策要求

- 先检查当前项目的 package.json、锁文件、目录结构、请求封装、权限体系和公共组件封装，再决定是否沿用。
- 如现有方案可满足需求，应直接复用，不新增第二套同类能力。
- 只有在现有方案明显无法满足需求、成本过高或已造成结构失控时，才讨论替换或升级。
- 讨论替换时，应明确迁移范围、影响面、兼容成本和回退方式，不做隐式替换。
- 如现有项目版本明显落后且已影响能力、生态兼容或模板价值，应单独发起升级评估，而不是长期冻结在旧版本。

## 关键能力选型边界

### 路由方案

- React 管理后台默认采用 React Router 体系。
- React、React Router、Vite 和组件库版本应优先采用最近验证通过的稳定组合，而不是各自独立追新或长期停留在老版本。
- 新项目应由用户确认是否需要权限路由、动态菜单映射、多标签页缓存和页面级 keep-alive 替代方案。
- 已有项目如已封装路由守卫、菜单注册或页面缓存机制，应优先复用。

### 请求层与缓存方案

- 请求层应保持统一入口，统一处理鉴权、错误拦截、重试策略和响应结构转换。
- 新项目如需在原生 fetch 封装、Axios 封装或更高层数据访问方案中选择，应让用户确认。
- 缓存、失效更新、轮询和乐观更新等能力如需引入专门方案，也应先由用户确认，不默认绑定唯一实现。
- 已有项目如已具备稳定请求层和缓存约定，应优先沿用，不并行引入第二套请求模型。

### 全局状态管理

- 全局状态只承载跨页面共享且确有必要的状态，不承载所有页面数据。
- 新项目如需在轻量状态方案和更强约束方案之间选择，应由用户确认。
- 已有项目如已使用 Zustand、Redux Toolkit、Context 组合或其他稳定方案，应直接沿用。
- 不应因单个页面需求引入整套新的全局状态体系。

### 表单方案

- 如使用 Ant Design，默认优先评估是否直接采用其 Form 体系。
- 如存在高性能、大规模动态表单或强表单状态控制需求，可再讨论其他表单方案，但应先由用户确认。
- 已有项目如已有统一表单封装、校验规范和字段渲染方式，应优先复用。

### 表格方案

- 常规后台列表默认优先采用 Ant Design Table 或项目内已有表格封装。
- 只有在存在虚拟滚动、复杂列配置、拖拽、冻结列或超大数据量等明确需求时，才讨论更复杂的表格方案。
- 已有项目如已有表格基础能力封装，应继续在其上扩展，不新起第二套表格体系。

### 权限方案

- 至少明确路由权限、菜单权限和按钮权限的处理方式。
- 如项目存在数据权限，应单独定义数据访问边界，不把数据权限混入简单按钮显隐逻辑。
- 权限判断应集中封装，不允许页面各处散落硬编码判断。

## 常见候选方案

### 请求层与缓存

- 轻量方案：原生 fetch 或 Axios 的统一封装。适合接口规模中等、缓存诉求不强、团队希望保持简单的项目。
- 数据访问增强方案：在统一请求封装之上增加查询缓存和失效更新能力。适合列表、详情、编辑回写、刷新和请求状态管理较多的中大型后台。
- 已有项目如当前请求层已覆盖鉴权、错误处理、上传下载、刷新 token 和统一响应转换，不应仅因偏好差异替换底层请求方案。

### 全局状态管理

- 轻量共享状态方案：Context 配合 Hooks。适合全局状态较少、主要共享用户信息和布局状态的项目。
- 轻量状态库方案：如 Zustand。适合需要简洁 API、较低心智负担、同时又有一定跨页面共享状态的后台项目。
- 强约束状态方案：如 Redux Toolkit。适合团队已有成熟 Redux 经验、状态规模大、规范要求强的大型项目。
- 如果页面数据主要由请求层和页面局部状态即可承载，不应为了“看起来完整”而额外引入重量级全局状态库。

### 表单方案

- 组件库原生表单方案：直接使用 Ant Design Form。适合大多数常规后台 CRUD、搜索表单和弹窗表单场景。
- 更强调性能和字段控制的方案：如 React Hook Form 配合校验库。适合超大表单、动态字段复杂、性能和渲染控制要求更高的项目。
- 已有项目如表单校验、字段渲染、联动逻辑都已围绕某一方案稳定运行，应优先延续而不是切换。

### 表格方案

- 常规列表方案：Ant Design Table 或项目现有表格封装。适合大多数后台列表、搜索、分页、行操作场景。
- 增强型表格方案：在现有 Table 基础上做统一二次封装，补充列配置、操作列、状态列、查询联动等能力。适合长期维护的管理后台。
- 高复杂度表格方案：在明确需要虚拟滚动、复杂列拖拽、冻结列或高性能大表格时，再讨论更复杂的表格体系。

## 常见组合建议

- 基础稳妥型：React + TypeScript + Vite + React Router + Ant Design + 统一请求封装。适合中小型后台、模块不算复杂、希望尽快稳定落地的项目。
- 数据状态增强型：React + TypeScript + Vite + React Router + Ant Design + 统一请求封装 + 查询缓存方案 + 轻量全局状态。适合中大型后台、列表和详情联动较多、请求状态管理复杂的项目。
- 强规范大型项目型：React + TypeScript + Vite + React Router + Ant Design + 统一请求封装 + 查询缓存方案 + 强约束全局状态。适合多人协作、状态复杂、生命周期长且团队规范要求高的项目。
- 已有项目优先采用“现状增强”策略，即在现有技术栈上补缺能力，而不是整体替换为另一套理想组合。

## 快速决策矩阵

- 如果是新项目，且团队希望先快速落地常规 CRUD 后台，应优先选择基础稳妥型组合，不默认引入复杂缓存和重状态体系。
- 如果是新项目，且列表、详情、编辑回写、刷新联动较多，应优先考虑带查询缓存能力的组合。
- 如果是新项目，且团队人数多、协作周期长、状态规模大，应优先考虑约束更强、边界更清晰的状态组织方式。
- 如果是已有项目，且现有请求层、状态管理和表格封装已稳定运行，应优先沿用并做局部增强。
- 如果是已有项目，且当前项目已同时存在多套请求层、状态管理或表格封装，应先收敛重复基础设施，再讨论新增能力。
- 如果主要需求是页面增量开发，而不是基础设施重构，应避免因局部页面需求触发整套选型迁移。

## 团队与项目匹配建议

- 团队 React 工程经验较强、项目生命周期长、后台模块持续增长时，可接受更完整的请求缓存和状态管理体系。
- 团队以业务交付效率为先、页面以常规列表和表单为主时，应优先保持方案简单，减少基础设施心智负担。
- 如果团队已有明确的 Redux、Zustand、React Hook Form 或某套请求层积累，应把既有经验纳入决策，而不是只看单次项目偏好。
- 如果项目后续预期会扩展为平台型后台、配置型后台或高交互控制台，应在初期就确认缓存、权限、模块边界和表格封装策略。
- 如果项目只是中短周期的单业务后台，不应过早引入过重的工程抽象和多层封装。

## 选型提问模板

- 这是新项目还是已有项目？
- 如果是已有项目，当前已使用的请求层、状态管理、表单方案和表格封装分别是什么？
- 当前项目更偏向常规 CRUD 后台，还是高交互、高状态复杂度的平台型后台？
- 是否需要较强的请求缓存、失效更新、轮询、乐观更新或离线恢复能力？
- 是否存在超大表单、动态字段表单、复杂审批流或超大数据量表格？
- 团队是否已有固定技术栈积累，例如 Redux、Zustand、React Hook Form 或某套请求封装？
- 本次是做增量迭代，还是准备重构基础设施？

## 选型结论输出要求

- 输出结论时，应明确区分“推荐方案”“备选方案”“不建议方案”三类信息。
- 每个候选方案至少说明适用场景、主要收益、主要限制和迁移或维护成本。
- 如为新项目，应明确哪些点已由用户确认，哪些点仍待确认，不把待确认项写成既定事实。
- 如为已有项目，应明确哪些能力直接复用现状，哪些能力只是做增量增强，哪些能力确实需要调整。
- 如涉及脚手架、模板仓库或演示项目，还应明确版本基线策略，而不是只写技术名不写版本治理口径。
- 最终结论应能直接指导下一步实施，例如进入页面规范、脚手架初始化或增量改造，而不是停留在泛泛讨论。

## 选型结论模板示例

### 通用输出模板

- 项目类型：新项目或已有项目。
- 当前约束：是否已存在既有技术栈、请求层、状态管理、表单方案、表格封装、权限体系。
- 推荐方案：给出本次建议采用的主方案。
- 推荐理由：说明为什么该方案更适合当前项目阶段、团队能力和业务复杂度。
- 备选方案：给出可行但不是首选的候选方案。
- 不建议方案：说明当前阶段不建议采用的方案及原因。
- 待确认项：列出仍需用户拍板的技术点。
- 下一步动作：明确应进入脚手架初始化、页面规范落地、还是已有项目增量改造。

### 新项目示例

- 项目类型：新项目。
- 当前约束：尚未确定请求层、缓存方案、状态管理和表单方案；目标是先落地常规 CRUD 管理后台。
- 推荐方案：React + TypeScript + Vite + React Router + Ant Design + 统一请求封装。
- 推荐理由：当前以常规后台页面为主，优先保证实现效率和可维护性，不先引入过重的缓存和状态管理体系。
- 备选方案：在统一请求封装基础上增加查询缓存方案和轻量全局状态。适合后续列表、详情和编辑回写联动增多时升级。
- 不建议方案：一开始直接上强约束重状态体系。当前需求复杂度不足，容易增加团队心智负担和初期工程成本。
- 待确认项：请求层采用哪种统一封装方式；是否需要查询缓存方案；是否需要独立表单方案。
- 下一步动作：先请用户确认待确认项，再进入 React + Ant Design 管理后台规范并开始目录与页面模板落地。

### 已有项目示例

- 项目类型：已有项目。
- 当前约束：项目已使用 React、Ant Design、Axios 封装和 Zustand，且已有统一表格封装与权限组件。
- 推荐方案：沿用现有请求层、状态管理、表格封装和权限体系，仅在现有基础上补齐缺失能力。
- 推荐理由：当前基础设施已可支撑新增页面开发，直接复用现状能降低迁移成本并保持项目一致性。
- 备选方案：只在数据访问复杂度明显上升时，增量引入更明确的查询缓存能力，但不替换现有请求入口。
- 不建议方案：直接迁移到另一套请求层或状态管理体系。当前收益不足以覆盖重构成本和兼容风险。
- 待确认项：现有请求层是否已满足缓存与失效更新需求；是否需要增强表单性能或表格能力。
- 下一步动作：先按现有工程规范落地页面，如出现明确能力缺口，再单独评估是否做局部增强。

## 方案调整触发条件

- 当现有请求层已经无法统一处理鉴权、错误拦截、上传下载或响应结构时，可以讨论调整请求层。
- 当页面请求状态、缓存刷新、详情回写和列表联动已经明显失控时，可以讨论引入更明确的数据访问或缓存方案。
- 当全局状态中混入大量页面临时态、难以维护且修改影响面过大时，可以讨论重构状态边界。
- 当表单复杂度、动态字段规模或性能问题已经超出当前表单体系承载能力时，可以讨论升级表单方案。
- 当表格需求已经超出当前封装能力，例如大数据量、复杂列交互、冻结列或拖拽配置时，可以讨论升级表格策略。
- 只要现有方案仍可满足需求，就不应仅因“社区更流行”或“理论上更先进”而发起迁移。

## 推荐决策顺序

- 先判断是新项目还是已有项目。
- 再确认是否已有明确框架和组件库约定。
- 再确认请求层、缓存、状态管理、表单和表格方案是否已有现成实现。
- 只有在缺口明确存在时，才列出候选方案让用户选择。
- 方案确认后，再进入页面规范和具体实现。

## 禁止事项

- 禁止在用户未确认的情况下，直接假定唯一请求库、缓存方案、状态管理或表单方案。
- 禁止在已有项目中无视现有架构，强行迁移到另一套技术体系。
- 禁止同一项目中长期并存两套请求层、两套状态管理方案或两套表格体系。
- 禁止只给出选型结论，不说明适用场景、收益和成本。

## 提交前自检

- 是否明确区分了新项目选型和已有项目复用两种场景。
- 是否先检查了项目现有技术栈和公共能力，而不是直接引入新方案。
- 是否对候选方案说明了适用场景、收益、限制和迁移成本。
- 是否避免了在同一项目中形成并行的重复基础设施。
