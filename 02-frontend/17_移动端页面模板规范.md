# 移动 Web 页面模板规范

## 目标

- 为移动 Web 业务 H5、移动 Web 工作台、移动 Web 活动页和移动 Web AI 助手页提供统一的模板输出要求。
- 让 Ant Design Mobile 不只停留在技术栈说明，而是有可直接落地的页面骨架规范。
- 避免输出只有移动视觉外壳、缺少状态闭环和关键动作反馈的半成品页面。

## 适用范围

- 适用于基于手机浏览器或 WebView 访问的业务办理页、审批页、结果页、轻量工作台、活动承接页和移动 Web AI 助手页。
- 涉及技术栈边界时，应结合 [16_AntDesignMobile前端规范](16_AntDesignMobile前端规范.md) 使用。
- 涉及移动 Web AI 助手边界时，应结合 [11_AntDesignX智能应用前端规范](11_AntDesignX智能应用前端规范.md) 使用。
- 页面场景不明时，应先读 [15_移动端页面场景选型规范](15_移动端页面场景选型规范.md)。
- 本文件不覆盖 Flutter、React Native、原生 iOS、原生 Android 等 App 工程页面模板。

## 使用原则

- 模板优先定义页面骨架、操作闭环和关键状态，不替代业务内容本身。
- 移动 Web 页面应优先保证任务完成路径清晰，不以复杂装饰替代操作反馈。
- 如项目已有成熟移动 Web 容器骨架、表单容器、底部操作条或导航框架，应优先复用现有实现。

## 业务表单页模板

### 默认结构

- 顶部标题与说明区。
- 分组表单区。
- 风险提示或补充说明区。
- 固定底部操作区。

### 默认输出要求

- 默认包含空白初始态、字段校验失败态、提交中状态、成功状态和失败重试入口。
- 默认处理键盘弹起后的输入可见性、必填项提示和底部提交按钮可点击性。
- 如存在分步流程，应明确当前步骤、上一步、下一步和中断返回能力。

## 审批详情或结果页模板

### 默认结构

- 状态摘要区。
- 关键信息卡片区。
- 时间线或明细区。
- 固定底部操作区。

### 默认输出要求

- 默认包含处理中、已完成、已拒绝、失败或失效等状态。
- 默认区分只读信息、可操作区域和风险操作，不混成一整块内容流。
- 如存在撤回、重新提交、催办或联系处理人等动作，应明确前置条件和反馈。

## 移动 Web 工作台模板

### 默认结构

- 顶部身份或组织摘要区。
- 待办卡片区。
- 快捷入口区。
- 列表或消息区。
- 底部导航区。

### 默认输出要求

- 默认包含加载态、空态、失败态、下拉刷新和分页或继续加载入口。
- 默认突出高频任务和待办数量，不把所有业务入口平铺成无主次宫格。
- 如同时存在通知、待办和统计，应明确层级，不让次要信息挤占首屏任务入口。

## 移动 Web 活动或转化页模板

### 默认结构

- 首屏主题区。
- 卖点与收益区。
- 证明材料区。
- 表单或行动区。
- 补充说明区。

### 默认输出要求

- 默认只保留一个主转化目标，例如报名、领取、预约或留资。
- 默认包含提交成功、失败和重复提交控制。
- 默认控制页面长度和区块数量，不直接复制桌面官网页的长链路区块结构。

## 移动 Web AI 助手页模板

### 默认结构

- 简化上下文区。
- 会话区。
- 输入区。
- 按需存在的建议问题区、结果卡片区和快捷操作区。

### 默认输出要求

- 默认包含欢迎态、发送中状态、流式输出状态、失败状态和中断能力。
- 默认区分消息正文、工具反馈和结果卡片，不全部混成普通消息流。
- 如 AI 助手附着在业务办理页中，应明确主业务区和 AI 辅助区的边界。

## 输出要求

- 输出页面代码时，应包含页面骨架、主要状态、关键交互和异常反馈，而不是只给静态手机端示意布局。
- 业务页应补齐提交和结果闭环；工作台应补齐刷新和加载状态；AI 助手应补齐消息状态和失败恢复。
- 如需求同时涉及桌面端与移动 Web，应拆分主次结构，不生成一套无边界的响应式万能模板。

## 提交前自检

- 是否先判断了页面属于业务表单页、审批结果页、移动 Web 工作台、活动承接页还是移动 Web AI 助手页。
- 是否补齐了对应模板要求中的默认结构和关键状态。
- 是否避免把桌面后台模板或官网长页模板直接套用到移动 Web 页面。
- 是否在已有项目中优先复用了既有移动容器和公共页面骨架。
