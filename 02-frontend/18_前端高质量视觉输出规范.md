# 前端高质量视觉输出规范

## 目标

- 把前端页面输出从“能用、整齐、像后台”提升到“有明确视觉方向、品牌辨识度和任务焦点”。
- 避免 AI 默认回落到白底卡片、默认蓝按钮、弱层级留白和缺少重点的中庸界面。
- 为官网、管理后台、AI 工作台和移动 Web 页面提供统一的高质量视觉约束与验收口径。

## 适用范围

- 适用于前端新页面生成、现有页面重做、演示项目搭建和视觉升级任务。
- 适用于 React、Vue、移动 Web 和其他前端页面形态，不替代具体组件库或页面模板规范。
- 如任务需要跨 Web、桌面端、小程序和移动 App 统一口径，应先补读 [../01-general/09_多终端体验与高质量输出规范.md](../01-general/09_多终端体验与高质量输出规范.md)。
- 涉及页面场景判断时，应先读 [13_前端页面场景选型规范](13_前端页面场景选型规范.md)。
- 涉及具体页面模板时，应与对应模板规范一起使用，而不是单独依赖本文件。

## 核心原则

### 1. 先定义视觉方向，再开始落代码

- 生成页面前，应先明确页面主目标、主要受众、品牌气质和本页最重要的视觉焦点。
- 不应在没有视觉方向的情况下，直接用组件库默认样式堆出页面。
- 如用户未给出完整设计稿，仍应先形成最小视觉方向卡，再进入实现。

### 2. 页面必须有主次关系，不做信息平铺

- 首屏、摘要区、主任务区和次级信息区应有明确层级，而不是所有卡片同权展示。
- 用户第一眼应能判断当前页面最重要的一个动作、一个信息组或一个状态摘要。
- 不应把统计卡、列表卡、说明卡、操作卡都做成完全同权的重复容器。

### 3. 不使用默认组件审美作为最终审美

- 组件库负责提供可用基础，不等于最终视觉方案。
- 不应直接交付明显保留默认边框、默认间距、默认字号节奏和默认主色的页面。
- 主题 token、容器层级、背景层次、版式节奏和重点区域应经过二次设计。

### 4. 惊艳不等于堆装饰

- 高质量视觉来自明确的方向、稳定的层级和受控的细节，而不是渐变乱用、阴影乱叠或动效泛滥。
- 视觉增强必须服务于任务理解、品牌表达和转化效率，不制造阅读干扰。
- 如果装饰元素不能提升信息理解或品牌辨识，应删除而不是保留。

## 视觉方向卡要求

开始页面实现前，至少应先明确以下内容：

1. 页面场景：管理后台、AI 工作台、官网落地页或移动 Web 页面。
2. 页面目标：本页最重要的任务、信息或转化动作是什么。
3. 受众判断：谁在使用页面，关注效率、可信度、品牌感还是任务完成速度。
4. 风格关键词：例如稳健治理、可信企业、未来执行、冷静高密度、品牌叙事。
5. 版式策略：例如高密度控制台、叙事型落地页、分栏工作台、轻量移动任务页。
6. 视觉焦点：首屏第一眼必须看到什么。
7. 禁止事项：本页明确不能出现的俗套写法。

## 主题与样式基线

### 色彩

- 必须先定义主色、强调色、状态色、中性色和背景层级，不直接沿用组件库默认配色作为最终结果。
- 状态色仍应遵循语义一致性，但品牌主色不强制等于组件库官方蓝。
- 背景应至少区分页面底色、主容器底色、次级容器底色和强调区底色，避免整页只有单一白底。

### 字体与字号

- 应明确标题层级、摘要层级、正文层级和辅助说明层级，不使用单一字号做整页信息铺排。
- 首页、工作台标题和核心指标可适度建立更强对比，但必须保持可读性。
- 不应出现标题不够突出、摘要和正文几乎同层、标签和正文抢层级的问题。

### 间距与密度

- 应根据场景控制信息密度，而不是一律宽松留白。
- 管理后台和执行工作台应优先保证有效信息密度，减少空旷大白块。
- 官网和活动页可有更舒展的节奏，但仍应维持清晰区块边界和阅读路径。

### 圆角、边框与阴影

- 应统一容器圆角、边框强弱和阴影层级，不让同页出现多套容器语言。
- 普通信息容器、强调容器、悬浮面板和弹层的层次应能一眼区分。
- 不应把所有容器都做成同一层级的白卡片加浅边框。

## 各场景专项要求

### 管理后台

- 目标是专业、可信、清晰、高效，不是保守和平淡。
- 应通过指标层级、重点操作区、筛选区收敛和列表密度控制建立专业感。
- 不应生成整页稀疏留白、卡片同构、重点信息缺失的后台页面。

### AI 工作台

- 应突出任务上下文、执行状态、结果沉淀和交互反馈，而不是把聊天区与普通信息卡简单拼接。
- 对话区、步骤区、结果区和状态区必须有清晰分工和视觉区分。
- 不应把 AI 工作台做成普通后台加一个聊天框，或做成只有消息气泡没有工程执行结构的伪工作台。

### 官网与落地页

- 应突出价值主张、核心卖点、证明材料和 CTA，而不是模板区块堆叠。
- 首屏必须回答“是什么、解决什么问题、为什么值得继续看”。
- 不应把官网做成后台信息搬运页，也不应只做一组漂亮容器却没有叙事和转化路径。

### 移动 Web

- 应突出单手操作、短链路和主动作，不做桌面页面压缩版。
- 顶部摘要区、表单区、结果区和固定底部操作区必须有明确层级。
- 不应在手机页面里复制桌面端多栏结构或高密度表格布局。

## 反平庸约束

- 禁止默认生成“大面积纯白背景 + 多个同构白卡 + 默认蓝主按钮 + 稀疏大留白”的无辨识度布局。
- 禁止把所有重点都交给颜色表达，而缺少版式层级、标题层级和信息组织。
- 禁止用装饰图形替代真实的信息设计、任务路径设计和转化设计。
- 禁止四个不同展示面共用一套毫无区分的视觉语言，导致官网、后台、门户和执行台看起来像同一页面换文案。

## 输出要求

- 输出代码前，应能说明当前页面的视觉方向卡。
- 输出代码时，应包含主题 token、关键容器层级和重点区域样式，而不是只改内容不改视觉骨架。
- 如为演示项目，应保证不同展示面共享品牌一致性，同时各自保留场景辨识度。

## 验收要求

- 至少进行一次截图级或页面级视觉验收，确认页面不是默认组件堆叠结果。
- 至少检查以下问题：是否有主视觉焦点、是否有信息主次、是否仍残留默认样式感、是否与页面场景匹配。
- 如为多页面项目，还应检查跨页面品牌一致性和场景差异化是否同时成立。

## 提交前自检

- 是否在实现前先定义了视觉方向卡。
- 是否建立了明确的标题、摘要、正文和辅助信息层级。
- 是否通过主题 token、背景层级、容器语言和重点区设计摆脱默认组件观感。
- 是否避免了无重点的白卡平铺、过度留白和信息主次缺失。
- 是否完成了至少一次截图级或页面级视觉验收。
