# 前端稳定版本基线清单

## 目标

- 为新项目、模板仓库、演示仓库和脚手架初始化提供一份可直接引用的前端依赖基线。
- 避免 AI 在初始化项目时沿用明显落后的历史版本，或在没有依据的情况下盲目追逐最新发布版本。
- 让“保持与时俱进”具备统一口径：优先采用最近一次可核实的公开稳定版本快照，并在目标仓库内完成一次落地验证后锁定。

## 适用范围

- 适用于前端新项目初始化、模板仓库升级、演示项目升级和技术选型讨论。
- 不替代已有项目的锁文件与兼容策略；已有项目默认优先复用现状。
- 与通用前端规范、具体技术栈规范和交付质量规范配合使用。

## 使用原则

### 1. 基线不是 latest 别名

- 本清单记录的是 ai-dev-os 当前推荐采用的公开稳定版本快照，不等于永远使用 latest。
- 版本采用前，仍应结合目标仓库的 Node 版本、现有依赖、构建链路和组件兼容性做一次落地验证。

### 2. 新项目优先，已有项目谨慎

- 新项目、样板仓库和演示仓库，默认优先向本清单靠拢。
- 已有项目若未出现能力阻塞、生态断层或安全维护风险，不因本清单强制跨主版本迁移。

### 3. 先看组合，再看单包

- 技术基线应优先按“React + 构建工具 + 路由 + 组件库”的组合判断，而不是逐个包独立升级。
- 如某一依赖已升到新主版本，但周边插件、路由或组件库尚未完成兼容验证，不应孤立追新。

## 当前推荐快照

### 通用 React 工程基线

- React：19.2.5
- React DOM：19.2.5
- TypeScript：6.0.3
- Vite：8.0.10
- @vitejs/plugin-react：6.0.1
- React Router DOM：7.14.2

### 管理后台基线

- UI 组件库：Ant Design 6.3.7
- 图表方案：按项目需要单独确认，不在本清单锁死

### AI 工作台基线

- AI 交互组件库：@ant-design/x 2.5.0
- 与 Ant Design 组合：优先采用 Ant Design 6.3.7 + @ant-design/x 2.5.0

## 兼容约束快照

### 运行时与插件要求

- Vite 8 的 Node 要求：^20.19.0 或 >=22.12.0
- @vitejs/plugin-react 6 依赖 Vite 8 主版本

### 组件库与 React 兼容要求

- Ant Design 6 的 React peer 范围：>=18.0.0
- @ant-design/x 2 的 React peer 范围：>=18.0.0
- @ant-design/x 2 的 antd peer 范围：^6.1.1

## 默认推荐组合

### 组合 A：新项目优先基线

- React 19.2.5
- React DOM 19.2.5
- TypeScript 6.0.3
- Vite 8.0.10
- @vitejs/plugin-react 6.0.1
- React Router DOM 7.14.2
- Ant Design 6.3.7

适用场景：

- 新启动的 React 前端项目
- 演示仓库、样板仓库、模板仓库
- 没有历史兼容包袱，希望默认靠近当前稳定生态的项目

### 组合 B：AI 工作台优先基线

- 组合 A 全量依赖
- @ant-design/x 2.5.0

适用场景：

- 独立 AI 工作台
- 带明显任务执行、消息流和结果区的 Agent 页面

### 组合 C：已有项目保守增强基线

- 主框架、构建工具和路由优先复用现状
- 仅在用户明确要求升级、能力受限或模板价值受损时，再评估是否迁移到组合 A 或 B

适用场景：

- 已经存在 React 18、Router 6、Ant Design 5 等历史组合的项目
- 当前仍能稳定支撑业务，只是需要补功能或局部视觉升级的项目

## 场景化建议

### 管理后台

- 新项目默认优先采用组合 A。
- 如项目已有稳定后台脚手架，但版本明显落后到影响组件生态、路由能力或模板复用价值，应优先评估升级。

### 官网与落地页

- 新项目默认优先采用组合 A。
- 如果同时使用 Landing 类区块方案，应先确认其与 React 19、Vite 8、Ant Design 6 的兼容性，再固定版本。

### AI 工作台

- 新项目默认优先采用组合 B。
- 不建议继续使用 Ant Design 5 与早期 Ant Design X 组合做新的样板工程，除非目标仓库已有明确历史约束。

### 移动 Web

- React、TypeScript、Vite 仍优先参考组合 A。
- 移动端 UI 组件库和容器桥接依赖不在本清单强制锁定，应按目标场景单独确认。

## 升级触发条件

- 当前模板仓库、样板仓库或演示仓库的关键依赖主版本明显落后于主流稳定生态。
- 当前版本已阻碍新组件能力、构建能力、路由能力或长期维护。
- 当前项目需要用更高版本能力解决已有问题，而不是单纯为了追新。

## 不建议的做法

- 在规则库中只写“使用稳定主版本”，却不给出任何版本快照。
- 把 latest 直接当成推荐版本写进规范。
- 在已有项目中无视锁文件和兼容成本，直接按清单强行跨主版本升级。
- 只升级单个核心包，不验证与构建工具、插件和组件库的组合兼容性。

## 维护要求

- 本清单应定期回看，建议至少按季度复核一次，或在样板工程升级后立即回写。
- 每次更新应至少回写：快照日期、版本来源、推荐组合、关键兼容说明。
- 如果团队已有内部验证流水线，可把“公开稳定快照”升级为“团队已验证稳定基线”。

## 提交前自检

- 是否明确区分了新项目基线和已有项目复用两种场景。
- 是否给出了可核实的快照日期和版本来源。
- 是否按组合而不是单包描述推荐方案。
- 是否补充了关键兼容信息，而不是只有裸版本号。
