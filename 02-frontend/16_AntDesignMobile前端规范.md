# Ant Design Mobile 移动 Web 前端规范

## 适用范围

- 适用于基于 React、Ant Design Mobile 的业务 H5、企业内部移动 Web 工作台、移动表单页、扫码进入页和轻量移动 Web 交互页面。
- 本文件不替代桌面管理后台规范；如页面主体是高密度表格、复杂配置台或多栏运营控制台，仍应优先遵循 [02_管理后台页面通用规范](02_管理后台页面通用规范.md) 和 [08_AntDesignReact管理后台前端规范](08_AntDesignReact管理后台前端规范.md)。
- 本文件也不替代 AI 应用前端规范；如页面主体是多轮对话、复杂任务编排或桌面工作台，应优先遵循 [11_AntDesignX智能应用前端规范](11_AntDesignX智能应用前端规范.md)。
- 新项目如未明确是否采用 Ant Design Mobile，应先让用户确认是否属于移动 Web 主场景，再按本规范输出。
- 已有项目如已存在稳定移动端组件封装、主题 token、表单容器或导航体系，应优先复用现状，不因本规范强行替换。
- 本文件不覆盖 Flutter、React Native、原生 iOS、原生 Android 等 App 工程实现。

## 目标

- 为移动 Web React 页面补齐页面结构、交互节奏、手势和固定操作区等实现边界。
- 明确 Ant Design Mobile 只解决移动 Web 页面问题，不上升为桌面管理后台、官网页面或原生 App 的默认组件体系。
- 保证移动 Web 页面在可用性、性能和工程一致性之间保持平衡。

## 技术栈约定

- 框架：React，优先使用函数组件与 Hooks。
- 语言：TypeScript。
- UI 基础库：Ant Design Mobile，使用项目锁定的稳定版本。
- 路由、请求层、缓存方案、状态管理、埋点、鉴权和容器桥接能力，优先复用项目现有实现；新项目需先由用户确认。
- 如无明确必要，不额外并行引入第二套移动 Web UI 框架。

## 场景边界

- 适合使用 Ant Design Mobile 的场景：移动表单、审批页、结果页、待办页、轻量工作台、扫码落地业务页、固定底部操作页。
- 可以结合使用的场景：移动 Web AI 助手、移动 Web 活动承接页、需要原生容器内嵌的业务 H5。
- 不适合使用 Ant Design Mobile 的场景：桌面后台大表格、复杂多栏工作台、品牌官网主站、桌面优先的 AI 控制台。
- 如果同一项目同时存在桌面后台和移动 Web H5，应在规范、目录和组件封装层面分开处理，不把桌面组件直接平移到移动 Web。

## 页面结构建议

- 业务表单页：优先采用“标题区 + 分组表单区 + 固定底部提交区”的结构。
- 审批或结果页：优先采用“状态摘要区 + 关键信息区 + 时间线或明细区 + 固定底部操作区”的结构。
- 移动 Web 工作台：优先采用“顶部信息区 + 待办区 + 快捷入口区 + 列表区 + 底部导航”的结构。
- 移动 Web AI 助手：优先采用“简化会话区 + 输入区 + 结果区”，避免在手机浏览器或 WebView 中复制桌面多栏工作台。

## 交互规范

- 主操作应尽量固定在底部安全区域内，按钮文案明确，不只使用图标表达关键动作。
- 表单输入应优先考虑单手操作、键盘弹起后的可视区域和焦点切换体验。
- 页面反馈应及时，包括提交中、加载中、成功、失败和可重试状态。
- 复杂选择器、日期时间、地区、级联选项等应优先使用移动 Web 更适配的弹层、选择器或分步交互，不直接复用桌面下拉体验。
- 涉及破坏性动作或高成本动作时，应提供明确确认，不依赖悬停、右键或桌面式提示交互。

## 组件使用建议

- NavBar、TabBar、CapsuleTabs、Tabs 用于移动导航与分段切换。
- Form、Input、Selector、Picker、DatePicker、TextArea 用于表单和录入场景。
- List、Card、Grid、Steps、Result、Empty、Skeleton、Toast、Dialog 按语义使用。
- Footer 与固定底部操作区应兼顾安全区和系统手势区域，不让关键按钮被遮挡。
- 如项目已有上传、扫码、地图、原生分享或容器桥接封装，应优先复用而不是在页面内直接分散实现。

## 数据与状态约定

- 页面本地表单态、分步流程态、临时筛选条件和列表滚动位置应优先在页面或领域模块内管理。
- 只有跨页面共享的用户身份、组织信息、全局筛选、容器能力和少量全局设置才进入全局状态。
- 移动 Web 网络状态不稳定时，应明确请求失败、重复提交、防抖和重试策略。
- 长列表、分页加载和下拉刷新应遵循项目现有数据访问约定，不并行引入第二套请求模型。

## 容器与兼容边界

- 如页面运行在企业微信、飞书、钉钉或 App WebView 中，应明确登录态、返回行为、分享能力和容器桥接方式。
- 如页面包含扫码、拍照、定位、文件上传或原生分享，应优先复用项目现有桥接层并明确权限失败处理。
- 页面应优先适配常见手机宽度和安全区，不把桌面断点策略照搬到移动 Web。

## 性能与可用性要求

- 首屏应优先保证关键任务可进入，不为少量视觉效果引入明显的体积和交互负担。
- 图片、图标、分包和异步资源应按移动网络环境控制体积。
- 长表单和长列表应注意分组、折叠、渐进加载和骨架屏，避免一屏过长失去任务节奏。
- 文本、按钮和交互热区应保证移动 Web 可点按性，不生成需要精细鼠标操作的界面。

## 输出要求

- 输出代码时，应包含页面骨架、关键状态、提交和反馈闭环，而不是只给静态手机壳界面。
- 如页面还涉及桌面端配套页面，应明确当前输出仅适用于移动 Web，不混成响应式万能页面模板。
- 如未确认容器环境、登录方式、桥接能力或埋点方案，应先向用户确认后再做具体实现。

## 推荐实现顺序

- 先确认页面是否以手机浏览器或 WebView 为主，以及属于业务 H5、移动 Web 工作台、活动页还是移动 Web AI 助手。
- 再确认是否沿用现有移动端路由、请求层、容器桥接、鉴权和埋点方案。
- 再确定页面骨架、底部操作、关键表单结构和异常反馈。
- 完成骨架后，再补空态、失败态、重试和容器能力调用。

## 不建议的做法

- 把桌面管理后台页面缩成手机宽度后直接交付。
- 在一个移动 Web 页面里塞入复杂多栏布局和高密度表格。
- 忽略键盘弹起、安全区和底部固定按钮的遮挡问题。
- 在已有项目中并行引入第二套移动端 UI 体系或桥接封装。

## 提交前自检

- 是否明确这是移动 Web 主场景，而不是桌面后台兼容视图。
- 是否补齐了提交中、失败、重试、空态和底部操作反馈。
- 是否复用了现有鉴权、请求层、容器桥接和埋点能力。
- 是否避免生成依赖桌面交互习惯的移动 Web 页面。
