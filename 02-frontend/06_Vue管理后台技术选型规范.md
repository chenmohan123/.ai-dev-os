# Vue 管理后台技术选型规范

## 目标

- 规范 Vue 管理后台在新项目和已有项目中的关键技术决策方式。
- 明确哪些技术点可以作为候选项让用户选择，哪些技术点应优先复用项目现状。
- 避免在页面生成或规范落地时擅自拍板组件库、请求层、状态管理、表单方案和表格体系。

## 适用范围

- 适用于 Vue 管理后台项目的技术选型、方案确认和实现前决策。
- 涉及 Vue 3 + Ant Design Vue 的页面与工程约束时，应结合 [04_AntDesignVue管理后台前端规范](04_AntDesignVue管理后台前端规范.md) 使用。
- 涉及 Vue 3 + Element Plus 的页面与工程约束时，应结合 [05_ElementPlus管理后台前端规范](05_ElementPlus管理后台前端规范.md) 使用。

## 总体原则

- 新项目如未明确指定组件库、请求层、状态管理、表单方案或表格策略，应先列出候选并让用户确认，不替用户拍板。
- 已有项目如已存在稳定技术栈、目录结构、组件封装、请求层和权限体系，应优先复用项目现状，不为追求理想方案强行重构。
- 选型应优先保证团队可维护性、项目一致性和长期演进成本，而不是只追求单点最优。
- 同一类能力只保留一套主方案，避免请求层、状态管理、表格或表单体系并存。

## 新项目决策要求

- 在进入实现前，至少确认以下内容：组件库、路由方案、请求层、状态管理、表单方案、表格策略、权限组织方式。
- 当存在多个合理候选时，应用简洁方式给出候选方案、适用场景和推荐理由，由用户做最终确认。
- 如用户未做选择，不应默认引入新的重量级依赖或复杂基础设施。
- 新项目一旦确认方案，应在初始化阶段一次性固定，避免开发过程中频繁切换。

## 已有项目决策要求

- 先检查当前项目的 package.json、锁文件、目录结构、组件库使用方式、请求封装、状态管理和权限体系，再决定是否沿用。
- 如现有方案可满足需求，应直接复用，不新增第二套同类能力。
- 只有在现有方案明显无法满足需求、成本过高或已造成结构失控时，才讨论替换或升级。
- 讨论替换时，应明确迁移范围、影响面、兼容成本和回退方式，不做隐式替换。

## 关键能力选型边界

### 组件库方案

- 新项目如未指定组件库，应先让用户确认采用 Ant Design Vue、Element Plus 或项目已有设计体系。
- Ant Design Vue 更适合需要 Ant 风格组件体系、表单和中后台交互统一性的场景。
- Element Plus 更适合 Vue 生态惯性更强、团队已有 Element 积累或已有项目已基于 Element Plus 的场景。
- 已有项目应优先沿用当前组件库，不因偏好差异并行引入第二套主组件体系。

### 路由方案

- Vue 管理后台默认采用 Vue Router 体系。
- 新项目应由用户确认是否需要权限路由、动态菜单映射、多标签页缓存和页面级 keep-alive。
- 已有项目如已封装路由守卫、菜单注册或页面缓存机制，应优先复用。

### 请求层与数据访问方案

- 请求层应保持统一入口，统一处理鉴权、错误拦截、重试策略和响应结构转换。
- 新项目如需在 fetch 封装、Axios 封装或项目已有服务层模式之间选择，应让用户确认。
- 已有项目如已具备稳定请求层和数据访问约定，应优先沿用，不并行引入第二套请求模型。

### 状态管理方案

- 全局状态只承载跨页面共享且确有必要的状态，不承载所有页面数据。
- 新项目如需在 Pinia、Vue 自带组合式状态组织或其他稳定方案之间选择，应由用户确认。
- 已有项目如已使用 Pinia、Vuex 或其他稳定方案，应优先沿用，不因单个页面需求重构整套状态体系。

### 表单与表格方案

- 常规后台表单默认优先采用组件库原生表单体系或项目现有封装。
- 常规后台列表默认优先采用组件库自带 Table 或项目现有表格封装。
- 只有在存在复杂动态表单、大数据量表格、虚拟滚动、冻结列或复杂列交互等明确需求时，才讨论升级方案。

### 权限方案

- 至少明确路由权限、菜单权限和按钮权限的处理方式。
- 如项目存在数据权限，应单独定义数据访问边界，不把数据权限混入简单按钮显隐逻辑。
- 权限判断应集中封装，不允许页面各处散落硬编码判断。

## 常见候选方案

### 组件库

- Ant Design Vue：适合企业级中后台风格、表单和表格交互较多、希望与 Ant 设计语言保持一致的场景。
- Element Plus：适合 Vue 团队积累更强、已有 Element 生态基础或已有项目已采用该体系的场景。
- 项目现有设计体系：如已有内部组件库或统一封装，应优先复用，不随意切换到外部组件库。

### 请求层与状态管理

- 统一请求封装：适合大多数 Vue 后台项目，应作为默认底层能力。
- Pinia：适合 Vue 3 项目中的轻中型全局状态管理，心智负担较低。
- 项目现有状态组织：如已有成熟 store 模块和约定，应优先沿用。

## 常见组合建议

- Vue 3 + Ant Design Vue + 统一请求封装：适合需要 Ant 风格后台、页面以常规 CRUD 为主的项目。
- Vue 3 + Element Plus + 统一请求封装：适合 Vue 生态惯性更强、希望快速落地标准后台页面的项目。
- Vue 3 + 项目现有组件库 + 现有请求层与状态管理：适合已有项目增量开发和长期维护场景。
- 已有项目优先采用“现状增强”策略，即在现有技术栈上补缺能力，而不是整体替换为另一套理想组合。

## 快速决策矩阵

- 如果是新项目，且团队希望先快速落地常规 CRUD 后台，应优先选择 Vue 3 + 既定组件库 + 统一请求封装的基础组合，不默认引入复杂基础设施。
- 如果是新项目，且团队已有明确的 Ant Design Vue 或 Element Plus 积累，应优先沿用该组件体系，减少迁移和学习成本。
- 如果是已有项目，且现有组件库、请求层、状态管理和表格封装已稳定运行，应优先沿用并做局部增强。
- 如果是已有项目，且当前项目已同时存在多套组件封装、请求层或状态管理，应先收敛重复基础设施，再讨论新增能力。
- 如果主要需求是页面增量开发，而不是基础设施重构，应避免因局部页面需求触发整套选型迁移。

## 团队与项目匹配建议

- 团队以 Vue 生态为主、业务交付效率优先时，应优先保持方案简单和一致，减少基础设施心智负担。
- 团队已有 Ant Design Vue 或 Element Plus 的稳定积累时，应把既有经验纳入决策，而不是只看单次项目偏好。
- 如果项目后续预期会扩展为平台型后台、配置型后台或高交互控制台，应在初期确认组件体系、权限组织、模块边界和表格封装策略。
- 如果项目只是中短周期的单业务后台，不应过早引入过重的工程抽象和多层封装。

## 选型提问模板

- 这是新项目还是已有项目？
- 如果是已有项目，当前使用的是 Ant Design Vue、Element Plus，还是内部组件体系？
- 当前项目的请求层、状态管理、表单封装和表格封装分别是什么？
- 当前项目更偏向常规 CRUD 后台，还是高交互、高状态复杂度的平台型后台？
- 是否存在超大表单、动态字段表单、复杂审批流或超大数据量表格？
- 团队是否已有固定技术栈积累，例如 Pinia、Vuex、Ant Design Vue 或 Element Plus？
- 本次是做增量迭代，还是准备重构基础设施？

## 选型结论输出要求

- 输出结论时，应明确区分“推荐方案”“备选方案”“不建议方案”三类信息。
- 每个候选方案至少说明适用场景、主要收益、主要限制和迁移或维护成本。
- 如为新项目，应明确哪些点已由用户确认，哪些点仍待确认，不把待确认项写成既定事实。
- 如为已有项目，应明确哪些能力直接复用现状，哪些能力只是做增量增强，哪些能力确实需要调整。
- 最终结论应能直接指导下一步实施，例如进入页面规范、脚手架初始化或增量改造，而不是停留在泛泛讨论。

## 选型结论模板示例

### 通用输出模板

- 项目类型：新项目或已有项目。
- 当前约束：是否已存在既有组件库、请求层、状态管理、表单方案、表格封装和权限体系。
- 推荐方案：给出本次建议采用的主方案。
- 推荐理由：说明为什么该方案更适合当前项目阶段、团队能力和业务复杂度。
- 备选方案：给出可行但不是首选的候选方案。
- 不建议方案：说明当前阶段不建议采用的方案及原因。
- 待确认项：列出仍需用户拍板的技术点。
- 下一步动作：明确应进入页面规范、脚手架初始化、还是已有项目增量改造。

### 新项目示例

- 项目类型：新项目。
- 当前约束：尚未确定组件库、请求层和状态管理方案；目标是先落地常规 CRUD 管理后台。
- 推荐方案：Vue 3 + 已确认组件库 + 统一请求封装。
- 推荐理由：当前以常规后台页面为主，优先保证实现效率和项目一致性，不先引入过重基础设施。
- 备选方案：在统一请求封装基础上增加项目需要的状态组织或表格封装增强能力。
- 不建议方案：在组件库未确认前并行引入多套组件体系或重状态基础设施。当前收益不足以覆盖认知和维护成本。
- 待确认项：组件库最终采用 Ant Design Vue、Element Plus 还是项目现有设计体系；状态管理是否沿用项目默认方案。
- 下一步动作：先请用户确认待确认项，再进入对应 Vue 页面规范并开始目录与页面模板落地。

### 已有项目示例

- 项目类型：已有项目。
- 当前约束：项目已使用 Vue 3、既有组件库、统一请求封装和固定状态管理方案，且已有表格封装与权限组件。
- 推荐方案：沿用现有组件库、请求层、状态管理和权限体系，仅在现有基础上补齐缺失能力。
- 推荐理由：当前基础设施已可支撑新增页面开发，直接复用现状能降低迁移成本并保持项目一致性。
- 备选方案：仅在组件、表格或状态组织能力明显不足时，做局部增强而不替换主体系。
- 不建议方案：直接迁移到另一套组件库或状态管理体系。当前收益不足以覆盖重构成本和兼容风险。
- 待确认项：现有表格封装、表单方案和权限能力是否已满足本次需求。
- 下一步动作：先按现有工程规范落地页面，如出现明确能力缺口，再单独评估是否做局部增强。

## 方案调整触发条件

- 当现有组件库或其二次封装已经无法支撑核心页面需求时，可以讨论调整组件体系或封装策略。
- 当现有请求层已经无法统一处理鉴权、错误拦截、上传下载或响应结构时，可以讨论调整请求层。
- 当全局状态中混入大量页面临时态、难以维护且修改影响面过大时，可以讨论重构状态边界。
- 当表单复杂度、动态字段规模或性能问题已经超出当前表单体系承载能力时，可以讨论升级表单方案。
- 当表格需求已经超出当前封装能力，例如大数据量、复杂列交互、冻结列或虚拟滚动时，可以讨论升级表格策略。
- 只要现有方案仍可满足需求，就不应仅因“社区更流行”或“理论上更先进”而发起迁移。

## 提交前自检

- 是否明确区分了新项目选型和已有项目复用两种场景。
- 是否先检查了项目现有技术栈和公共能力，而不是直接引入新方案。
- 是否对候选方案说明了适用场景、收益、限制和迁移成本。
- 是否避免了在同一项目中形成并行的重复基础设施。
