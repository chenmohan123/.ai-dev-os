# 移动 Web 页面场景选型规范

## 目标

- 为移动 Web 页面先做场景分流，明确什么时候按业务 H5、轻量移动 Web 工作台、活动转化页或移动 Web AI 助手理解。
- 避免看到手机尺寸、扫码入口或 WebView，就直接默认采用 Ant Design Mobile 而不判断页面目标。
- 让后续进入 Ant Design Mobile 前端规范和移动 Web 页面模板规范时有统一前置判断。

## 适用范围

- 适用于移动端 H5、企业内部移动 Web 工作台、轻量业务页、移动表单页、扫码进入页和移动 Web AI 助手页的场景判断。
- 适用于新项目场景归类，也适用于已有项目新增移动 Web 页面时的归类判断。
- 不替代具体技术栈规范或页面模板规范；只负责先判断移动 Web 页面属于哪一类。
- 不覆盖 Flutter、React Native、原生 iOS、原生 Android 等 App 工程场景。

## 使用原则

- 先判断页面主要目标，再判断交互复杂度和交付形态，最后才决定是否采用 Ant Design Mobile。
- 一个移动 Web 页面只应有一个主场景归类；如确实跨场景，应先确定主场景，再把次级能力作为附属模块嵌入。
- 不因为页面跑在手机里，就默认它是营销页或简化版后台。

## 四类主场景

### 业务 H5 页面

- 主要目标：发起流程、录入数据、查看结果、提交审批、扫码后完成单次业务动作。
- 典型结构：标题区、表单区、步骤区、结果区、固定底部操作区。
- 典型特征：任务明确、链路短、字段适中、按钮明显、强调单手操作和快速完成。
- 进入路径：先读 [16_AntDesignMobile前端规范](16_AntDesignMobile前端规范.md)，再按需读 [17_移动端页面模板规范](17_移动端页面模板规范.md)。

### 移动 Web 工作台页面

- 主要目标：集中查看待办、消息、统计摘要和高频快捷操作。
- 典型结构：顶部身份或组织信息区、任务卡片区、快捷入口区、列表区、底部导航区。
- 典型特征：信息密度中等、卡片化组织、多入口并存，但每个入口都应足够轻量。
- 进入路径：先读 [16_AntDesignMobile前端规范](16_AntDesignMobile前端规范.md)，再按需读 [17_移动端页面模板规范](17_移动端页面模板规范.md)。

### 移动 Web 活动或转化页

- 主要目标：报名、留资、领取权益、承接活动流量。
- 典型结构：首屏主题区、卖点区、证明材料区、表单或行动区、补充说明区。
- 典型特征：强调转化，不追求后台式高信息密度；通常比桌面官网页更短更直接。
- 进入路径：优先读 [15_移动端页面场景选型规范](15_移动端页面场景选型规范.md) 和 [17_移动端页面模板规范](17_移动端页面模板规范.md)；如项目已有官网体系，再补读 [12_AntDesignLanding官网与落地页规范](12_AntDesignLanding官网与落地页规范.md)。

### 移动 Web AI 助手页

- 主要目标：在手机浏览器或 WebView 中完成提问、检索、任务发起和结果查看。
- 典型结构：会话区、输入区、建议问题区、结果区、按需存在的快捷操作区。
- 典型特征：交互链路比桌面 AI 工作台更短，展示更聚焦，不适合承载复杂多栏工作台。
- 进入路径：先读 [11_AntDesignX智能应用前端规范](11_AntDesignX智能应用前端规范.md)，再补读 [16_AntDesignMobile前端规范](16_AntDesignMobile前端规范.md) 和 [17_移动端页面模板规范](17_移动端页面模板规范.md)。

## 快速判断问题

- 用户进入页面后，首要目标是完成一个业务动作、管理一组待办、完成一次转化，还是与 AI 做一轮轻量交互。
- 页面主体是表单和步骤、卡片和入口、内容和 CTA，还是消息流和结果反馈。
- 页面是否需要单手操作、固定底部按钮、弱化复杂表格和降低一次性信息量。
- 页面是否真的以手机浏览器或 WebView 为主，而不是桌面页面简单缩窄后的兼容视图。

## 常见混合场景处理

### 后台系统的移动 Web 补充页

- 主场景通常仍是业务 H5 或移动 Web 工作台页面。
- 不应把桌面管理后台原样压缩到手机宽度；应按移动任务链路重新组织字段、操作和反馈。
- 进入路径：先读 [16_AntDesignMobile前端规范](16_AntDesignMobile前端规范.md)，必要时补读 [02_管理后台页面通用规范](02_管理后台页面通用规范.md)。

### 官网活动在移动 Web 中的承接页

- 主场景仍是移动 Web 活动或转化页。
- 如同一活动同时有桌面官网页和手机承接页，应分别组织内容层级，不直接复制桌面区块顺序。

### 移动 Web 中的 AI 辅助能力

- 如果主体是业务办理，AI 应作为结果解释区、侧滑面板或辅助问答区附着。
- 如果主体是对话与任务发起，才归为移动 Web AI 助手页。

## 不建议的做法

- 看见手机页面就直接默认用营销页写法。
- 把桌面管理后台组件和布局整体缩放到手机端继续使用。
- 在一个移动 Web 页面里同时堆工作台、营销页和聊天页结构，导致主任务失焦。
- 把 Ant Design Mobile 当成所有 React 页面默认 UI 方案，而不先判断设备和交互边界。

## 输出要求

- 给出移动 Web 方案时，应明确写出主场景归类、判断依据和推荐进入的下一篇规范。
- 如果页面存在混合场景，应明确主场景是什么，次级场景以什么形式附着。
- 如果需求还不足以判断场景，应先补问页面目标、主要动作和目标设备，而不是直接开始出页面代码。

## 提交前自检

- 是否先判断了移动 Web 页面的主目标，而不是先选组件库。
- 是否明确了这是业务 H5、移动 Web 工作台、移动 Web 活动页还是移动 Web AI 助手页。
- 是否给出了后续应进入的技术栈规范和页面模板规范。
- 是否避免把桌面后台、官网和 AI 工作台直接缩放成一个手机页面。
