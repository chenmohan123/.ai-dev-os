# Ant Design Vue 管理后台前端规范

## 适用范围

- 适用于基于 Vue 3 和 Ant Design Vue 的企业级管理后台页面。
- 除本文件另有说明外，页面布局、页面结构、交互模式、数据处理和输出要求遵循 [02_管理后台页面通用规范](02_管理后台页面通用规范.md)。

## 技术栈约定

- 框架：Vue 3，使用 Composition API 和 script setup。
- UI 库：Ant Design Vue 4.x。
- 构建工具：Vite，使用稳定主版本。
- 路由：Vue Router 4。
- 图表：ECharts 与 vue-echarts，仅在看板页或确有图表需求时使用。
- 如无特殊说明，默认基于以上技术栈和项目现有依赖实现，不额外引入新依赖。
- 如确需新增依赖，应先说明使用理由、适用场景以及为什么现有依赖无法满足。
- 新项目应优先采用最近验证通过的稳定依赖基线，不直接沿用历史脚手架中的旧版本习惯值。
- 已有项目应与当前仓库中的 package.json 和锁文件保持一致，不随意跨主版本升级。
- 具体版本以 package.json 和锁文件为准，不使用 latest 作为规范描述。
- 如需查看当前推荐版本快照，应继续结合 [19_前端稳定版本基线清单](19_前端稳定版本基线清单.md) 评估 Vue、Vite 和 Ant Design Vue 的稳定组合。

## 视觉风格

- 应先定义当前后台的视觉方向、信息密度和 theme token，不直接把 Ant Design Vue 默认样式当作最终视觉结果。
- 整体风格保持专业、可信、清晰、高效，并允许在此基础上建立更强品牌感和页面辨识度。
- 主色、强调色、背景层级、圆角、边框和阴影应通过 ConfigProvider 的 theme token 统一配置，不强制等于官方默认蓝。
- 保持信息密度与留白平衡，避免整页大面积空白、同构白卡平铺和视觉重点缺失。
- 状态颜色语义统一：绿色表示正常或成功，红色表示异常或危险，橙色表示警告或待处理，蓝色表示进行中，灰色表示停用或维护。
- 如任务要求更高视觉品质或演示级输出，应同时遵循 [18_前端高质量视觉输出规范](18_前端高质量视觉输出规范.md)。

## 组件使用规范

- 优先使用原生 Ant Design Vue 组件，避免自定义风格过重的替代组件。
- Table 用于数据列表展示。
- Form、Input、Select、DatePicker 用于搜索筛选区和弹窗表单。
- Modal 用于字段较少的新增、编辑等简单表单操作。
- Drawer 用于详情查看或字段较多的复杂表单。
- Button 用于操作触发。
- Tag 用于只读状态标签展示。
- Switch 用于启用、停用等二元状态切换，并带明确文案。
- Popconfirm 用于删除、作废等危险操作的二次确认。
- Statistic、Card、Descriptions、Empty、Spin、message 按各自语义场景使用。

## 数据请求与状态约定

- 页面级接口请求、加载状态、错误提示和刷新逻辑，应优先遵循项目现有请求层和数据访问约定。
- 全局状态只承载跨页面共享且确有必要的状态，例如当前用户信息、权限信息、全局布局状态和少量跨页筛选条件。
- 列表结果、详情数据和表单临时态优先放在页面或领域内管理，不把所有数据都提升到全局状态。
- 新项目如需选择请求层、状态管理、表单方案或表格策略，应结合 [06_Vue管理后台技术选型规范](06_Vue管理后台技术选型规范.md) 先让用户确认。

## 权限与路由组织约定

- 路由配置、菜单配置、页面准入控制和按钮级权限判断应采用统一入口，不在页面内部重复散落实现。
- 权限体系至少区分路由权限、菜单权限和按钮权限；如项目存在数据权限要求，应单独定义数据访问边界。
- 页面级准入应在路由层或页面入口统一处理，不在页面渲染到一半后再临时跳转或隐藏主体内容。
- 按钮显隐和操作可用性应基于统一权限判断工具或权限组件实现，避免每个页面各写一套字符串判断。
- 菜单、路由和权限码之间的映射关系应集中维护，避免同一权限含义在多个文件中重复定义。

## 页面输出约束

- 页面类型判断、默认页面骨架和模板级输出要求，应遵循 [03_管理后台页面模板规范](03_管理后台页面模板规范.md)。
- 在 Vue 3 + Ant Design Vue 场景下，新增、编辑或详情交互默认优先结合 Modal、Drawer 或独立页面组织，并与字段复杂度保持一致。
- 输出代码应优先复用项目现有 Layout、请求封装、权限组件、表格封装和主题配置，不重复造壳层和基础设施。
- 如需求未明确请求层、状态管理、表单方案或表格策略，且项目中也无现有约定，应先向用户确认后再决定。
- 生成页面前，应能说明页面视觉焦点、信息主次和关键容器层级；不应只交付默认样式下的组件拼装页。

## 推荐实现顺序

- 先确认是否采用 Vue 3 + Ant Design Vue，以及是否沿用项目现有请求层、状态管理、权限体系和表单方案。
- 再结合 [03_管理后台页面模板规范](03_管理后台页面模板规范.md) 确认页面类型、页面骨架和关键交互结构。
- 再确定接口调用、查询参数、分页参数、页面状态和权限点的组织方式。
- 完成页面骨架后，再补字段渲染、状态展示、表单校验、二次确认和异常反馈。
- 最后统一检查主题风格、交互反馈、权限控制、加载态、空态和错误态是否完整。
- 如页面承担演示、样板或中后台品牌门面职责，应额外检查截图级视觉质量和跨页面风格一致性。

## 代码要求

- 代码应整洁、规范、可直接运行。
- 使用 script setup，不使用 Options API。
- 需要做本地派生展示时，可使用 ref 配合 computed 组织前端视图状态，但不要替代默认的服务端列表处理方式。
- 表单数据优先使用 reactive 管理。
- 命名语义化，避免无意义变量名。
- 主题 token、全局样式和组件外观优先复用项目现有 Ant Design Vue 配置。

## 提交前自检

- 是否先确认了新项目选型，或在已有项目中遵循了现有技术栈与工程约定。
- 是否复用了项目现有 Layout、请求封装、权限组件、表格封装和主题配置。
- 是否存在把页面数据、表单临时态或列表结果无必要提升到全局状态的情况。
- 是否补齐了加载态、空态、错误提示、成功反馈、防重复提交和危险操作二次确认。
- 是否仍存在默认白卡平铺、视觉主次不清、theme token 未收敛或页面焦点不明确的问题。
- 是否存在临时调试代码、无用状态、无用 composables、无用 import 或注释掉的旧实现未清理。
