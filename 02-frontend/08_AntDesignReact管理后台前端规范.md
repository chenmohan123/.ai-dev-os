# Ant Design React 管理后台前端规范

## 适用范围

- 适用于基于 React 和 Ant Design 的企业级管理后台页面。
- 除本文件另有说明外，页面布局、页面结构、交互模式、数据处理和输出要求遵循 [02_管理后台页面通用规范](02_管理后台页面通用规范.md)。
- 新项目如未明确指定前端框架或组件库，应先让用户确认是否采用 React + Ant Design，再继续按本规范输出。
- 已有项目如已存在 React 技术栈、组件封装、请求层、状态管理或工程约定，应优先遵循项目现状，不因本规范强行替换现有实现。

## 技术栈约定

- 框架：React，优先使用函数组件与 Hooks。
- 语言：TypeScript。
- UI 库：Ant Design 5.x。
- 构建工具：Vite，使用稳定主版本。
- 路由：React Router，使用当前项目已采用的稳定主版本。
- 图表：ECharts，仅在看板页或确有图表需求时使用。
- 请求库、缓存方案、全局状态管理、表单方案等工程依赖，属于可选技术决策；新项目应先让用户确认，已有项目应优先复用现有方案。
- 如无特殊说明，默认基于以上技术栈和项目现有依赖实现，不额外引入新依赖。
- 如确需新增依赖，应先说明使用理由、适用场景以及为什么现有依赖无法满足。
- 新项目应在初始化阶段一次性确认 React、路由、请求层、状态管理、表单方案和权限组织方式，避免边开发边切换。
- 已有项目应与当前仓库中的 package.json 和锁文件保持一致，不随意跨主版本升级。
- 具体版本以 package.json 和锁文件为准，不使用 latest 作为规范描述。

## 视觉风格

- 主色使用官方蓝 #1677FF，并通过 Ant Design 的 theme token 统一配置。
- 整体风格保持清爽、简洁、规整、专业。
- 保持留白合理，避免颜色杂乱和装饰过多。
- 状态颜色语义统一：绿色表示正常或成功，红色表示异常或危险，橙色表示警告或待处理，蓝色表示进行中，灰色表示停用或维护。

## 组件使用规范

- 优先使用原生 Ant Design 组件，避免自定义风格过重的替代组件。
- Table 用于数据列表展示；复杂表格能力应优先在现有表格封装基础上扩展，不随意并行引入第二套表格体系。
- Form、Input、Select、DatePicker 用于搜索筛选区和弹窗表单。
- Modal 用于字段较少的新增、编辑等简单表单操作。
- Drawer 用于详情查看或字段较多的复杂表单。
- Button 用于操作触发。
- Tag 用于只读状态标签展示。
- Switch 用于启用、停用等二元状态切换，并带明确文案。
- Popconfirm 用于删除、作废等危险操作的二次确认。
- Statistic、Card、Descriptions、Empty、Spin、message、notification 按各自语义场景使用。

## 数据请求与状态约定

- 页面级接口请求、缓存、重试、失效更新和加载状态，应优先遵循项目现有请求层和数据访问约定。
- 全局状态只承载跨页面共享且确有必要的状态，例如当前用户信息、权限信息、全局布局状态和少量全局筛选条件。
- 列表查询结果、详情数据和表单临时态优先放在页面或领域内管理，不把所有数据都提升到全局状态。
- 新项目若需选型请求库、缓存方案或状态管理方案，应先由用户确认，不在规范中默认绑定唯一实现。

## 目录与模块组织约定

- 目录结构可根据项目路由模式、业务规模和工程约定调整，不强制规定唯一目录名。
- 无论采用何种目录结构，都应将路由级页面、页面私有组件、复用 hooks、数据访问、工具函数、类型定义和常量配置按职责拆开。
- 页面私有实现应优先就近放置；只有在跨页面复用后，才上提到公共目录。
- 公共 UI 组件、权限组件、请求封装、表格封装和表单封装应集中沉淀，避免每个页面各自复制一套。
- 一个目录内不应同时混放页面、接口、工具、测试数据和临时脚本；职责混杂时应及时拆分。
- 新项目开始前，应先明确页面目录、路由目录、共享组件目录和领域模块边界；已有项目则沿用当前结构，不为追求统一而大范围搬迁。

## 权限与路由组织约定

- 路由配置、菜单配置、页面准入控制和按钮级权限判断应采用统一入口，不在页面内部重复散落实现。
- 权限体系至少区分路由权限、菜单权限和按钮权限；如项目存在数据权限要求，应单独定义数据访问边界。
- 页面级准入应在路由层或页面入口统一处理，不在页面渲染到一半后再临时跳转或隐藏主体内容。
- 按钮显隐和操作可用性应基于统一权限判断工具或权限组件实现，避免每个页面各写一套字符串判断。
- 菜单、路由和权限码之间的映射关系应集中维护，避免同一权限含义在多个文件中重复定义。

## 页面输出约束

- 页面类型判断、默认页面骨架和模板级输出要求，应遵循 [03_管理后台页面模板规范](03_管理后台页面模板规范.md)。
- 在 React + Ant Design 场景下，新增、编辑或详情交互默认优先结合 Modal、Drawer 或独立页面组织，并与字段复杂度保持一致。
- 生成页面代码时，应优先复用项目现有 Layout、请求封装、权限组件、表格封装和主题配置，不重复造壳层和基础设施。
- 如需求未明确请求库、缓存方案、状态管理方案或表单方案，且项目中也无现有约定，应先向用户确认后再决定，不自行假定唯一技术选型。
- 输出代码应以可直接落地为目标，不只给静态界面；必要时应同时包含类型定义、接口调用占位、状态处理和关键交互逻辑。

## 代码要求

- 代码应整洁、规范、可直接运行。
- 使用函数组件与 Hooks，不使用 class 组件作为默认实现。
- 组件职责清晰；页面负责路由级编排、页面级状态和交互组织，复用逻辑应下沉为独立 hooks、服务或工具模块。
- 需要做本地派生展示时，可使用 useState、useReducer 或派生变量组织前端视图状态，但不要替代默认的服务端列表处理方式。
- 命名语义化，避免无意义变量名。
- 主题 token、全局样式和组件外观优先复用项目现有 Ant Design 配置。
- 权限判断、路由守卫、按钮显隐和数据访问边界应集中封装，不把权限逻辑散落在各个页面节点中。

## 推荐实现顺序

- 先确认是否采用 React + Ant Design，以及是否沿用项目现有请求层、状态管理、权限体系和表单方案。
- 再结合 [03_管理后台页面模板规范](03_管理后台页面模板规范.md) 确认页面类型、页面骨架和关键交互结构。
- 再确定接口调用、查询参数、分页参数、页面状态和权限点的组织方式。
- 完成页面骨架后，再补字段渲染、状态展示、表单校验、二次确认和异常反馈。
- 最后统一检查主题风格、交互反馈、权限控制、加载态、空态和错误态是否完整。

## 提交前自检

- 是否先确认了新项目选型，或在已有项目中遵循了现有技术栈与工程约定。
- 是否复用了项目现有 Layout、请求封装、权限组件、表格封装和主题配置。
- 是否存在把页面数据、表单临时态或列表结果无必要提升到全局状态的情况。
- 是否把权限判断、菜单控制和按钮显隐集中处理，而不是散落在页面内部。
- 是否补齐了加载态、空态、错误提示、成功反馈、防重复提交和危险操作二次确认。
- 是否存在临时调试代码、无用状态、无用 hooks、无用 import 或注释掉的旧实现未清理。

## 输出目标

- 风格统一。
- 布局标准。
- 交互完整。
- 数据流清晰。
- 能直接用于实际项目。
