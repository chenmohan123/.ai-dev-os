# Electron 桌面端技术选型规范

## 目标

- 规范 Electron 桌面端在新项目和已有项目中的关键技术决策方式。
- 明确哪些技术点必须先让用户确认，哪些能力应优先复用项目现状。
- 避免在桌面端方案设计时擅自拍板渲染框架、构建工具、打包方案、持久化方案和自动更新策略。

## 适用范围

- 适用于 Electron 桌面端项目的技术选型、方案确认和实现前决策。
- 涉及主进程、preload、渲染进程、安全边界和桌面能力约束时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及目录设计、结构治理和分层落地时，应结合 [13_Electron桌面端目录结构规范](13_Electron桌面端目录结构规范.md) 使用。
- 涉及自动更新策略、灰度、回滚和发布节奏时，应结合 [17_Electron桌面端自动更新规范](17_Electron桌面端自动更新规范.md) 使用。
- 涉及权限边界、外链控制、协议注册和敏感信息保护时，应结合 [18_Electron桌面端安全规范](18_Electron桌面端安全规范.md) 使用。

## 总体原则

- 新项目如未明确渲染框架、构建工具、打包方案、状态管理、持久化方案、自动更新方案或测试方案，应先列出候选并让用户确认，不替用户拍板。
- 已有项目如已存在稳定的 Electron 工程结构、构建链路、打包产物、更新链路和日志方案，应优先复用项目现状。
- 选型优先保证长期维护性、桌面端安全边界、发布稳定性和团队认知成本，而不是只追求单点新潮。
- 同一类能力只保留一套主方案，避免同时存在两套打包器、两套 IPC 桥接体系或两套本地存储主方案。

## 新项目决策要求

- 在进入实现前，至少确认以下内容：渲染框架、构建工具、Electron 工程组织方式、打包方案、安装包目标平台、自动更新策略、持久化方案、日志方案、测试方案。
- 若用户未指定渲染框架，应先确认使用 React、Vue、Svelte 还是轻量原生页面，不默认选择。
- 若用户未指定构建链路，应先确认采用 electron-vite、Electron Forge、手工 Vite 配置或现有团队模板，不默认绑定唯一方案。
- 若用户未指定打包和分发方式，应确认使用 Electron Builder、Forge 打包链路或团队已有发布平台。

## 已有项目决策要求

- 先检查当前项目的 package.json、锁文件、主进程入口、preload 组织方式、渲染层框架、打包脚本、更新链路和安装产物。
- 如现有方案可满足需求，应直接复用，不新增第二套同类能力。
- 只有在现有方案明显无法满足跨平台构建、自动更新、安全要求或开发效率时，才讨论替换或升级。
- 讨论替换时，应明确迁移范围、兼容成本、平台影响和回退方式，不做隐式迁移。

## 关键能力选型边界

### 渲染框架

- React：适合已有 React 积累、桌面端页面复杂、状态交互较多的场景。
- Vue：适合团队以 Vue 为主、希望复用现有 Vue 页面经验的场景。
- 轻量原生页面或极简框架：适合页面数量少、交互简单、桌面壳层能力占主导的工具类应用。
- 如已有 Web 端可复用模块，应优先考虑与既有前端栈保持一致，降低共享成本。

### 构建与开发链路

- electron-vite：适合追求主进程、preload、渲染进程统一开发体验的新项目。
- Electron Forge：适合希望使用官方生态脚手架、插件化构建与分发流程的团队。
- 项目现有自定义构建链路：适合已有项目且脚本、打包、CI 已围绕现有方案稳定运行的场景。
- 如需多平台签名、复杂 native 依赖或公司内部分发流程，应把这些约束纳入构建方案选择，而不是只看开发体验。

### 打包与分发方案

- Electron Builder：适合需要较完整安装包能力、自动更新适配和多平台分发的项目。
- Forge 自带打包链路：适合与 Forge 工程深度配套、发布需求相对标准的项目。
- 团队已有企业分发方案：如公司已有制品平台、私有更新服务或安装包签名流程，应优先复用。
- 安装包目标格式如 nsis、msi、dmg、pkg、AppImage 等，应由目标平台和分发渠道决定，不默认一刀切。

### 自动更新策略

- 不需要自动更新：适合内网工具、低频版本发布或手工安装可接受的场景。
- 私有更新服务：适合企业内部分发、对版本节奏和权限控制要求高的场景。
- 公共制品或云存储更新：适合标准化桌面产品，但需提前确认签名、权限、回滚和灰度策略。
- 自动更新是否启用、何时检查、是否静默下载、何时提示重启，必须由用户或产品规则确认。

### 本地持久化方案

- 配置类数据：可采用受控 JSON、YAML 或应用配置封装。
- 结构化业务数据：可采用 SQLite 或团队已有本地数据库方案。
- 轻量本地状态：可采用 IndexedDB 或渲染层状态持久化方案，但不应用于敏感信息长期存储。
- 凭证和敏感信息：优先采用系统安全存储或安全封装，不落明文。

### 日志与诊断方案

- 轻量日志模块：适合工具类应用或诊断诉求较轻的项目。
- 分级日志与崩溃收集：适合桌面端生命周期长、需要定位现场问题的项目。
- 只要桌面能力接入较多，就不应省略主进程日志、IPC 失败日志和关键动作审计。

### 测试方案

- 单元测试用于桥接函数、纯逻辑模块、配置处理和数据转换。
- 集成测试用于 IPC 协议、主进程与 preload 协作、本地存储和更新流程边界。
- 端到端测试用于窗口流程、菜单/托盘行为、核心业务路径和安装包冒烟。
- 是否引入 Playwright、Vitest、Jest 或项目现有测试方案，应由项目现状和团队习惯决定。

## 常见组合建议

- 桌面中后台型：Electron + 用户确认的前端框架 + 统一 IPC 桥接 + 结构化日志 + 明确持久化方案。适合页面较多、业务交互较重的桌面应用。
- 工具壳层型：Electron + 轻量渲染层 + 受控 preload + 简洁打包链路。适合小型工具、配置助手和系统辅助类应用。
- 现状增强型：沿用已有渲染栈、打包链路、更新方案和目录结构，只补缺能力。适合已有项目增量开发。

## 快速决策矩阵

- 如果是新项目，且团队已有稳定 React 或 Vue 积累，应优先沿用该前端栈，再确认 Electron 工程和发布链路。
- 如果是新项目，且目标是快速落地标准桌面应用，应优先选择成熟的统一构建链路，不默认自研复杂脚本。
- 如果是已有项目，且打包、签名、更新、日志和目录结构已稳定运行，应优先现状增强，不轻易更换主方案。
- 如果项目需要强依赖本地文件、系统权限、深链接、托盘、自动更新和跨平台分发，应把这些约束提前放入决策，而不是编码后补洞。

## 选型提问模板

- 这是新项目还是已有项目。
- 如果是已有项目，当前使用的渲染框架、构建工具、打包方案和自动更新方案分别是什么。
- 目标平台是 Windows、macOS、Linux，还是其中一部分。
- 是否需要托盘、菜单、快捷键、文件系统访问、深链接、自动启动、自动更新等桌面能力。
- 是否需要离线能力、本地数据库、配置持久化或敏感凭证安全存储。
- 团队已有的 Electron、React、Vue、打包签名和测试工具积累分别是什么。
- 本次是增量迭代，还是准备调整现有工程基础设施。

## 选型结论输出要求

- 输出结论时，应明确区分推荐方案、备选方案、不建议方案三类信息。
- 每个候选方案至少说明适用场景、主要收益、主要限制和迁移或维护成本。
- 如为新项目，应明确哪些点已由用户确认，哪些点仍待确认，不把待确认项写成既定事实。
- 如为已有项目，应明确哪些能力直接复用现状，哪些能力只做增量增强，哪些能力确实需要调整。
- 最终结论应能直接指导下一步进入工程初始化、窗口能力落地或目录分层设计。

## 提交前自检

- 是否先区分了新项目选型和已有项目复用两种场景。
- 是否先检查了现有构建、打包、更新和目录结构，而不是直接引入新方案。
- 是否对渲染框架、构建工具、打包方案、持久化方案和更新策略说明了收益、限制和成本。
- 是否避免在用户未确认时默认绑定唯一框架或唯一构建发布方案。
- 是否避免同一项目中长期并存多套打包链路、桥接体系或本地存储主方案。
