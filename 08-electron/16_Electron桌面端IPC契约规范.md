# Electron 桌面端 IPC 契约规范

## 目标

- 统一 Electron 桌面端 IPC 通道、桥接 API、请求响应结构和事件订阅模型。
- 让主进程、preload 和渲染进程之间的交互具备稳定边界和可演进契约。
- 避免 IPC 长期退化为随意字符串、隐式返回对象和无法回收的事件监听堆积。

## 适用范围

- 适用于 Electron 桌面端中的 ipcMain、ipcRenderer、contextBridge 以及跨窗口事件通信设计。
- 涉及桌面端安全边界、桥接原则和高权限能力暴露时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及目录放置和 shared 契约位置时，应结合 [13_Electron桌面端目录结构规范](13_Electron桌面端目录结构规范.md) 使用。
- 涉及最小工程骨架和桥接模块初始化时，应结合 [15_Electron桌面端脚手架模板规范](15_Electron桌面端脚手架模板规范.md) 使用。

## 总体原则

- IPC 契约优先按领域能力组织，不按技术实现细节组织。
- 通道名、请求类型、响应类型、错误模型和事件名应集中定义，不允许散落在页面和主进程多个角落。
- 渲染层只消费 preload 暴露的受控 API，不直接面向原始 ipcRenderer 编程。
- 事件订阅必须具备明确的解绑方式和生命周期，不允许长期残留匿名监听器。

## 通道命名规范

- 通道名采用领域前缀加动作语义，例如 app:getVersion、window:minimize、file:openDialog、settings:save、update:check。
- 命名应体现资源对象和动作，不使用 doSomething、commonAction、invokeAll 这类低信息量名称。
- 请求-响应通道与事件推送通道应区分命名，不混用同一字符串承担多种语义。
- 内部事件广播如必须存在，应明确标识来源和目标，例如 update:downloadProgress、sync:statusChanged。

## 契约分层要求

### shared 契约层

- 统一定义通道名常量、请求响应类型、事件载荷类型和错误码。
- shared 层不依赖具体主进程实现或渲染层页面实现。
- 共享契约一旦对外使用，就应保持稳定，破坏性变更必须显式标记。

### preload 桥接层

- 将 shared 契约映射为 window 上的受控 API，例如 window.appApi.getVersion、window.fileApi.openDialog。
- preload 负责做参数收敛、有限校验和异常归一化，不直接透传整个 ipcRenderer 对象。
- 订阅型 API 应返回取消订阅函数，或提供显式 off 方法。

### 主进程处理层

- 主进程只注册已声明的通道，不临时动态拼接任意通道名。
- handle 用于请求-响应场景；事件型 on/send 只用于明确需要推送的状态变化。
- 处理函数应明确参数、返回值和错误映射，不把原始异常对象直接抛给渲染层。

## 请求响应模型

- 同一类 IPC 调用应保持统一响应模型，避免一个接口返回原始数据，另一个接口返回 data 包裹结构。
- 推荐统一为成功结果或明确错误，不返回语义模糊的 null、undefined 或真假值混合判断。
- 如项目需要包裹响应对象，应至少包含 success、data、errorCode、message 这类稳定字段。
- 对文件路径、系统权限、窗口 ID、版本信息和任务状态这类核心桌面字段，应有明确类型定义，不使用 any 或松散对象。

## 错误模型规范

- IPC 错误应至少区分参数非法、权限不足、资源不存在、环境异常、系统错误、用户取消。
- 错误码和错误文案应集中定义，避免主进程和渲染层各自写一套近义错误字符串。
- 用户取消文件选择、取消更新或主动关闭对话框，不应一律视为系统失败。
- preload 层应把底层异常转换为稳定错误结构，避免把堆栈、绝对路径或敏感系统信息直接暴露给页面。

## 事件订阅规范

- 事件订阅型 API 必须说明触发时机、载荷结构、订阅返回值和销毁时机。
- 页面或 hooks 中订阅事件时，必须在卸载或窗口销毁时解绑。
- 不允许在多个页面内重复注册全局事件监听，而没有统一状态源或去重机制。
- 高频事件如下载进度、同步进度、文件监听变化，应明确节流或合并策略，避免把渲染层刷爆。

## 常见契约模板

### 请求-响应型

- app:getVersion：读取当前应用版本。
- settings:load：读取配置。
- settings:save：保存配置。
- file:openDialog：打开文件选择对话框。

### 事件推送型

- update:downloadProgress：推送更新下载进度。
- sync:statusChanged：推送同步状态变化。
- window:focusChanged：推送窗口焦点状态变化。

## 不建议的做法

- 直接在渲染层随处写 ipcRenderer.invoke('xxx')，没有桥接 API 和共享类型。
- 用一个通用通道承担几十种动作，再靠 action 字段二次分发。
- 请求响应和事件订阅混用同一通道名。
- 监听器注册后不解绑，或把解绑责任交给调用方自行猜测。
- 错误返回完全不稳定，有时抛异常，有时返回 false，有时返回 message 字符串。

## 输出要求

- 输出 IPC 设计时，应至少给出通道命名、共享类型位置、preload API 形态和主进程注册边界。
- 如项目已有成熟 IPC 封装，应优先复用现有模式，并补齐命名、类型和销毁约束。
- 如需求同时包含多个桌面能力，应分别定义各自契约，不把全部能力塞进单一桥接对象。

## 提交前自检

- 是否为通道名、请求响应结构、错误码和事件载荷提供了统一定义位置。
- 是否避免了渲染层直接依赖原始 ipcRenderer。
- 是否为所有订阅型 API 补齐了解绑和生命周期约束。
- 是否保证错误模型稳定、可判定，并且没有泄露底层敏感信息。
