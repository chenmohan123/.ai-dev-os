# Electron 桌面端目录结构规范

## 目标

- 规范 Electron 桌面端项目的目录组织方式、进程边界和文件放置原则。
- 降低主进程、preload、渲染进程、共享模块、桌面能力和发布脚本长期混杂的风险。
- 让新项目能够快速建立清晰结构，让已有项目能够判断哪些目录应继续沿用、哪些应调整。

## 适用范围

- 适用于 Electron 桌面端项目的目录设计、模块拆分和结构治理。
- 涉及通用桌面端实现约束时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及构建、打包、更新、渲染框架和持久化等选型判断时，应结合 [12_Electron桌面端技术选型规范](12_Electron桌面端技术选型规范.md) 使用。
- 涉及最小工程骨架、初始化输出结构和推荐目录模板时，应结合 [15_Electron桌面端脚手架模板规范](15_Electron桌面端脚手架模板规范.md) 使用。
- 涉及本地配置、缓存、数据库和迁移资产放置时，应结合 [20_Electron桌面端本地存储与数据迁移规范](20_Electron桌面端本地存储与数据迁移规范.md) 使用。

## 核心原则

- 目录按进程边界、职责边界和业务边界组织，不按个人习惯随意堆放。
- 主进程、preload、渲染进程、共享协议、桌面能力模块、构建脚本和发布资源应清晰分层。
- 页面私有实现优先就近放置；跨窗口或跨页面复用后再上提到公共目录。
- 同一类基础设施只保留一套主入口，例如 IPC 注册、窗口管理、日志能力、配置管理和更新模块。
- 结构设计优先服务长期维护，不为了短期方便把高权限能力和页面实现混放。

## 新项目目录设计要求

- 新项目开始前，应先确定主进程、preload、渲染层、共享模块和发布资源的目录边界，再进入大规模开发。
- 如果用户未指定目录结构，应先给出候选方案并让用户确认，不替用户拍板唯一模板。
- 新项目应至少明确以下边界：主进程入口、preload 入口、渲染层目录、共享协议目录、窗口管理目录、桌面能力目录、构建发布目录。
- 目录层级应保持可读，避免刚起步就形成深层嵌套和跨进程相互引用。

## 已有项目结构判断要求

- 已有项目优先检查当前主进程入口、preload 放置方式、IPC 注册入口、窗口模块、渲染页面目录、打包脚本和发布资源位置。
- 如当前结构已能支撑新增能力开发，应优先沿用现状，不因理想模板强行整体搬迁。
- 只有在职责明显混杂、跨进程依赖失控、同类能力重复分散或发布链路混乱时，才讨论调整。
- 讨论调整时，应优先做局部收敛和渐进迁移，而不是一次性整体翻目录。

## 推荐目录边界

### 主进程目录

- 放应用启动入口、单实例控制、窗口创建、菜单、托盘、协议处理、自动更新和 IPC 注册。
- 主进程目录负责桌面壳层编排，不承载渲染层页面代码。
- 主进程内的业务编排、系统能力和生命周期管理可按模块继续拆分，但不应散落到页面目录。

### preload 目录

- 放 contextBridge 暴露逻辑、桥接 API 定义、参数校验和事件订阅封装。
- preload 目录只承载桥接层职责，不承载页面组件、窗口生命周期或主业务流程。
- 如果存在多个窗口或多个 preload 入口，应按窗口或能力明确拆分，而不是在单个超大 preload 文件中堆积。

### 渲染层目录

- 渲染层目录按业务域、窗口域或路由域组织，不按杂乱的组件类型随意拆散。
- 每个窗口页面只承载页面编排、页面状态和交互组织，不直接堆积桌面高权限逻辑。
- 页面私有组件、hooks、样式、类型和测试文件应优先与页面同目录放置。

### 共享协议与类型目录

- 主进程、preload 和渲染层共用的 IPC 通道名、请求响应类型、事件协议和错误码应集中沉淀到共享目录。
- 共享目录只放跨进程共享的稳定契约，不放任一侧私有实现。
- 通道命名、返回结构和错误枚举应保持集中管理，避免多处定义后逐渐漂移。

### 桌面能力目录

- 文件系统、通知、剪贴板、系统托盘、菜单、快捷键、协议处理、更新和安全存储等桌面能力，应按能力模块组织。
- 高权限模块的底层实现优先靠近主进程或 preload，不应散落在渲染层页面中。
- 同一能力只保留一个主入口，不让多个页面私自复制封装。

### 配置、日志与持久化目录

- 应用配置、环境映射、日志写入、缓存目录策略和持久化封装应集中管理。
- 配置模板、默认值、迁移逻辑和环境差异应归拢在统一位置，不分散在页面和脚本中。
- 敏感配置与普通配置应区分存放和读取方式。

### 构建与发布目录

- 打包配置、签名脚本、安装包资源、图标、协议注册资源和发布脚本应集中在构建发布目录。
- 构建目录只承载打包和发布相关资源，不混入业务逻辑实现。
- 不同平台的发布资源应按平台拆分，避免同目录下长期堆积难以辨认的产物配置。

## 推荐目录模板

- 可采用“main + preload + renderer + shared + desktop + config + build”的组织思路。
- 一个可接受的模板通常包含以下层次：应用入口、窗口管理、IPC 注册、preload 桥接、业务页面、共享协议、桌面能力、配置日志、构建发布资源。
- 如果项目规模较小，可以适当合并部分目录，但仍应保持高权限能力与页面实现分离。
- 如果项目规模较大，可以在窗口域或业务域下继续细分页面、领域逻辑和私有组件，但不要无限制纵深嵌套。

## 结构失衡信号

- 一个目录同时混放主进程代码、preload 桥接、页面组件、打包脚本和测试数据。
- 通道名、共享类型、桥接 API 和错误码分散在多个不一致位置。
- 多个页面各自复制一套窗口控制、文件读写或通知能力封装。
- 单个主进程文件既有窗口创建、菜单逻辑、IPC 注册、自动更新、系统能力和业务流程，已经难以维护。
- 打包、签名、图标和平台资源散落在根目录或业务目录中。
- 历史 preload 文件、旧版窗口目录、废弃脚本或临时安装产物长期留在正式结构中。

## 调整策略

- 优先先收敛 IPC 主入口、窗口管理和高权限桌面能力，再处理渲染层页面私有实现迁移。
- 优先局部移动和逐步重命名，不一次性大范围搬迁所有目录。
- 每次调整都应同步更新引用、构建脚本和发布配置，避免旧路径与新路径并存。
- 如结构调整涉及多个窗口或平台发布链路，应先明确迁移范围和落地顺序。

## 提交前自检

- 是否明确区分了主进程、preload、渲染层、共享协议和构建发布目录的职责。
- 是否存在高权限能力直接混入页面目录，或页面私有实现被过早提升到共享目录的情况。
- 是否存在重复 IPC 注册入口、重复桥接 API、重复窗口管理或重复日志封装。
- 是否有目录命名模糊、用途不清、历史遗留未清理的问题。
- 是否在已有项目中做了不必要的大范围结构迁移。
