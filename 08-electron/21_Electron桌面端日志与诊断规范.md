# Electron 桌面端日志与诊断规范

## 目标

- 统一 Electron 桌面端中主进程、preload、渲染进程和桌面基础设施的日志记录与诊断输出方式。
- 避免日志只在局部临时打印，无法支撑线上问题归因、更新失败排查和本地数据损坏诊断。
- 让错误定位、现场恢复和问题上报具备稳定的一致口径。

## 适用范围

- 适用于 Electron 桌面端中的运行日志、错误日志、审计日志、诊断导出、问题排查信息和用户反馈包设计。
- 涉及桌面端通用规则、进程职责和错误处理边界时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及本地目录、日志落盘和保留策略时，应结合 [20_Electron桌面端本地存储与数据迁移规范](20_Electron桌面端本地存储与数据迁移规范.md) 使用。
- 涉及自动更新失败、回滚排查和版本状态诊断时，应结合 [17_Electron桌面端自动更新规范](17_Electron桌面端自动更新规范.md) 使用。
- 涉及敏感信息脱敏和安全边界时，应结合 [18_Electron桌面端安全规范](18_Electron桌面端安全规范.md) 使用。
- 涉及崩溃恢复、故障分级、止血动作和应急处置闭环时，应结合 [28_Electron桌面端故障恢复与运维处置规范](28_Electron桌面端故障恢复与运维处置规范.md) 使用。

## 核心原则

### 1. 日志必须分层而不是混写

- 主进程、preload、渲染进程和桌面基础设施应至少具备可区分的日志来源。
- 不能把所有异常都混成一类字符串打印到控制台。
- 只要问题需要跨进程排查，就应能从日志中看出来源、时间线和影响范围。

### 2. 日志应服务排查，不应制造噪音

- 日志字段应聚焦行为、状态、结果和失败原因，不堆砌低价值重复输出。
- 高频事件应做级别控制、合并或采样，避免把关键日志淹没。
- 调试日志、临时日志和开发期追踪信息不应无差别长期进入正式日志流。

### 3. 诊断信息默认可导出、可脱敏

- 对关键故障应能够导出最小诊断包，例如版本、运行环境、日志片段、更新状态和存储结构摘要。
- 导出的诊断信息应默认脱敏，不暴露令牌、私钥、完整用户路径或认证头。
- 诊断导出应有边界，不把整个用户数据目录无差别打包输出。

## 日志分层要求

### 主进程日志

- 记录应用启动、单实例恢复、窗口创建销毁、托盘行为、协议处理、更新流程和高权限能力调用结果。
- 主进程日志应优先支撑生命周期和系统级问题排查。
- 关键错误至少包含时间、模块、动作、参数摘要和失败结果。

### preload 日志

- 记录桥接 API 调用入口、参数校验失败、事件订阅异常和关键桥接失败结果。
- preload 日志不应直接泄露页面原始敏感输入或完整底层异常堆栈。
- 只要桥接层做了参数转换、异常归一化或权限拦截，就应有可定位日志。

### 渲染进程日志

- 记录页面初始化失败、关键交互失败、状态恢复失败和 IPC 调用异常。
- 页面级日志应与页面路由、窗口上下文或业务模块关联，避免只输出无上下文 message。
- 纯界面细节日志不应长期保留为正式日志资产。

### 桌面基础设施日志

- 配置读取、缓存清理、数据库迁移、文件导入导出、更新状态和系统命令调用应有独立模块日志。
- 对长任务应记录开始、进度、结束、失败和取消状态，而不是只记录最终结果。

## 日志级别要求

- 至少区分 debug、info、warn、error 四类级别。
- debug 用于开发和临时排查，不默认在正式环境全量开启。
- info 用于关键行为节点和状态变更。
- warn 用于可恢复但值得关注的问题，例如重试、降级、权限受限。
- error 用于明确失败且需要排查的问题。

## 关键日志场景

- 应用启动和退出。
- 单实例恢复和深链接分发。
- 主窗口和副窗口创建、恢复、销毁。
- 更新检查、下载、安装和回滚。
- 本地配置读取、写入和迁移。
- 文件导入导出、系统命令调用和原生能力失败。
- 数据库迁移、数据损坏和恢复动作。

## 诊断导出要求

- 诊断包至少应包含应用版本、平台、运行模式、最近关键日志片段和模块状态摘要。
- 如涉及更新问题，应附带更新状态、渠道信息和最近一次更新失败原因。
- 如涉及本地数据问题，应附带配置版本、数据库 schema 版本和迁移状态摘要，而不是直接打包全部数据。
- 诊断导出应明确导出范围、文件命名和用户确认行为。

## 脱敏要求

- 日志中不得直接输出令牌、密码、私钥、完整认证头、敏感下载地址凭证。
- 用户文件路径、邮箱、设备标识等信息如非必要，应做掩码或摘要化处理。
- 错误堆栈如包含绝对路径或系统敏感信息，应在导出前做清洗。

## 不建议的做法

- 只依赖控制台输出，应用退出后完全无日志可查。
- 每个模块都自行定义一套日志格式，无法按时间线和来源统一排查。
- 出问题时才临时加日志，平时没有关键链路日志。
- 把完整用户数据、认证信息或整个应用目录直接作为诊断包导出。

## 输出要求

- 输出日志方案时，应至少说明日志分层、级别策略、关键场景、落盘位置和诊断导出方式。
- 如项目已有日志模块，应优先复用现有实现，再补齐分层、脱敏和诊断能力。
- 如某些日志仅在开发或灰度环境开启，应显式说明开关和范围，不写成默认事实。

## 提交前自检

- 是否区分了主进程、preload、渲染进程和桌面基础设施日志来源。
- 是否为更新、本地存储、窗口协同和原生能力接入补齐了关键诊断日志。
- 是否为日志导出和问题反馈设计了最小必要诊断包。
- 是否完成了敏感信息脱敏，而不是直接输出原始现场数据。
