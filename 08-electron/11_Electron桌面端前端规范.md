# Electron 桌面端前端规范

## 适用范围

- 适用于基于 Electron 的桌面端应用、桌面壳层、渲染进程页面和桌面能力接入场景。
- 本文档定义 Electron 桌面端的共性约束，不绑定 React、Vue、Svelte 或原生 HTML 技术栈。
- 如任务涉及新项目技术选型、已有项目方案判断或关键工程依赖确认，应结合 [12_Electron桌面端技术选型规范](12_Electron桌面端技术选型规范.md) 使用。

## 核心原则

### 1. 主进程、preload、渲染进程职责分离

- 主进程负责应用生命周期、窗口创建、菜单、托盘、权限申请、系统能力调度和跨窗口协调。
- preload 负责暴露经过收敛和白名单控制的桥接能力，不承载复杂业务流程。
- 渲染进程负责页面展示、用户交互、页面状态和业务编排，不直接调用 Node.js 或 Electron 高权限 API。
- 不允许把文件系统、系统命令、原生能力或敏感配置读取逻辑直接堆在渲染层页面中。

### 2. 桌面能力必须经由受控桥接暴露

- 渲染层访问文件、系统通知、剪贴板、菜单、窗口控制、系统托盘和其他桌面能力时，必须通过 preload 暴露的显式 API 进入。
- preload 暴露的 API 应按业务能力或资源能力分组，不以随意透传 ipcRenderer 或 shell 对象的方式开放。
- 每个桥接方法都应定义输入、输出、失败场景和权限边界，不提供“大而全”的万能调用入口。

### 3. 默认安全优先

- BrowserWindow 默认启用 contextIsolation，默认关闭 nodeIntegration，默认开启 sandbox；如确需例外，必须说明原因和补偿措施。
- 不允许在渲染层拼接和执行不可信脚本，不允许直接渲染未经处理的远端 HTML。
- IPC 通道名、事件名和桥接 API 名称应语义清晰，避免过于通用导致误用或权限扩散。
- 远端内容加载、外链打开、文件读写、协议注册和下载执行都应经过显式校验与白名单控制。

### 4. 新项目优先采用最近验证通过的稳定基线

- Electron 新项目、桌面端模板仓库和演示仓库，不应长期沿用明显过时的 Electron、构建链路和打包工具版本。
- 如需具体版本快照，应继续结合 [35_Electron稳定版本基线清单](35_Electron稳定版本基线清单.md) 判断。
- 已有项目默认优先复用现状，只有在现有版本已影响能力、维护性或安全要求时才讨论升级。

### 5. 桌面端界面不只求能用

- 桌面端窗口页、设置页、工作台页和托盘相关交互页，除功能正确外，还应具备明确视觉层级、任务焦点和产品辨识度。
- 若渲染层承担产品门面或演示职责，不应交付默认组件拼装风格的普通页面。
- 桌面端体验质量的总口径应继续结合 [../01-general/09_多终端体验与高质量输出规范](../01-general/09_多终端体验与高质量输出规范.md) 理解，而不是把高质量输出误当成只属于浏览器页面的要求。

## 进程边界规范

### 主进程

- 负责应用启动、单实例控制、窗口生命周期、菜单、托盘、协议处理、自动启动、自动更新和系统集成入口。
- 负责统一注册 IPC handler、权限申请、深链接处理和跨窗口广播。
- 不在主进程中直接堆积页面业务规则；业务编排应下沉到独立模块。

### preload

- 只暴露最小必要能力，例如窗口控制、文件选择、日志上报、通知触发和有限的系统信息读取。
- 使用 contextBridge 暴露稳定 API，避免把 Electron 原生对象直接挂到 window 上。
- 输入参数应做基本校验，返回结构应保持稳定，避免把底层异常原样暴露给页面。

### 渲染进程

- 负责页面结构、交互反馈、表单处理、数据展示和与桥接 API 的有限协作。
- 页面只依赖已暴露的桥接 API，不直接依赖 Electron 模块或 Node.js 全局对象。
- 渲染层涉及高频 IPC 调用时，应封装领域服务或 hooks，不把事件监听和资源清理散落在各个页面组件内。

## IPC 设计规范

- IPC 按领域能力组织，例如 app、window、file、settings、update、notification，不使用杂糅的大仓库式命名。
- invoke/handle 用于请求-响应型调用；事件型通信仅用于明确需要订阅推送的场景。
- 通道协议应显式定义请求参数、返回结构、错误码或失败原因，避免返回随意对象。
- 渲染层订阅事件后必须提供解绑和销毁机制，避免窗口关闭或页面切换后残留监听器。
- 主进程处理 IPC 时应校验来源窗口、参数合法性和当前运行环境，不默认信任任意请求。

## 窗口与桌面交互规范

- 窗口创建参数、最小尺寸、标题栏模式、是否可调整大小、关闭行为和恢复行为应集中定义，不分散在多个文件重复声明。
- 若支持多窗口，应明确主窗口、副窗口、模态窗口和无边框窗口的职责，不让各类窗口随意复制生命周期代码。
- 托盘、菜单、快捷键、通知和深链接应统一由主进程或桌面壳层模块管理，不在页面代码中各自注册。
- 最小化到托盘、关闭到后台、启动恢复窗口位置等行为必须明确规则，并与用户预期保持一致。

## 本地资源与持久化规范

- 配置、缓存、用户数据、日志和临时文件应区分目录与生命周期，不混放在同一位置。
- 需要持久化的数据应明确存储策略，例如配置文件、SQLite、IndexedDB、系统安全存储或受控本地文件。
- 敏感信息优先使用系统安全存储或安全封装，不直接明文写入普通配置文件。
- 下载目录、导出目录和临时目录应可配置或至少可追踪，不写死在不可见路径中。

## 错误处理与可观测性

- 主进程、preload 和渲染进程都应有独立可定位的错误日志，不把所有异常混成一类前端提示。
- 用户可见错误提示应聚焦当前操作结果；排查信息、错误堆栈和上下文应写入日志或诊断通道。
- IPC 调用失败时，应区分权限问题、参数问题、资源不存在、环境异常和系统错误。
- 长任务、下载、同步、导出和自动更新等流程应提供状态反馈与可追踪日志。

## 打包、发布与环境约束

- 开发环境、测试环境和生产环境的资源路径、配置读取和静态文件定位方式应保持清晰一致。
- 不允许在业务代码中硬编码开发路径、构建产物路径或机器本地绝对路径。
- 安装包、自动更新、代码签名、平台差异处理和本地依赖分发方式应在方案阶段明确，不在临近发布时临时拼装。
- 如项目需要同时支持 Windows、macOS、Linux，应显式列出平台差异点和例外能力，不默认所有桌面能力跨平台一致。

## 输出约束

- 输出 Electron 桌面端代码时，应至少包含主进程入口、preload 桥接、渲染层调用方式和基础异常处理说明。
- 如项目已有现成窗口管理、IPC 封装、日志模块、更新模块或配置模块，应优先复用现有实现。
- 如需求未明确渲染框架、构建工具、打包方案、自动更新方式或持久化方案，应先向用户确认后再进入具体实现。
- 如渲染层页面承担工作台、设置中心或演示职责，应同时说明页面视觉焦点和关键交互层级，而不是只说明进程边界。

## 提交前自检

- 是否明确区分了主进程、preload 和渲染进程的职责。
- 是否避免在渲染层直接使用 Node.js 或 Electron 高权限 API。
- 是否对 IPC 通道、桥接 API、权限边界和失败场景做了清晰约束。
- 是否补齐了窗口生命周期、托盘、菜单、通知、深链接或自动更新等桌面能力的边界说明。
- 是否存在硬编码路径、直接暴露原生对象、无清理监听器或无日志的高风险实现。
