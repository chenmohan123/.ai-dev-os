# Electron 桌面端安全规范

## 目标

- 统一 Electron 桌面端的安全边界、权限暴露、外链控制、协议处理和敏感信息保护要求。
- 避免桌面端因为高权限运行环境而把页面逻辑、系统能力和敏感数据暴露在错误边界上。
- 让安全要求成为默认工程规则，而不是上线前临时补洞。

## 适用范围

- 适用于 Electron 桌面端中的 BrowserWindow 配置、preload 暴露、外链处理、文件访问、协议注册、本地存储和敏感信息保护。
- 涉及桌面端通用边界、进程职责和桥接设计时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及 IPC 契约、错误模型和事件订阅时，应结合 [16_Electron桌面端IPC契约规范](16_Electron桌面端IPC契约规范.md) 使用。
- 涉及更新来源、签名和下载校验时，应结合 [17_Electron桌面端自动更新规范](17_Electron桌面端自动更新规范.md) 使用。

## 核心原则

### 1. 默认最小权限

- 渲染层默认不直接拥有 Node.js、文件系统、系统命令和 Electron 高权限 API 访问权。
- preload 只暴露最小必要能力，不透传原始对象，不提供万能调用接口。
- 页面可见能力必须先经过主进程与 preload 的白名单控制。

### 2. 默认隔离执行环境

- BrowserWindow 默认启用 contextIsolation。
- BrowserWindow 默认关闭 nodeIntegration。
- BrowserWindow 默认开启 sandbox；如需关闭，必须说明业务原因和补偿措施。
- 不应为了图省事在开发阶段关闭安全配置，然后带入生产环境。

### 3. 默认不信任输入与来源

- 不信任来自页面输入、深链接参数、文件路径、更新地址、协议参数和外部 URL 的原始值。
- 所有高权限动作都应先校验来源、参数格式、允许范围和运行上下文。
- 不允许因为是桌面应用就默认信任本地输入或内网来源。

## BrowserWindow 安全要求

- 窗口创建时应集中定义安全相关 webPreferences，不在多个文件各自散落覆盖。
- 默认禁止远端任意内容加载；如确需加载远端地址，应明确白名单、CSP 和导航控制。
- 默认禁止新窗口任意弹出；如需打开外部地址，应通过受控外链策略处理。
- 自定义标题栏、无边框窗口和拖拽区域不应成为绕开安全边界的理由。

## preload 与桥接安全要求

- preload 暴露的 API 应按能力模块拆分，例如 window、file、settings、update，不提供全局透传入口。
- 所有桥接参数应做基础校验，拒绝明显越权、非法格式或超出白名单范围的输入。
- 返回值应做结构归一化，不把绝对路径、底层异常堆栈和敏感系统信息直接暴露给页面。
- 订阅型 API 必须可解绑，避免隐式常驻监听导致信息泄露或资源泄露。

## 外链与导航控制

- 外部链接默认通过系统默认浏览器打开，不在应用内部直接无约束加载。
- 打开外链前应校验协议和域名白名单，不允许任意 javascript:、file: 或未知协议直接执行。
- 页面内跳转、深链接恢复和协议回调应区分应用内路由与外部资源，不混用同一处理逻辑。
- 导航拦截规则应集中管理，不允许页面各自写一套放行逻辑。

## 文件访问与本地资源控制

- 文件打开、保存、导入、导出和目录选择都应通过主进程受控入口执行。
- 不允许页面直接拼接绝对路径访问任意本地文件。
- 对可访问目录、可导出位置、临时文件目录和缓存目录应有明确边界。
- 处理用户选择的文件时，应区分用户取消、权限不足、路径不存在和格式非法等不同结果。

## 协议处理与深链接安全

- 注册自定义协议前，应明确协议格式、可接受参数和来源校验规则。
- 深链接参数必须做解析与校验，不允许直接拼接进入窗口路由、命令执行或文件访问路径。
- 协议回调只允许触发白名单动作，不允许透传任意 action 或脚本片段。
- 多实例或已有窗口接收深链接时，应统一在主进程收口处理。

## 敏感信息保护

- 凭证、令牌、私钥、会话密钥和高敏配置优先使用系统安全存储或受控加密封装。
- 不应把敏感信息明文写入 JSON 配置、渲染层 localStorage、日志文件或错误提示。
- 页面中展示敏感值时，应默认脱敏，不在无必要场景回显完整值。
- 日志和诊断信息应避免输出用户目录、认证头、完整下载地址凭证或内部安全配置细节。

## 执行与命令调用安全

- 如项目需要调用系统命令、脚本或原生进程，应在主进程集中封装，不让页面直接触发任意命令字符串。
- 命令参数应做白名单和转义控制，不接受渲染层直接拼接完整 shell 指令。
- 原生依赖、脚本执行和外部程序调用应明确失败边界、超时和日志记录策略。

## 不建议的做法

- 为了方便调试而长期开启 nodeIntegration 或关闭 contextIsolation。
- 直接把 ipcRenderer、shell、fs 或 child_process 挂到 window 全局对象上。
- 将外链、协议参数、文件路径或更新地址直接作为可信输入使用。
- 把敏感 token、数据库密码或签名信息写进普通配置和日志。

## 输出要求

- 输出安全方案时，应至少说明 BrowserWindow 配置、桥接 API 暴露边界、外链与协议处理策略、文件访问边界和敏感信息保护方式。
- 如项目已有安全基线，应优先复用现有策略，再补齐缺失边界和日志脱敏要求。
- 如某项安全配置因业务限制无法采用，应显式写明例外原因、影响范围和补偿措施。

## 提交前自检

- 是否默认启用了 contextIsolation，关闭了 nodeIntegration，并评估了 sandbox。
- 是否避免了渲染层直接拿到高权限原始对象。
- 是否对白名单 URL、协议参数、文件路径和桥接参数做了校验。
- 是否为敏感信息存储、展示和日志输出设置了最小暴露策略。
