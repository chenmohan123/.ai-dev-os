# Electron 桌面端脚手架模板规范

## 目标

- 为 Electron 桌面端提供一套可直接落地的最小工程骨架和初始化输出要求。
- 让新项目在 main、preload、renderer、shared、build 等核心目录上保持稳定边界。
- 避免脚手架只生成空目录或单文件入口，导致后续主进程、桥接层和渲染层职责快速混杂。

## 适用范围

- 适用于 Electron 桌面端新项目初始化、最小工程模板输出、目录骨架设计和基础模块铺设。
- 涉及主进程、preload、渲染进程、安全边界和桌面能力约束时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及技术选型、构建链路、打包方式和自动更新策略时，应结合 [12_Electron桌面端技术选型规范](12_Electron桌面端技术选型规范.md) 使用。
- 涉及目录分层、结构治理和渐进迁移时，应结合 [13_Electron桌面端目录结构规范](13_Electron桌面端目录结构规范.md) 使用。
- 涉及 IPC 契约和桥接 API 设计时，应结合 [16_Electron桌面端IPC契约规范](16_Electron桌面端IPC契约规范.md) 使用。
- 涉及新项目整体阶段安排、初始化路线和后续阅读顺序时，应结合 [32_Electron桌面端新项目落地指引](32_Electron桌面端新项目落地指引.md) 使用。

## 使用原则

- 脚手架模板用于给出最小可维护骨架，不替代用户对渲染框架、打包方案和更新方案的选择。
- 如项目已有稳定脚手架、公司模板或工程基线，应优先复用现有模板，而不是重复生成第二套主骨架。
- 模板应保证目录能直接承载业务增长，不追求极简到牺牲边界清晰度。
- 新项目输出时，应至少包含主进程入口、preload 桥接、渲染层入口、共享协议位置和构建资源位置。

## 推荐最小目录模板

```text
project-root/
  src/
    main/

      app/
      windows/

      tray/

    preload/

    renderer/

      src/
        app/

    desktop/
      config/
      logger/
      storage/
      updater/
  build/
    icons/
    installers/
    scripts/
```

## 目录职责要求

### src/main

- 放应用生命周期入口、窗口管理、IPC 注册、菜单、托盘、协议处理和系统能力编排。
- 不在 main 目录中直接承载渲染层组件、页面样式或页面私有业务逻辑。
- 如存在多个窗口，应优先在 windows 下按窗口拆分，不在 index.ts 中堆积全部生命周期代码。

### src/preload

- 放 contextBridge 暴露逻辑、桥接 API、参数校验和事件订阅封装。
- preload 入口默认只做注册和导出，不在单个入口文件中堆积全部桥接实现。
- bridges 目录按能力拆分，例如 window、file、settings、update、notification。

### src/renderer

- 放渲染层应用入口、页面、业务组件、hooks、页面服务和样式资源。
- 如果使用 React、Vue 或其他前端框架，应在 renderer/src 内继续沿用该技术栈的目录约定。
- 渲染层不直接引用 Electron 高权限对象，只依赖 preload 暴露的 API 和 shared 契约。

### src/shared

- 放主进程、preload 和渲染层共用的 IPC 通道名、类型定义、错误模型和常量。
- shared 目录只放稳定契约和共享常量，不放任一进程侧私有实现。
- IPC 契约、错误码和通道常量应优先集中在 shared/ipc 与 shared/errors 中维护。

### src/desktop

- 放桌面端横切能力，例如配置、日志、存储、安全封装、自动更新和平台适配。
- 这里承载的是桌面基础设施，不应混入页面级状态和窗口私有业务实现。
- 若项目较小，可将部分能力并入 main；但只要出现跨模块复用，就应单独沉淀。

### build 与 resources

- build 放打包脚本、平台安装包配置、图标处理和签名辅助资源。
- resources 放运行时资源、静态模板或平台相关附加文件，不放业务源码。
- 平台相关资源应按平台区分，避免脚本、图标和安装器配置长期混堆。

### tests

- 放桥接函数、IPC 协议、桌面能力模块和渲染层关键流程的测试资产。
- 不把一次性调试脚本、手工验证文件和临时产物混入 tests 正式目录。

## 最小初始化输出要求

- 必须包含一个主进程入口文件，用于应用启动和窗口创建。
- 必须包含一个 preload 入口文件，用于桥接 API 注册。
- 必须包含一个渲染层入口文件，用于页面挂载或窗口内容加载。
- 必须包含一个共享 IPC 契约位置，用于统一管理通道名与类型。
- 必须包含基础构建和运行脚本，不要求一次性补齐所有发布脚本，但至少保证本地开发可启动。

## 推荐初始化模块

- 窗口工厂模块：负责创建主窗口和基础窗口参数。
- IPC 注册模块：负责集中注册 handle、on 和内部事件分发。
- 桥接 API 模块：负责向渲染层暴露受控能力。
- 配置模块：负责应用配置加载、默认值和持久化。
- 日志模块：负责主进程和关键桌面能力的日志输出。

## 不建议的初始化方式

- 只生成一个 main.ts，主进程、preload 和渲染层逻辑全部写在同层级。
- 直接把 ipcMain、ipcRenderer、window 控制和文件系统访问散落在多个页面组件中。
- 没有 shared 契约目录，通道名和错误码全靠字符串散落维护。
- 过早引入复杂平台分支和多层脚本，但没有先把基础边界搭清楚。

## 输出要求

- 输出脚手架模板时，应给出目录树、核心入口说明和每个关键目录的职责。
- 如果用户尚未确认渲染框架、构建工具或打包方案，应在模板中保留可替换位置，不把待确认项写成既定事实。
- 如项目已有成熟模板，应优先在现有模板上补齐缺失边界，而不是另起一套结构。

## 提交前自检

- 是否补齐了 main、preload、renderer、shared、build 这些最小必要边界。
- 是否避免把高权限能力直接初始化在渲染层页面目录中。
- 是否为 IPC 契约、共享类型和错误模型预留了统一位置。
- 是否保证初始化输出能直接支撑后续窗口、桥接层和桌面能力的增量扩展。
