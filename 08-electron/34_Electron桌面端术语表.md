# Electron 桌面端术语表

## 目标

- 统一 Electron 桌面端规范体系中的高频术语口径。
- 降低团队在阅读规范、做设计评审、写交付材料和做故障复盘时的理解偏差。
- 让首次接触 Electron 分册的使用者能先建立共同语言，再进入具体规范。

## 适用范围

- 适用于 Electron 规范阅读、团队培训、方案评审、交付准备、发布运维和事故复盘场景。
- 本文档用于解释术语，不替代具体专题规范中的约束和执行要求。

## 进程与结构术语

### main

- 指 Electron 主进程。
- 负责应用生命周期、窗口创建、菜单、托盘、协议处理、系统级能力调度和跨窗口协调。
- 在本套规范中，main 不等同于“所有桌面端逻辑都放这里”，它只负责高权限和全局协调职责。

### preload

- 指 BrowserWindow 加载页面前注入的桥接层。
- 负责把受控的桌面能力以最小必要 API 暴露给渲染层。
- 在本套规范中，preload 不承载复杂业务编排，也不直接等同于“渲染层的工具库”。

### renderer

- 指渲染进程或页面运行层。
- 负责界面展示、用户交互、页面状态和业务编排。
- 在本套规范中，renderer 默认不直接接触 Node.js 或 Electron 高权限原始对象。

### shared

- 指主进程、preload、渲染层之间共享的稳定结构区域。
- 常用于放置类型定义、IPC 契约、常量枚举和跨进程共享模型。
- shared 的价值在于减少跨层各自定义一套结构而发生漂移。

## 桥接与通信术语

### 桥接 API

- 指 preload 暴露给渲染层的受控接口。
- 常见形式是文件操作、窗口控制、通知触发、配置读取、任务状态订阅等。
- 桥接 API 的重点不是“能调通”，而是输入输出、失败语义和权限边界稳定。

### IPC

- 指 Electron 进程间通信机制。
- 在本套规范中，IPC 更强调契约化设计，而不是简单的消息发送动作。
- 只要涉及跨进程请求、响应、通知和订阅，都应按 IPC 契约规范理解和设计。

### 通道

- 指 IPC 中被命名的通信入口。
- 一个通道通常对应一类稳定能力，而不是任意拼装的临时消息字符串。
- 通道命名需要语义明确，避免因命名过宽导致权限扩散。

### 订阅

- 指渲染层或其他消费方持续接收状态更新、进度变化或事件通知的方式。
- 在本套规范中，订阅必须同时关注建立、消费和解绑，不把它理解成“注册一次就不管了”。

## 安全与能力术语

### 桌面能力

- 指文件系统、剪贴板、通知、快捷键、协议处理、shell 打开、系统命令等 Electron 桌面侧能力。
- 它们属于受控能力，不应被页面直接随意调用。

### 白名单

- 指被明确允许访问、执行或暴露的对象集合。
- 常用于外链域名、文件类型、协议、命令参数、桥接 API、配置字段等边界控制。
- 在本套规范中，白名单意味着显式许可，而不是模糊地“默认大多数都能过”。

### 安全边界

- 指不同层级、不同能力、不同数据之间的访问和调用边界。
- 例如 main 与 renderer 的边界、敏感配置与普通配置的边界、外链打开与应用内处理的边界。
- 安全边界关注的是“谁能做什么”，不是单纯某个安全开关是否开启。

## 运行与交付术语

### 灰度

- 指在正式全量发布前，对有限样本或有限渠道进行逐步投放的发布方式。
- 它的重点是控制影响范围、观察问题和保留停推空间。
- 灰度不等同于“测试环境”，它仍然可能面向真实用户或真实安装路径。

### 渠道

- 指制品分发、更新推送和环境配置的目标分组，例如正式版、灰度版、测试版、内测版。
- 同一版本在不同渠道下可能对应不同更新源、开关和验收要求。

### 停推

- 指停止某个渠道继续分发当前版本或更新包。
- 常用于发现严重问题后的止血动作。
- 停推强调“先阻断继续扩大影响”，不必等完整根因分析完成后才执行。

### 回滚

- 指把版本、渠道、配置或数据状态恢复到上一稳定状态的动作。
- 在本套规范中，回滚不是单一动作，至少应区分更新回滚、制品回滚、配置回滚和数据恢复。

### 验收门禁

- 指在进入下一阶段前必须通过的关键检查点。
- 常见门禁包括测试通过、安装可用、更新链路验证、签名状态正确和关键流程可用。
- 它的意义在于阻止高风险版本继续流转，而不是作为形式化打勾项。

## 排障与运维术语

### 诊断包

- 指为定位问题而导出的最小必要信息集合。
- 可能包含版本信息、平台信息、日志片段、配置摘要、更新状态和迁移状态。
- 在本套规范中，诊断包强调最小必要和默认脱敏，不是整包打走用户数据目录。

### 止血

- 指在故障处理中优先降低影响范围、恢复基本可用性的动作。
- 例如停推、关闭高风险开关、切回稳定源、启用安全模式。
- 止血优先于完整优化和全面解释。

### 复盘

- 指对事故或发布问题进行结构化回顾的过程。
- 重点包括影响范围、时间线、根因、止血动作和防再发措施。
- 在本套规范中，复盘的目标不是追责，而是反哺测试、发布和工程治理。
