# Electron 桌面端页面模板规范

## 目标

- 统一 Electron 桌面端常见窗口页和桌面工作流的默认输出结构与交付要求。
- 让不同渲染框架在页面模板层保持一致，不重复定义同一类桌面窗口骨架。
- 避免只输出静态窗口壳子，缺少 IPC 协作、窗口状态、错误反馈和桌面行为约束。

## 适用范围

- 适用于 Electron 桌面端的窗口页模板、设置页模板、托盘工作流、更新提示流程和文件导入导出流程设计。
- 涉及进程职责、安全边界、窗口管理和桌面能力约束时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及渲染框架、构建链路、打包、更新方案和持久化等技术选型时，应结合 [12_Electron桌面端技术选型规范](12_Electron桌面端技术选型规范.md) 使用。
- 涉及目录放置、主进程与渲染进程分层以及 IPC 契约位置时，应结合 [13_Electron桌面端目录结构规范](13_Electron桌面端目录结构规范.md) 使用。

## 使用原则

- 默认按窗口类型和工作流类型组织模板，不按渲染框架重复维护多份模板说明。
- 模板用于定义窗口骨架、关键状态和基础交互，不替代具体业务字段与视觉细节设计。
- 如项目已有成熟窗口模板、设置页壳层、托盘交互或更新模块，应优先复用现有实现。
- 如需求未明确窗口类型，应先判断是主工作台窗口、设置窗口、模态窗口还是后台工作流，再进入对应模板。

## 主工作台窗口模板

### 默认结构

- 顶部标题区或工具栏区。
- 主内容区。
- 按需存在的侧边导航区、状态栏区、底部任务区。
- 与窗口生命周期相关的加载、错误和恢复提示区。

### 默认输出要求

- 默认包含窗口初始化状态、数据加载状态、窗口标题同步和基础错误反馈。
- 如果页面依赖主进程资源，应明确 preload API 调用入口、失败提示和重试方式。
- 如支持多标签、分栏、分屏或拖拽区，应明确状态来源和销毁时机，不只给静态布局。
- 如关闭窗口会中断任务、丢失草稿或影响后台流程，应明确关闭前确认规则。

## 设置页模板

### 默认结构

- 设置导航区。
- 设置内容区。
- 保存或应用操作区。
- 按需存在的重置入口、校验提示区、重启应用提示区。

### 默认输出要求

- 默认包含配置读取、默认值回填、字段校验、保存中状态和保存结果反馈。
- 修改后若需要立即生效、重载窗口或重启应用，应明确提示和后续动作。
- 配置项应按主题分组，例如通用设置、同步设置、通知设置、更新设置、安全设置。
- 敏感配置默认不明文回显；涉及凭证、密钥或受限目录时，应明确权限与展示策略。

## 模态窗口与子窗口模板

### 默认结构

- 标题区。
- 主体内容区。
- 底部操作区。

### 默认输出要求

- 默认只承载单步、短流程或主窗口的局部补充任务，不把完整业务工作台塞进模态窗口。
- 默认包含打开、关闭、确认、取消、销毁和状态回传时机。
- 子窗口关闭后，应明确是否回传结果、刷新主窗口或仅写入后台状态。
- 如窗口无边框、自定义标题栏或固定尺寸，应补齐窗口控制能力和异常退路。

## 托盘工作流模板

### 默认结构

- 托盘入口状态。
- 托盘菜单动作集。
- 与主窗口或后台任务的联动逻辑。
- 按需存在的未读状态、同步状态或任务提示区。

### 默认输出要求

- 默认明确点击托盘图标、双击、右键菜单和退出动作的行为规则。
- 默认明确托盘菜单如何打开主窗口、切换任务状态、显示通知或退出应用。
- 如果应用支持关闭到托盘、最小化到托盘或后台驻留，应明确用户提示和恢复入口。
- 托盘状态变化应与后台任务、同步状态或登录状态保持一致，不出现菜单文案和真实状态脱节。

## 更新提示流程模板

### 默认结构

- 当前版本信息区。
- 更新状态提示区。
- 更新操作区。
- 按需存在的更新日志区、重启提示区、失败重试区。

### 默认输出要求

- 默认包含检查更新、发现新版本、下载中、下载完成、安装失败和等待重启等关键状态。
- 默认明确静默检查、手动检查、强制更新和可跳过更新的差异行为。
- 若采用后台下载，应明确下载过程反馈是展示在页面、通知还是托盘状态中。
- 更新失败时应给出可执行下一步，例如重试、查看日志、稍后提醒，而不是只显示失败文案。

## 文件导入导出工作流模板

### 默认结构

- 文件选择入口。
- 当前文件或任务状态区。
- 执行操作区。
- 按需存在的预览区、结果摘要区、错误明细区。

### 默认输出要求

- 默认包含文件选择、取消、重新选择、执行中状态、成功反馈和失败反馈。
- 需要主进程参与的文件读写、路径选择和权限校验，应明确通过 preload API 进入。
- 对批量文件、超大文件或长时间导入导出任务，应明确进度反馈、中断处理和结果汇总方式。
- 导出成功后，应明确打开目录、复制路径、继续导出或关闭窗口等后续动作。

## 通知与后台任务联动模板

### 默认结构

- 任务状态源。
- 用户可见提示入口。
- 点击通知后的跳转或恢复逻辑。
- 按需存在的静默模式、勿扰模式和失败补偿入口。

### 默认输出要求

- 默认明确哪些事件触发系统通知，哪些只在窗口内提示，哪些只记录状态不打扰用户。
- 默认明确点击通知后的行为，例如打开主窗口、跳到指定页面、查看任务详情或忽略。
- 后台任务与页面状态之间应有统一状态源，不同时维护多份难以同步的提示逻辑。
- 通知失败或系统权限不足时，应有降级策略，而不是静默丢失关键反馈。

## 输出要求

- 输出 Electron 桌面端页面代码时，应包含窗口骨架、必要状态、关键 IPC 调用点和基础异常处理，而不是只输出静态界面。
- 如项目已有现成窗口壳层、设置页布局、托盘模块、更新模块或文件选择封装，应优先复用现有实现。
- 如需求同时涉及多个窗口页或工作流，应分别按对应模板组织，而不是把设置、更新、导入导出和托盘逻辑混成单一页面。

## 提交前自检

- 是否先判断了当前需求属于主工作台窗口、设置页、模态窗口、托盘流程、更新提示还是文件工作流。
- 是否补齐了对应模板要求中的关键结构、窗口状态和 IPC 协作点。
- 是否存在只输出静态窗口界面、未包含桌面行为和状态流转的情况。
- 是否在已有项目中优先复用了现有窗口模板、托盘模块、更新模块和文件能力封装。
