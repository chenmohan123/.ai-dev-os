# Electron 桌面端原生能力接入规范

## 目标

- 统一 Electron 桌面端中文件系统、剪贴板、系统通知、快捷键、协议、系统命令和其他原生能力的接入边界。
- 避免原生能力在页面内被随意调用，导致权限扩散、行为不一致和难以排查的问题。
- 让原生能力接入既满足业务可用性，也满足安全、可维护和可诊断要求。

## 适用范围

- 适用于 Electron 桌面端中的文件读写、目录选择、剪贴板访问、系统通知、全局或窗口快捷键、协议处理、shell 打开、系统命令执行等能力接入。
- 涉及桌面端通用边界、进程职责和桥接原则时，应结合 [11_Electron桌面端前端规范](11_Electron桌面端前端规范.md) 使用。
- 涉及 IPC 通道和桥接 API 契约时，应结合 [16_Electron桌面端IPC契约规范](16_Electron桌面端IPC契约规范.md) 使用。
- 涉及安全白名单、外链和权限边界时，应结合 [18_Electron桌面端安全规范](18_Electron桌面端安全规范.md) 使用。
- 涉及日志、失败排查和诊断输出时，应结合 [21_Electron桌面端日志与诊断规范](21_Electron桌面端日志与诊断规范.md) 使用。

## 核心原则

### 1. 原生能力默认经由主进程受控接入

- 渲染层不直接拿到 Electron 原生对象或 Node.js 高权限模块。
- 所有原生能力应先由主进程封装，再通过 preload 暴露最小必要 API。
- 页面只消费具体能力接口，不直接决定底层实现细节。

### 2. 一类能力只保留一个主入口

- 文件选择、剪贴板访问、通知触发、快捷键注册和系统命令调用应各自有统一入口。
- 不允许多个页面各自封装一套文件读取、通知发送或快捷键注册逻辑。
- 统一入口应负责校验、日志、错误归一化和资源清理。

### 3. 能力接入必须说明边界和失败方式

- 每个原生能力应说明调用条件、返回结构、失败场景、用户取消语义和权限限制。
- 不应把“调用失败”混成统一 false 或 message 字符串。
- 对系统差异明显的能力，应显式写明平台差异，不默认跨平台完全一致。

## 文件系统能力要求

- 文件打开、保存、目录选择、批量导入导出和文件监听应通过受控入口调用。
- 文件读写前应校验路径来源、格式和允许范围，不接收页面直接拼接的任意路径。
- 对用户取消、权限不足、路径不存在、占用冲突和格式非法应区分处理。
- 批量文件和长时间读写任务应提供进度、取消或结果汇总方式。

## 剪贴板能力要求

- 剪贴板读取和写入应只暴露业务所需格式，不默认开放任意格式读写能力。
- 如复制内容包含敏感信息，应先经过脱敏或用户显式确认。
- 剪贴板失败应返回稳定结果，不静默吞掉异常。

## 系统通知能力要求

- 系统通知应说明触发条件、文案来源、点击行为和失败降级方式。
- 通知内容应与真实状态一致，不用过期文案或固定模板误导用户。
- 如系统权限不足或平台不支持，应有应用内提示或状态栏提示作为降级方案。

## 快捷键能力要求

- 快捷键应区分窗口内快捷键和全局快捷键，不混用注册逻辑。
- 快捷键注册应集中管理，避免多个模块重复注册同一按键组合。
- 全局快捷键应谨慎启用，必须说明使用场景、冲突风险和解绑时机。
- 窗口销毁或功能关闭后，应及时注销不再需要的快捷键。

## 协议与 shell 能力要求

- 打开外部 URL、文件或系统默认应用前，应校验协议、域名或文件类型白名单。
- 自定义协议处理应统一由主进程收口，不在页面侧直接解析和执行协议动作。
- shell 打开能力不应用于绕开受控文件访问或外链策略。

## 系统命令与外部程序调用要求

- 系统命令、脚本和外部程序调用应集中封装，不允许页面传入完整命令字符串直接执行。
- 参数必须做白名单、映射或转义控制，避免命令注入和越权执行。
- 对执行超时、权限不足、找不到程序和退出码异常应有明确处理与日志。
- 只要命令调用会修改本地状态、文件或系统设置，就应保留审计日志。

## 平台差异要求

- 文件选择、通知、快捷键、默认应用打开和命令执行在 Windows、macOS、Linux 上可能存在差异，应显式说明。
- 平台不支持的能力应提供降级路径或明确标明不可用，而不是静默失败。

## 不建议的做法

- 在页面中直接 import fs、shell、clipboard 或 child_process。
- 同一原生能力在多个页面和 hooks 中各自封装一套不兼容调用方式。
- 不区分用户取消和系统失败，导致页面无法正确反馈。
- 全局快捷键长期开启且无注销逻辑，影响用户系统使用。

## 输出要求

- 输出原生能力方案时，应至少说明能力入口、桥接方式、参数边界、失败语义和平台差异。
- 如项目已有能力封装，应优先复用现有入口，再补齐白名单、日志和清理机制。
- 如某项原生能力是否启用仍待用户确认，应显式标明待确认项，不写成既定事实。

## 提交前自检

- 是否为文件系统、剪贴板、通知、快捷键、协议和系统命令定义了统一入口。
- 是否避免了渲染层直接接触高权限原始对象。
- 是否对参数、失败结果、用户取消和平台差异做了清晰区分。
- 是否为关键原生能力补齐了日志、解绑和资源清理要求。
