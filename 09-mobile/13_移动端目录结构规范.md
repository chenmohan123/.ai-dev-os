# 移动端目录结构规范

## 目标

- 规范移动应用工程的目录组织方式、模块边界和文件放置原则。
- 降低应用入口、导航、页面、原生桥接、设备能力、数据访问和发布资源长期混杂的风险。
- 让新项目能够快速建立清晰结构，让已有项目能够判断哪些目录应继续沿用、哪些应调整。

## 适用范围

- 适用于 Flutter、React Native、原生 iOS、原生 Android 及混合移动应用工程的目录设计、模块拆分和结构治理。
- 涉及移动应用工程共性边界时，应结合 [11_移动端通用规范](11_移动端通用规范.md) 使用。
- 涉及 Flutter、React Native、原生 iOS、原生 Android 及混合方案选型时，应结合 [12_移动端技术选型规范](12_移动端技术选型规范.md) 使用。
- 不适用于移动 Web、H5、活动页或 WebView 动态页的目录组织；这些场景应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。

## 核心原则

- 目录按运行边界、职责边界和业务边界组织，不按个人习惯或页面名称随意堆放。
- 应用入口、导航层、页面层、领域模块、原生能力层、数据访问层和发布资源应清晰分层。
- 页面私有实现优先就近放置；跨页面、跨模块或跨平台复用后再上提到共享目录。
- 同一类基础设施只保留一套主入口，例如导航注册、状态管理、原生桥接、埋点、配置和日志能力。
- 结构设计优先服务长期维护和跨平台演进，不为了短期开发方便把业务页面、系统能力和发布脚本混放在同一层。

## 新项目目录设计要求

- 新项目开始前，应先确定应用入口、导航、页面、原生能力、数据访问和发布资源的目录边界，再进入大规模开发。
- 如果用户未指定目录结构，应先给出候选方案并让用户确认，不替用户拍板唯一模板。
- 新项目应至少明确以下边界：应用入口、导航层、页面目录、领域模块目录、原生能力目录、数据访问目录、配置与日志目录、构建发布目录。
- 目录层级应保持可读，避免刚起步就形成深层嵌套和跨层相互依赖。

## 已有项目结构判断要求

- 已有项目优先检查当前应用入口、导航方案、页面组织方式、原生模块放置位置、共享桥接层、数据访问层和发布资源位置。
- 如当前结构已能支撑新增功能开发，应优先沿用现状，不因理想模板强行整体搬迁。
- 只有在职责明显混杂、平台边界失控、同类能力重复分散或发布链路混乱时，才讨论调整。
- 讨论调整时，应优先做局部收敛和渐进迁移，而不是一次性整体翻目录。

## 推荐目录边界

### 应用入口与宿主层

- 放应用启动入口、环境初始化、登录态恢复、全局异常处理、生命周期监听和全局配置装载逻辑。
- 宿主层负责应用启动和全局编排，不承载具体业务页面和原生能力细节实现。
- 如同时存在多平台宿主层，应明确平台入口边界，不在单一入口文件中长期堆积平台分支判断。

### 导航层

- 放导航注册、路由映射、标签页或底部导航、深链落点映射和全局返回策略。
- 导航层负责页面切换规则和入口收口，不承载具体业务逻辑。
- 深链、推送唤起和外部跳转的目标映射应统一收口到导航层，不散落在各页面生命周期中。

### 页面目录

- 页面目录按业务域、功能域或稳定导航域组织，不按零散组件类型拆成难以理解的碎片目录。
- 每个页面只承载页面编排、页面级状态和交互组织，不直接堆积大量原生桥接和数据转换逻辑。
- 页面私有组件、页面私有 hooks、局部类型和局部样式应优先与页面同目录放置。

### 领域模块目录

- 与具体业务域绑定的数据访问、状态转换、领域规则、枚举常量和复用逻辑，应优先沉淀到对应领域模块目录。
- 同一业务域的接口定义、数据映射和领域服务应尽量就近组织，减少跨目录跳转成本。
- 领域模块目录用于承载业务共性，不替代页面目录和原生能力目录。

### 原生能力与桥接目录

- 相机、相册、定位、蓝牙、推送、文件、通知、深链、分享、生物识别和系统权限等能力，应按能力模块组织。
- 原生桥接目录只承载原生能力封装、平台差异处理、错误模型和统一调用入口，不承载业务页面逻辑。
- 同一能力只保留一个主入口，不让多个页面私自复制桥接层。
- 混合方案中，如存在 Flutter / React Native 与原生模块协同，应把桥接层和业务层边界明确拆开。

### 数据访问与状态目录

- 请求客户端、鉴权处理、错误拦截、缓存策略、离线持久化和统一数据访问封装应集中在数据访问层目录。
- 状态管理目录只承载跨页面共享且确有必要的状态，不承载所有页面临时态。
- 当存在领域级数据访问封装时，应明确区分底层请求能力与业务数据访问能力的边界。

### 配置、日志与基础设施目录

- 环境配置、功能开关、日志上报、埋点、监控、主题 token、国际化和安全存储封装应集中管理。
- 配置模板、默认值、环境差异和敏感配置读取方式应归拢在统一位置，不分散在页面和脚本中。
- 日志、埋点和错误上报应有统一入口，不在业务页面中散落多套埋点或日志封装。

### 构建与发布目录

- 打包配置、签名脚本、渠道配置、图标与启动图资源、发布脚本和环境产物模板应集中在构建发布目录。
- 构建目录只承载打包和发布相关资源，不混入业务逻辑实现。
- 不同平台、不同渠道和不同环境的发布资源应按平台或渠道明确拆分，避免长期堆积难以辨认的产物配置。

## 推荐目录模板

- 可采用“app + navigation + features + native + data + shared + config + build”的组织思路。
- 一个可接受的模板通常包含以下层次：应用入口、导航层、业务页面、领域模块、原生能力、数据访问、共享组件或共享类型、配置日志、构建发布资源。
- 如果项目规模较小，可以适当合并部分目录，但仍应保持业务页面、原生能力和发布资源分离。
- 如果项目规模较大，可以在业务域下继续细分页面、领域逻辑和私有组件，但不要无限制纵深嵌套。

## 结构失衡信号

- 一个目录同时混放页面、原生模块、请求封装、配置、打包脚本和测试数据。
- 同类桥接层、状态仓库、导航配置和数据访问代码分散在多个不一致位置。
- 多个页面各自复制一套相机、定位、推送或深链处理逻辑。
- 单个入口文件既有应用启动、导航注册、权限申请、深链处理、推送唤起和业务流程，已经难以维护。
- 构建、签名、图标、渠道和环境资源散落在根目录或业务目录中。
- 历史模块、废弃桥接层、旧版页面目录或临时发布产物长期留在正式结构中。

## 调整策略

- 优先先收敛应用入口、导航层和原生能力主入口，再处理页面私有实现迁移。
- 优先局部移动和逐步重命名，不一次性大范围搬迁所有目录。
- 每次调整都应同步更新引用、桥接注册、导航映射和发布配置，避免旧路径与新路径并存。
- 如结构调整涉及多平台或多渠道发布链路，应先明确迁移范围和落地顺序。

## 提交前自检

- 是否明确区分了应用入口、导航层、页面目录、领域模块、原生能力目录和构建发布目录的职责。
- 是否存在原生能力直接混入页面目录，或页面私有实现被过早提升到共享目录的情况。
- 是否存在重复导航注册入口、重复桥接层、重复状态仓库或重复请求层封装。
- 是否有目录命名模糊、用途不清、历史遗留未清理的问题。
- 是否在已有项目中做了不必要的大范围结构迁移。
