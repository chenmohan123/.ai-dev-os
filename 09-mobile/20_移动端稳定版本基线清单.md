# 移动端稳定版本基线清单

## 目标

- 为真正的移动 App 工程提供版本治理口径，避免长期冻结在明显过时的跨端或原生构建链路上。
- 明确移动端的“最新稳定版”不应等于无条件追 latest，而应等于“当前官方稳定通道或公开稳定快照 + 团队落地验证后锁定”。

## 使用原则

- 新项目、样板工程和演示工程应优先采用当前官方稳定通道或公开稳定快照。
- 已有项目默认优先复用现状，只有在现有版本已阻碍能力、生态兼容或发布要求时才讨论升级。
- 移动端版本治理必须按工程主方案分别判断，Flutter、React Native、原生 iOS、原生 Android 不应混成一条口径。

## 快照说明

- 快照日期：2026-04-27
- 快照来源：React Native 使用 npm view 查询结果；Flutter 与原生平台建议以官方稳定通道和目标平台工具链要求为准。

## 当前推荐快照

### React Native

- React Native：0.85.2
- peer React：^19.2.3

适用场景：

- 新建 React Native 项目
- 需要前端团队优先复用 React / TypeScript 能力的项目

### Flutter

- 默认口径：采用 Flutter 官方 stable 通道的最近稳定版本，并在目标仓库完成真机构建、运行和插件兼容验证后锁定。

适用场景：

- 新建双端移动应用工程
- 以 Flutter 作为默认优先评估方案的项目

### 原生 iOS / Android

- 默认口径：采用目标平台官方稳定工具链和组织已验证的构建基线，不把单次网络查询到的最新版本直接写成团队标准。

适用场景：

- 重原生能力、重平台特性或高性能场景

## 不建议的做法

- 把“最新稳定版”等同于每次初始化都自动拉最新版本。
- 在 React Native、Flutter、原生方案之间不区分地套用同一条版本升级策略。
- 没做真机构建与发布链路验证，就把新的主版本写进模板工程。
