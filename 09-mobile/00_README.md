# 目录索引

## 目标

- 为移动端应用目录提供统一入口，明确什么时候应该按 Flutter、React Native、原生 iOS、原生 Android 或移动应用工程来理解任务。
- 把真正的移动应用工程与 02-frontend 中的移动 Web、H5 和 WebView 页面规范分开，避免入口混淆。
- 在本目录刚建立时先给出边界、进入方式和后续扩展方向，避免先建目录却没有使用说明。

## 使用说明

- 进入移动端应用规范目录后，默认先阅读本文件，再判断当前任务是做技术选型、工程结构、原生能力接入、发布交付，还是先补齐分册。
- 如任务主体是手机浏览器页面、外链页、活动页、扫码页或 WebView 中的 H5 页面，返回 [../02-frontend/00_README.md](../02-frontend/00_README.md)，不要在本目录继续展开。
- 如任务主体是微信小程序或同类平台宿主内应用，返回 [../10-miniapp/00_README.md](../10-miniapp/00_README.md)，不要在本目录继续展开。
- 如任务主体是 App 工程、原生容器、设备能力、应用发布、安装包、商店分发、深链、推送、离线能力或跨端移动应用方案，继续留在本目录。
- 如任务涉及技术选型、框架选型、组件库选型、架构选型或模板选型，先询问用户确认后再继续输出具体方案。
- 如任务需要确定移动 App 新项目、样板工程或演示工程的版本治理口径，应继续阅读 [20_移动端稳定版本基线清单](20_移动端稳定版本基线清单.md)。
- 如任务涉及移动 App 页面、关键流程或样板工程的高质量体验与视觉输出，应先补读 [../01-general/09_多终端体验与高质量输出规范](../01-general/09_多终端体验与高质量输出规范.md)。

## 什么时候先读本页

- 你还不确定当前需求到底属于移动 Web 页面，还是属于真正的移动应用工程。
- 任务涉及 Flutter、React Native、iOS、Android、推送、深链、应用签名、商店发布或设备能力调用。
- 需要先给团队明确 02-frontend 与 09-mobile 的边界，再决定后续规则应该写到哪个目录。
- 需要为后续移动端应用规范扩展建立统一入口，而不是把 App 工程能力继续塞进前端页面目录。

## 按任务类型快速入口

- 做移动端与移动 Web 的边界判断：先读本页中的“与 02-frontend 的分工”。
- 做移动端通用边界、工程约束、生命周期、原生能力、发布前共性要求：先读 [11_移动端通用规范](11_移动端通用规范.md)。
- 做 Flutter、React Native、原生 iOS、原生 Android 的技术选型判断：先读 [12_移动端技术选型规范](12_移动端技术选型规范.md)。如属于真正的新建双端 App 工程，默认先看 Flutter 的优先评估与转出规则。
- 做移动应用工程结构、模块边界或基础设施规划：先读 [13_移动端目录结构规范](13_移动端目录结构规范.md)。
- 做相机、定位、蓝牙、推送、文件、通知、深链、分享等设备或系统能力接入：先读 [14_移动端原生能力接入规范](14_移动端原生能力接入规范.md)。
- 做打包、签名、渠道、灰度、商店上架、应用发布和回滚：先读 [15_移动端发布交付规范](15_移动端发布交付规范.md)。
- 做移动端应用测试、验收和发布门禁：先读 [16_移动端测试与验收规范](16_移动端测试与验收规范.md)，并补读 [../05-tests/00_README.md](../05-tests/00_README.md)。
- 做版本上线前执行清单、发布前检查项和交付准备：先读 [17_移动端项目交付清单模板](17_移动端项目交付清单模板.md)。
- 做版本发布验收留档、上线确认记录和灰度放量确认：先读 [18_移动端发布验收记录模板](18_移动端发布验收记录模板.md)。
- 做线上故障、发布异常、灰度事故和问题留档复盘：先读 [19_移动端事故复盘模板](19_移动端事故复盘模板.md)。
- 做移动 App 新项目版本选择、样板工程升级或跨端框架版本基线确认：先读 [20_移动端稳定版本基线清单](20_移动端稳定版本基线清单.md)。

## 与 02-frontend 的分工

- [../02-frontend/00_README.md](../02-frontend/00_README.md) 负责浏览器渲染页面，包括桌面 Web、移动 Web、H5、活动页、外链页和 WebView 页面。
- [../10-miniapp/00_README.md](../10-miniapp/00_README.md) 负责微信小程序及同类平台型轻应用。
- 本目录负责真正的移动应用工程，包括 App 工程结构、跨端移动框架、原生平台能力、安装包、签名、分发和运行时集成问题。
- 判断标准优先看交付形态，而不是只看显示设备是否是手机。
- 如果主体是页面模板、表单页、工作台页、营销页或 WebView 承载的动态页，优先回到 02-frontend。
- 如果主体是 App 生命周期、导航栈、原生桥接、设备权限、推送、离线能力、商店发布或安装包交付，优先留在 09-mobile。

## 阅读顺序建议

- 如任务首先卡在边界判断，应先读本页，再决定回到 02-frontend 还是继续留在 09-mobile。
- 如任务已经明确是移动应用工程，默认先读 [11_移动端通用规范](11_移动端通用规范.md)，再按需进入 [12_移动端技术选型规范](12_移动端技术选型规范.md)、[13_移动端目录结构规范](13_移动端目录结构规范.md)、[14_移动端原生能力接入规范](14_移动端原生能力接入规范.md)、[15_移动端发布交付规范](15_移动端发布交付规范.md)、[16_移动端测试与验收规范](16_移动端测试与验收规范.md)、[17_移动端项目交付清单模板](17_移动端项目交付清单模板.md)、[18_移动端发布验收记录模板](18_移动端发布验收记录模板.md) 和 [19_移动端事故复盘模板](19_移动端事故复盘模板.md)。
- 如任务跨移动 Web 与移动 App 两类形态，先确定主交付形态，再分别进入 02-frontend 或 09-mobile，不混写为同一套规则。

## 不建议的做法

- 把所有在手机上显示的内容都直接归到 09-mobile，而不区分移动 Web 与移动 App。
- 还没明确目标平台和交付形态，就先默认 Flutter、React Native 或原生方案。
- 在分册尚未形成稳定主题前，预先批量创建空白规范文件。
- 把页面模板规范和移动应用工程规范混写在同一个入口下，导致读者既看不清页面边界，也看不清 App 工程边界。
