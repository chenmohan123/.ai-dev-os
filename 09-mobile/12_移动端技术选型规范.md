# 移动端技术选型规范

## 目标

- 规范移动应用工程在新项目和已有项目中的关键技术选型方式。
- 明确 Flutter、React Native、原生 iOS、原生 Android 分别适合什么场景，哪些结论必须先让用户确认。
- 避免在移动端方案设计时擅自拍板跨端框架、导航体系、状态管理、原生桥接、更新机制和发布方案。

## 适用范围

- 适用于 Flutter、React Native、原生 iOS、原生 Android 及其组合方案的技术选型、方案确认和实现前决策。
- 涉及移动应用工程共性边界时，应结合 [11_移动端通用规范](11_移动端通用规范.md) 使用。
- 涉及浏览器页面、H5、WebView 动态页或移动 Web UI 方案时，应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。

## 总体原则

- 新项目如未明确技术栈、目标平台、导航方案、状态管理、原生桥接、更新方案或测试方案，应先列出候选并让用户确认，不替用户拍板。
- 对真正的移动应用工程新项目，如需求目标是 iOS、Android 双端产品且尚未出现明确排除条件，默认应先把 Flutter 作为优先评估方案，而不是从多个方案中无差别平铺展开。
- 已有项目如已存在稳定 App 工程、导航体系、发布链路、原生模块和测试方案，应优先复用项目现状。
- 选型优先考虑团队能力、交付速度、原生能力复杂度、长期维护成本和发布稳定性，而不是只看单点开发体验。
- 同一类能力只保留一套主方案，避免同一项目同时并存两套导航体系、两套桥接层或两套主状态管理方案。
- 新项目在确认 Flutter、React Native 或原生链路时，还应明确是否采用当前官方稳定通道或最近验证通过的稳定基线，而不是默认沿用过时模板。

## 新项目决策要求

- 在进入实现前，至少确认以下内容：目标平台、是否必须共享代码、是否重度依赖原生能力、团队技术栈、导航方案、状态管理、原生桥接策略、更新方式、测试策略和发布渠道。
- 如项目会直接产出关键页面、演示页面或产品级移动端界面，还应先补一份视觉方向卡，至少说明主用户、主任务、视觉焦点、风格关键词和禁止事项，再进入页面实现。
- 如用户未明确是否必须一套代码覆盖多端，应先确认跨端目标，再讨论 Flutter 或 React Native。
- 如用户未明确是否需要深度原生能力、极致性能或平台特性定制，应先确认原生依赖强度，再讨论是否采用原生开发。
- 如用户未明确发布渠道、灰度策略和签名方式，应先补问，而不是把打包发布当作后置细节。
- 如属于真正的新建移动应用工程，应先从“是否优先评估 Flutter”开始判断；只有在出现明确转出条件时，才把 React Native、原生或混合方案提升为主候选。
- 如项目要作为样板工程、模板仓库或演示产品长期复用，应额外确认关键版本基线和升级策略。

## 已有项目决策要求

- 先检查当前项目的技术栈、目录结构、导航方案、状态管理、原生模块、发布脚本和测试链路。
- 如现有方案可满足需求，应直接复用，不新增第二套同类能力。
- 只有在现有方案明显无法满足性能要求、原生能力要求、多人维护成本或跨平台一致性目标时，才讨论替换或升级。
- 讨论替换时，应明确迁移范围、平台影响、兼容成本、灰度方式和回退路径，不做隐式迁移。
- 如现有工程版本明显落后且已影响生态兼容、插件能力或样板工程价值，应单独发起升级评估。

## 关键能力选型边界

### Flutter

- 在真正的新建移动应用工程中，Flutter 是默认优先评估方案，但不是无条件唯一答案。
- 适合需要较强跨平台一致性、独立 UI 体系、完整一体化开发体验和中长期产品演进的项目。
- 适合团队愿意长期维护 Dart 技术栈，且希望 iOS、Android 在大部分业务层和 UI 层共享代码的场景。
- 不适合只是补几页 H5、已有原生项目很重且仅做局部增量、或团队完全没有 Dart 维护意愿的场景。
- 如项目高度依赖现有原生模块，也要评估 Flutter 插件生态、桥接复杂度和团队调试成本。

### Flutter 转出条件

- 如果团队明确不接受 Dart 技术栈长期维护成本，应转向 React Native 或原生方案评估。
- 如果项目重度依赖系统级能力、复杂后台任务、极深平台定制、复杂音视频链路或高强度原生集成，应优先转向原生方案评估。
- 如果项目本质上只是已有原生 App 的局部业务增量，而不是新建主工程，应优先评估混合接入，而不是直接默认 Flutter 主导全量工程。
- 如果团队的核心优势明确集中在 React/TypeScript，且原生能力复杂度处于中低水平，可转向 React Native 作为主候选。

### React Native

- 适合团队已有 React/JavaScript/TypeScript 积累，希望复用前端人才和部分业务模型的场景。
- 适合跨平台业务逻辑较多，但仍需要一定原生能力和原生界面融合的项目。
- 不适合对平台一致性、渲染性能、复杂动画或原生能力深度耦合要求极高且团队缺少原生支撑的场景。
- 需重点评估原生模块维护成本、版本升级成本和桥接复杂度，不因“前端团队上手快”而忽略长期维护问题。

### 原生 iOS / Android

- 适合重度依赖平台特性、系统能力、性能、后台任务、蓝牙、音视频、地图、复杂推送或系统级集成的场景。
- 适合产品对平台体验差异有明确要求，或一个平台优先级显著高于另一个平台的项目。
- 不适合只是为了“更正统”而在跨平台诉求明确、团队原生力量不足时强行双端完全原生开发。
- 如选择原生方案，应同时评估双端团队配比、需求同步成本和发布节奏一致性。

### 混合方案

- 如已有成熟原生 App，仅在部分业务模块引入 Flutter 或 React Native，应明确边界，不把混合接入误写成全量跨端方案。
- 混合方案适合已有大型原生工程、希望在局部业务上提速的项目。
- 混合方案的风险在于工程复杂度、包体、桥接、导航和发布协同成本上升，应显式说明。

## 常见决策维度

- 交付目标：是双端统一产品，还是某一平台优先。
- 团队能力：是以前端团队为主，还是有稳定 iOS、Android 原生团队。
- 原生能力依赖：是否涉及蓝牙、定位、后台任务、复杂音视频、系统级推送和设备硬件能力。
- 性能要求：是否存在高频动画、复杂列表、重图形渲染或长生命周期后台能力。
- 发布形态：是否需要企业分发、商店上架、灰度、渠道包、多环境签名和热更新策略。
- 生命周期：项目是短期验证、中期业务应用，还是长期演进的产品级 App。

## 常见组合建议

- 跨平台产品优先型：Flutter + 统一业务分层 + 受控原生插件接入。适合希望长期统一双端体验、接受 Dart 栈的团队。
- 前端能力复用型：React Native + 受控原生桥接 + 明确升级策略。适合前端团队主导、需要一定跨平台复用的项目。
- 平台能力优先型：原生 iOS + 原生 Android。适合平台能力复杂、性能要求高、系统集成深的项目。
- 现状增强型：沿用已有原生或跨端工程，只在当前基础上补缺能力。适合已有项目增量迭代。

## 快速决策矩阵

- 如果是新项目，且目标是双端统一体验、团队接受 Dart、产品生命周期长，应默认先评估 Flutter，并在没有明确转出条件时优先考虑 Flutter。
- 如果是新项目，且团队以 React 为主、需要尽快复用前端工程能力，同时原生能力复杂度中等，可优先考虑 React Native。
- 如果项目重度依赖系统级能力、性能要求高或平台差异大，应优先考虑原生方案。
- 如果是已有原生项目做局部增量，不应直接默认全量迁移到跨端框架，应先评估混合接入边界。
- 如果主要需求只是外链页、活动页、表单页或 WebView 承载的动态业务页，不应进入 09-mobile，应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。

## 默认优先评估与转出规则

- 默认优先评估只适用于真正的新建移动应用工程，不适用于移动 Web 页面、H5 页面或 WebView 动态页。
- 默认优先评估不等于自动定案；仍需用户确认团队能力、发布目标、原生能力复杂度和维护意愿。
- 一旦命中 Dart 维护意愿不足、原生集成过深、现有原生工程过重或团队明确以 React 工程体系为核心等条件，应显式转出 Flutter，改评 React Native、原生或混合方案。

## 选型提问模板

- 这是新项目还是已有项目。
- 目标平台是 iOS、Android，还是双端同时上线。
- 是否必须共享一套主要代码，还是可以接受双端分别维护。
- 当前团队是以前端为主、移动端为主，还是已有稳定原生团队。
- 是否依赖蓝牙、定位、音视频、推送、深链、后台任务、地图或其他复杂原生能力。
- 发布目标是企业内部分发、App Store / 应用商店上架，还是两者都有。
- 本次是做增量功能、局部嵌入，还是准备新建一套移动端产品。

## 选型结论输出要求

- 输出结论时，应明确区分推荐方案、备选方案、不建议方案三类信息。
- 每个候选方案至少说明适用场景、主要收益、主要限制、团队要求和迁移或维护成本。
- 如为新项目，应明确哪些点已由用户确认，哪些点仍待确认，不把待确认项写成既定事实。
- 如为已有项目，应明确哪些能力沿用现状，哪些能力只是局部增强，哪些能力确实需要架构调整。
- 如涉及样板工程、模板仓库或演示项目，还应明确版本基线策略，而不是只给技术名。
- 如后续要直接进入页面、流程页或演示界面实现，还应附带视觉方向卡摘要，不允许在页面目标、信息主次和体验气质未说明时直接生成 UI。
- 最终结论应能直接指导下一步进入工程初始化、目录规划或原生能力设计，而不是停留在泛泛比较。

## 选型结论模板示例

### 通用输出模板

- 项目类型：新项目或已有项目。
- 当前约束：目标平台、现有技术栈、团队结构、原生能力依赖、发布要求。
- 推荐方案：本次建议采用的主方案。
- 推荐理由：为什么该方案更适合当前团队、业务复杂度和交付节奏。
- 备选方案：可行但不是首选的候选方案。
- 不建议方案：当前阶段不建议采用的方案及原因。
- 待确认项：仍需用户拍板的技术点。
- 下一步动作：进入工程初始化、目录规范、原生能力设计或发布方案设计。

### 新项目示例

- 项目类型：新项目。
- 当前约束：目标是 iOS 和 Android 双端上线；团队以前端为主；存在相机、推送和深链能力，但不属于极重原生集成。
- 推荐方案：Flutter。
- 推荐理由：当前场景属于真正的新建双端 App 工程，且未出现明显转出条件，因此先按默认优先评估策略选择 Flutter 作为统一主工程。
- 备选方案：React Native。适合希望更强复用前端人才的团队，但需额外关注桥接和升级成本。
- 不建议方案：双端完全原生。当前团队结构和交付节奏不足以支撑双端独立长期维护。
- 待确认项：是否接受 Dart 技术栈；是否已有现成原生插件需要复用；发布渠道是否包含企业内部分发。
- 下一步动作：请用户确认待确认项后，进入移动端通用规范并继续补工程结构与原生能力分层方案。

### 已有项目示例

- 项目类型：已有项目。
- 当前约束：项目已是原生 Android + iOS 架构，已有稳定登录、推送、深链和发布链路；当前需求是新增一组业务模块。
- 推荐方案：沿用现有原生工程，在现有架构上做增量扩展。
- 推荐理由：当前基础设施稳定，直接复用现状能降低迁移成本并保持发布链路可控。
- 备选方案：仅在局部独立业务模块中评估 Flutter 或 React Native 混合接入，但需单独评估桥接、导航和发布影响。
- 不建议方案：为单次业务增量整体迁移到另一套跨端框架。当前收益不足以覆盖迁移风险。
- 待确认项：现有原生架构是否存在明确瓶颈；是否真的需要跨平台代码共享。
- 下一步动作：先按现有工程结构落地需求，如出现明确基础设施缺口，再单独评估局部增强方案。

## 方案调整触发条件

- 当现有工程无法支撑目标平台扩展、原生能力接入或发布要求时，可以讨论升级或替换主技术栈。
- 当现有跨端框架升级成本、原生桥接成本或性能问题已经明显影响交付时，可以讨论切换或分层重构。
- 当团队结构已经变化，例如从前端主导转向稳定原生团队主导，或反之，可以重新评估技术栈选择。
- 只要现有方案仍可满足需求，就不应仅因“社区更流行”或“看起来更统一”而发起迁移。

## 推荐决策顺序

- 先判断是新项目还是已有项目。
- 再确认目标平台、交付形态和是否必须共享代码。
- 再确认原生能力复杂度、团队能力和发布要求。
- 只有在缺口明确存在时，才列出候选方案让用户选择。
- 方案确认后，再进入工程规范和具体实现。

## 禁止事项

- 禁止在用户未确认的情况下，直接假定 Flutter、React Native 或原生方案为唯一答案。
- 禁止在已有项目中无视现有工程和发布链路，强行整体迁移到另一套技术体系。
- 禁止同一项目中长期并存两套主导航、两套主桥接层或两套主状态管理体系。
- 禁止只给出选型结论，不说明适用场景、收益、限制和成本。

## 提交前自检

- 是否明确区分了新项目选型和已有项目复用两种场景。
- 是否先检查了目标平台、团队能力、原生依赖和发布要求，而不是直接代替用户选框架。
- 是否对候选方案说明了收益、限制和维护成本。
- 如任务将继续进入页面实现，是否已补齐视觉方向卡，而不是让 AI 直接开始画页面。
- 是否避免把移动 Web 页面方案误判为移动应用工程选型问题。
