# 敏感信息与密钥管理规范

## 适用范围

- 适用于密码、Access Token、API Key、数据库连接串、证书、私钥、环境变量和配置中心。

## 目标

- 让敏感信息在生成、存储、分发、使用、轮换和吊销各阶段都有明确控制边界。
- 避免凭据长期散落在源码、脚本、镜像、日志或本地开发材料中，形成难以追踪的泄露面。

## 基本要求

- 敏感信息不得硬编码在源码、脚本、镜像、示例配置和测试用例中。
- 密钥必须集中存储、按环境隔离、按用途分发，并支持轮换与吊销。
- 口令类信息必须使用不可逆摘要与合规参数存储，禁止明文或可逆加密落库。
- 日志、异常、监控、埋点和审计中禁止打印完整密钥、令牌、身份证号、手机号等高敏字段。
- 本地开发样例必须提供占位值和注入方式，不提供真实凭据。

## 存储与分发要求

- 凭据、令牌、证书和连接串应通过密钥管理系统、配置中心或受控注入机制分发。
- 不同环境、不同服务和不同用途的密钥应隔离管理，不混用同一份高权限凭据。
- 示例配置、README、部署脚本和测试模板只保留占位值，不保留真实值或历史残留值。
- 本地开发若必须注入敏感配置，应明确最小权限、最短有效期和替换方式。

## 管理约束

- 密钥访问权限按人、服务、环境最小化分配。
- 证书和令牌必须记录签发主体、用途、过期时间和轮换负责人。
- 泄露事件必须具备快速吊销、替换和影响面评估流程。

## 轮换与过期要求

- 高权限令牌、数据库账号、证书和长期 API Key 应定义轮换周期和过期提醒机制。
- 轮换应尽量支持双凭据过渡或等价兼容期，避免硬切造成服务不可用。
- 失效或废弃凭据应明确清理路径，不长期残留在配置系统和历史脚本中。

## 泄露处置要求

- 一旦发现凭据泄露，应优先吊销、替换、限制权限并评估影响范围。
- 泄露处置应明确涉及系统、使用方、日志痕迹和可能被访问的数据范围。
- 修复后应检查是否仍存在源码残留、镜像残留、日志残留或文档残留。

## 常见风险信号

- 同一份凭据被多个环境或多个服务长期共用。
- 示例配置、部署脚本或文档里残留真实连接串或令牌。
- 高敏字段在日志、报错或导出文件中被完整打印。
- 凭据已过期或已轮换，但旧值仍在配置系统、脚本或镜像里残留。

## 验收清单

- 是否存在源码、提交记录、镜像或文档中的真实凭据。
- 是否定义了轮换周期、过期提醒和泄露应急处置。
- 是否对日志、错误和导出文件做了敏感字段掩码。

## 输出模板建议

- 敏感对象范围：说明涉及哪些密钥、令牌、连接串、证书和配置。
- 存储分发方式：说明保管位置、注入路径、权限范围和环境隔离方式。
- 生命周期：说明签发、过期、轮换、吊销和清理策略。
- 泄露处置：说明发现后止血、替换、影响评估和复查步骤。
- 风险与限制：说明当前历史残留、兼容负担和待治理项。

## 不建议的做法

- 用环境变量方便为由，把高权限长期凭据散落到多个脚本和机器。
- 轮换只替换当前运行值，不清理旧脚本、旧镜像和旧文档中的历史残留。
- 在日志、异常或监控里输出完整令牌、连接串或个人高敏标识。
- 为了省事让多个系统或多个环境共用同一份密钥。
