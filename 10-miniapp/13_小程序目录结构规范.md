# 小程序目录结构规范

## 目标

- 规范小程序工程的目录组织方式、模块边界和文件放置原则。
- 降低应用入口、页面、分包、平台能力封装、数据访问和提审发布资源长期混杂的风险。
- 让新项目能够快速建立清晰结构，让已有项目能够判断哪些目录应继续沿用、哪些应调整。

## 适用范围

- 适用于微信小程序及后续纳入的同类平台型轻应用的目录设计、模块拆分和结构治理。
- 涉及小程序工程共性边界时，应结合 [11_小程序通用规范](11_小程序通用规范.md) 使用。
- 涉及原生小程序方案、Taro、uni-app 及同类方案选型时，应结合 [12_小程序技术选型规范](12_小程序技术选型规范.md) 使用。
- 不适用于移动 Web、H5、活动页或 WebView 动态页的目录组织；这些场景应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。
- 不适用于真正移动 App 工程的目录组织；这些场景应回到 [../09-mobile/00_README.md](../09-mobile/00_README.md)。

## 核心原则

- 目录按运行边界、职责边界和业务边界组织，不按个人习惯或页面名称随意堆放。
- 应用入口、页面层、业务模块层、平台能力层、数据访问层和提审发布资源应清晰分层。
- 页面私有实现优先就近放置；跨页面、跨分包或跨业务域复用后再上提到共享目录。
- 同一类基础设施只保留一套主入口，例如路由注册、状态管理、平台能力封装、埋点、配置和日志能力。
- 结构设计优先服务长期维护、分包演进和平台审核发布，不为了短期开发方便把页面、平台 API 和提审资源混放在同一层。

## 新项目目录设计要求

- 新项目开始前，应先确定应用入口、页面目录、分包边界、平台能力目录、数据访问目录和提审发布资源目录，再进入大规模开发。
- 如果用户未指定目录结构，应先给出候选方案并让用户确认，不替用户拍板唯一模板。
- 新项目应至少明确以下边界：应用入口、页面目录、分包目录、业务模块目录、平台能力目录、数据访问目录、配置与日志目录、发布与提审目录。
- 目录层级应保持可读，避免刚起步就形成深层嵌套和跨层相互依赖。

## 已有项目结构判断要求

- 已有项目优先检查当前应用入口、页面组织方式、分包策略、平台封装位置、共享能力层、数据访问层和提审资源位置。
- 如当前结构已能支撑新增功能开发，应优先沿用现状，不因理想模板强行整体搬迁。
- 只有在职责明显混杂、分包边界失控、同类能力重复分散或发布链路混乱时，才讨论调整。
- 讨论调整时，应优先做局部收敛和渐进迁移，而不是一次性整体翻目录。

## 推荐目录边界

### 应用入口层

- 放应用启动入口、环境初始化、登录态恢复、全局异常处理、宿主生命周期监听和全局配置装载逻辑。
- 入口层负责应用启动和全局编排，不承载具体业务页面和平台能力细节实现。
- 如同时存在多个宿主平台入口，应明确平台入口边界，不在单一入口文件中长期堆积平台分支判断。

### 页面与路由目录

- 放页面注册、路由映射、Tab 页、页面级生命周期处理和页面编排逻辑。
- 页面目录按业务域、功能域或稳定导航域组织，不按零散组件类型拆成难以理解的碎片目录。
- 页面只承载页面编排、页面级状态和交互组织，不直接堆积大量平台 API 调用和数据转换逻辑。
- 页面私有组件、页面私有 hooks、局部类型和局部样式应优先与页面同目录放置。

### 分包目录

- 主包与分包应按稳定业务域、访问频率和首屏要求组织，不按开发人员或临时任务拆包。
- 分包只承载明确属于该业务域的页面、局部组件和局部资源，不把共享基础设施散落复制到多个分包。
- 分包边界调整前，应先评估首屏体积、跳转链路、公共依赖和后续维护成本。
- 不因短期包体告警而临时把大量共享逻辑拆散到多个分包，导致后续治理失控。

### 业务模块目录

- 与具体业务域绑定的数据访问、状态转换、领域规则、枚举常量和复用逻辑，应优先沉淀到对应业务模块目录。
- 同一业务域的接口定义、数据映射和领域服务应尽量就近组织，减少跨目录跳转成本。
- 业务模块目录用于承载业务共性，不替代页面目录和平台能力目录。

### 平台能力与宿主适配目录

- 登录授权、手机号授权、支付、订阅消息、分享、定位、相机、文件、扫码、客服和宿主 API 适配，应按能力模块组织。
- 平台能力目录只承载平台能力封装、平台差异处理、错误模型和统一调用入口，不承载业务页面逻辑。
- 同一能力只保留一个主入口，不让多个页面私自复制能力封装。
- 如同时存在微信小程序与其它平台宿主适配，应把平台差异处理和业务层边界明确拆开。

### 数据访问与状态目录

- 请求客户端、鉴权处理、错误拦截、缓存策略和统一数据访问封装应集中在数据访问层目录。
- 状态管理目录只承载跨页面共享且确有必要的状态，不承载所有页面临时态。
- 当存在业务域级数据访问封装时，应明确区分底层请求能力与业务数据访问能力的边界。

### 配置、日志与基础设施目录

- 环境配置、功能开关、日志上报、埋点、监控、主题 token 和安全存储封装应集中管理。
- 配置模板、默认值、环境差异和敏感配置读取方式应归拢在统一位置，不分散在页面和脚本中。
- 日志、埋点和错误上报应有统一入口，不在业务页面中散落多套埋点或日志封装。

### 提审与发布目录

- 小程序配置、提审脚本、环境资源、版本说明、图标资源和发布辅助脚本应集中在提审发布目录。
- 发布目录只承载提审和发布相关资源，不混入业务逻辑实现。
- 不同环境、不同宿主平台和不同渠道的发布资源应明确拆分，避免长期堆积难以辨认的配置。

## 推荐目录模板

- 可采用“app + pages + packages + features + platform + data + shared + config + release”的组织思路。
- 一个可接受的模板通常包含以下层次：应用入口、页面层、分包层、业务模块、平台能力、数据访问、共享组件或共享类型、配置日志、提审发布资源。
- 如果项目规模较小，可以适当合并部分目录，但仍应保持页面、平台能力和发布资源分离。
- 如果项目规模较大，可以在业务域下继续细分页面、领域逻辑和私有组件，但不要无限制纵深嵌套。

## 结构失衡信号

- 一个目录同时混放页面、平台 API 封装、请求层、配置、提审脚本和测试数据。
- 同类能力封装、状态仓库、请求层和埋点代码分散在多个不一致位置。
- 多个页面各自复制一套登录授权、支付、订阅消息或分享处理逻辑。
- 单个入口文件既有应用启动、页面注册、登录恢复、授权处理、支付回调和业务流程，已经难以维护。
- 提审配置、环境资源和发布脚本散落在根目录或业务目录中。
- 历史模块、废弃分包、旧版页面目录或临时发布产物长期留在正式结构中。

## 调整策略

- 优先先收敛应用入口、页面路由主入口和平台能力主入口，再处理页面私有实现迁移。
- 优先局部移动和逐步重命名，不一次性大范围搬迁所有目录。
- 每次调整都应同步更新页面注册、分包配置、能力封装入口和发布配置，避免旧路径与新路径并存。
- 如结构调整涉及多个宿主平台或多个环境发布链路，应先明确迁移范围和落地顺序。

## 提交前自检

- 是否明确区分了应用入口、页面目录、分包目录、业务模块目录、平台能力目录和发布目录的职责。
- 是否存在平台能力直接混入页面目录，或页面私有实现被过早提升到共享目录的情况。
- 是否存在重复页面注册入口、重复状态仓库、重复请求层或重复平台能力封装。
- 是否有目录命名模糊、用途不清、历史遗留未清理的问题。
- 是否在已有项目中做了不必要的大范围结构迁移。
