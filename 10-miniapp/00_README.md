# 目录索引

## 目标

- 为小程序与平台型轻应用提供统一入口，明确什么时候应该按微信小程序及同类平台应用来理解任务。
- 把小程序与 02-frontend 中的浏览器页面、以及 09-mobile 中的真正移动 App 工程分开，避免入口混淆。
- 在目录刚建立时先给出边界、进入方式和扩展方向，避免先建目录却没有使用说明。

## 使用说明

- 进入小程序规范目录后，默认先阅读本文件，再判断当前任务是做边界判断、技术选型、目录结构、平台能力接入，还是后续补齐分册。
- 如任务主体是浏览器渲染页面、H5 页面、移动 Web 页面、活动页、外链页或 WebView 页面，返回 [../02-frontend/00_README.md](../02-frontend/00_README.md)，不要在本目录继续展开。
- 如任务主体是真正移动 App 工程、安装包、商店分发、原生宿主、Flutter、React Native、原生 iOS 或原生 Android，返回 [../09-mobile/00_README.md](../09-mobile/00_README.md)。
- 如任务主体是微信小程序，或同类运行在超级 App / 平台宿主内的轻应用，继续留在本目录。
- 如任务涉及技术选型、框架选型、组件库选型、架构选型或模板选型，先询问用户确认后再继续输出具体方案。

## 什么时候先读本页

- 你还不确定当前需求到底属于移动 Web 页面、真正移动 App，还是属于微信小程序这类平台型轻应用。
- 任务涉及微信小程序路由、生命周期、登录授权、支付、订阅消息、平台审核发布、宿主 API 或平台能力接入。
- 需要先给团队明确 02-frontend、09-mobile 和 10-miniapp 三者的边界，再决定后续规则应该写到哪个目录。
- 需要为后续支付宝小程序、抖音小程序或其它同类平台应用扩展预留统一入口。

## 按任务类型快速入口

- 做小程序与移动 Web、移动 App 的边界判断：先读本页中的“与 02-frontend、09-mobile 的分工”。
- 做小程序通用边界、工程约束、宿主生命周期、平台能力和发布前共性要求：先读 [11_小程序通用规范](11_小程序通用规范.md)。
- 做微信小程序或同类平台应用的技术选型、宿主选择、跨平台方案判断：先读 [12_小程序技术选型规范](12_小程序技术选型规范.md)。如属于真正以微信小程序为主交付形态的新项目，默认先看原生小程序方案的优先评估与转出规则。
- 做小程序目录结构、分包、页面组织和公共能力规划：先读 [13_小程序目录结构规范](13_小程序目录结构规范.md)。
- 做登录授权、支付、订阅消息、分享、扫码、文件和其它宿主能力接入：先读 [14_小程序平台能力接入规范](14_小程序平台能力接入规范.md)。
- 做平台审核、提审发布、灰度和发布前检查：先读 [15_小程序发布交付规范](15_小程序发布交付规范.md)。

- 做版本上线前执行清单、发布前检查项和交付准备：先读 [16_小程序项目交付清单模板](16_小程序项目交付清单模板.md)。

## 与 02-frontend、09-mobile 的分工

- [../02-frontend/00_README.md](../02-frontend/00_README.md) 负责浏览器渲染页面，包括桌面 Web、移动 Web、H5、活动页、外链页和 WebView 页面。
- [../09-mobile/00_README.md](../09-mobile/00_README.md) 负责真正的移动 App 工程，包括 Flutter、React Native、原生 iOS、原生 Android、安装包、签名、分发和运行时集成问题。
- 本目录负责小程序与平台型轻应用，包括微信小程序，以及后续可能纳入的同类宿主内应用形态。
- 判断标准优先看交付宿主和运行时，而不是只看显示设备是不是手机。
- 如果主体是浏览器页面和 WebView 页面，优先回到 02-frontend。
- 如果主体是安装包、商店分发、原生宿主和 App 生命周期，优先回到 09-mobile。
- 如果主体是宿主平台内运行的小程序页面、平台 API、审核提审、订阅消息、支付和平台登录授权，优先留在 10-miniapp。

## 阅读顺序建议

- 如任务首先卡在边界判断，应先读本页，再决定回到 02-frontend、09-mobile 还是继续留在 10-miniapp。
- 如任务已经明确属于微信小程序或同类平台型轻应用，默认先读 [11_小程序通用规范](11_小程序通用规范.md)，再按需进入 [12_小程序技术选型规范](12_小程序技术选型规范.md)、[13_小程序目录结构规范](13_小程序目录结构规范.md)、[14_小程序平台能力接入规范](14_小程序平台能力接入规范.md)、[15_小程序发布交付规范](15_小程序发布交付规范.md)、[19_小程序测试与验收规范](19_小程序测试与验收规范.md)、[16_小程序项目交付清单模板](16_小程序项目交付清单模板.md)、[17_小程序发布验收记录模板](17_小程序发布验收记录模板.md) 和 [18_小程序事故复盘模板](18_小程序事故复盘模板.md)。
- 如任务跨移动 Web、小程序和移动 App 三类形态，先确定主交付形态，再分别进入对应目录，不混写为同一套规则。

## 不建议的做法

- 把微信小程序直接归入 02-frontend，而不区分浏览器运行时与平台宿主运行时。
- 把微信小程序直接归入 09-mobile，而不区分小程序交付和真正 App 工程交付。
- 还没明确宿主平台、运行时和发布链路，就先默认把小程序当作 H5 或 App 工程处理。
- 在分册尚未形成稳定主题前，预先批量创建空白规范文件。
