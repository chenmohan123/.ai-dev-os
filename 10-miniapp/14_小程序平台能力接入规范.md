# 小程序平台能力接入规范

## 目标

- 统一小程序工程中登录授权、手机号授权、支付、订阅消息、分享、定位、相机、文件、扫码、客服等平台能力的接入边界。
- 避免平台能力在页面中被随意调用，导致授权扩散、平台行为不一致、错误难以归因和安全风险放大。
- 让平台能力接入既满足业务可用性，也满足可维护、可诊断、可回滚和可审计要求。

## 适用范围

- 适用于微信小程序及后续纳入的同类平台型轻应用中的宿主能力和平台 API 接入。
- 涉及小程序工程共性边界时，应结合 [11_小程序通用规范](11_小程序通用规范.md) 使用。
- 涉及原生小程序方案、Taro、uni-app 及同类方案选型时，应结合 [12_小程序技术选型规范](12_小程序技术选型规范.md) 使用。
- 涉及小程序工程目录、分包和平台能力模块放置方式时，应结合 [13_小程序目录结构规范](13_小程序目录结构规范.md) 使用。
- 不适用于 H5、移动 Web 或 WebView 页面中的浏览器能力调用；这些场景应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。
- 不适用于真正移动 App 工程中的原生能力调用；这些场景应回到 [../09-mobile/14_移动端原生能力接入规范.md](../09-mobile/14_移动端原生能力接入规范.md)。

## 核心原则

### 1. 一类能力只保留一个主入口

- 登录授权、手机号授权、支付、订阅消息、分享、定位、相机、文件、扫码、客服等能力，应各自有统一入口。
- 不允许多个页面、多个模块各自封装一套不兼容的支付、登录或分享处理逻辑。
- 统一入口应负责参数校验、授权判断、平台差异适配、错误归一化和日志记录。

### 2. 页面只消费能力接口，不直接处理平台细节

- 页面层应调用能力服务或宿主适配层暴露的稳定接口，而不是直接拼装平台调用细节。
- 授权申请、失败重试、平台差异和异常分类应由能力层统一处理。
- 页面只接收业务可用的成功结果、失败结果和可恢复提示，不直接感知底层异常对象。

### 3. 能力接入必须说明边界和失败方式

- 每个能力应说明触发条件、参数边界、返回结构、失败场景、用户取消语义和平台差异。
- 不应把授权拒绝、用户取消、系统不可用、能力未初始化和业务失败混成一个布尔结果。
- 对平台差异明显的能力，应显式写明微信小程序与后续其它宿主平台的差异，不默认跨平台完全一致。

## 登录与授权能力要求

- 登录、静默鉴权、手机号授权和用户信息获取应通过统一身份入口调用。
- 应区分首次登录、会话恢复、授权拒绝、token 失效和后端换票失败等结果。
- 授权申请应与具体业务动作绑定，不在应用启动时一次性申请无关授权。
- 登录态刷新、session 过期和重新授权路径应统一收口，不允许每个页面各自维护一套处理逻辑。

## 支付能力要求

- 支付发起、支付参数换取、支付结果确认和取消支付处理应通过统一支付能力入口管理。
- 应区分用户取消、参数失效、签名错误、平台失败、网络异常和后端确认失败等结果。
- 支付回调后的业务落点、订单状态刷新和幂等处理应有统一规则，不在页面中散落多套订单收尾逻辑。
- 涉及企业资金、敏感订单或退款链路时，应补读 [../07-security/00_README.md](../07-security/00_README.md)。

## 订阅消息与通知能力要求

- 订阅消息授权、模板选择、发送前状态校验和触达结果处理应通过统一消息能力入口调用。
- 应说明触发条件、用户拒绝后的降级方式、消息状态同步和失败重试策略。
- 如订阅消息唤起页面，应由页面路由层统一收口，不在业务页面中散落多套跳转判断。

## 分享与外部进入能力要求

- 页面分享、业务卡片分享、参数拼装和分享回流处理应通过统一分享能力入口调用。
- 应校验分享参数、目标页面映射和高风险动作边界，不允许外部入参直接决定内部高风险动作。
- 分享回流、扫码进入和模板消息进入后的落点应统一收口，并明确来源识别和过期参数处理策略。

## 定位、相机、扫码与文件能力要求

- 定位、相机、扫码、拍照、文件选择、下载、导出和预览应通过统一媒体与文件能力入口调用。
- 应区分用户取消、授权不足、平台失败、文件类型不支持、空间不足和写入失败等结果。
- 如涉及图片压缩、二维码解析、文件导出、上传前预处理和本地缓存，应在能力层或受控服务层完成，不在页面层散落实现。
- 涉及身份证、营业执照、企业敏感票据、位置考勤或敏感附件时，应补读 [../07-security/00_README.md](../07-security/00_README.md)。

## 客服、反馈与宿主扩展能力要求

- 客服消息、意见反馈、宿主内跳转、打开外部页面和其它宿主扩展能力，应通过统一宿主适配入口调用。
- 应说明能力可用条件、平台限制、失败反馈和降级路径。
- 对仅在部分宿主可用的能力，应显式标明不可用范围，不以静默失败替代边界说明。

## 平台差异要求

- 登录、支付、订阅消息、分享、文件系统、客服和开放接口在不同宿主平台上可能存在显著差异，应显式说明。
- 平台不支持的能力应提供降级路径或明确标明不可用，而不是静默失败。
- 跨端框架中的平台 API 封装如受插件版本、编译目标或宿主限制影响，也应写入已知约束，而不是假定官方 API 行为绝对一致。

## 日志、诊断与回滚要求

- 关键平台能力应记录统一日志，包括调用入口、参数摘要、授权状态、结果状态和错误分类。
- 对登录、支付、订阅消息、分享回流、扫码进入和文件导出这类高频故障点，应保留可排查的关键事件链。
- 新接入平台能力时，应说明是否可灰度、可关闭、可降级和可回滚，不把高风险能力一次性强耦合到主流程。
- 只要能力接入失败可能影响核心业务闭环，就应定义止血路径和人工兜底方案。

## 输出要求

- 输出平台能力方案时，应至少说明能力入口、授权模型、参数边界、失败语义、平台差异和日志要求。
- 如项目已有能力封装，应优先复用现有入口，再补齐授权、错误模型、日志和资源清理机制。
- 如某项平台能力是否启用仍待用户确认，应显式标明待确认项，不写成既定事实。

## 不建议的做法

- 在页面中直接散落调用登录授权、支付、分享、扫码或文件读写逻辑。
- 同一平台能力在多个页面和模块中各自封装一套不兼容调用方式。
- 不区分用户取消、授权拒绝、平台失败和业务失败，导致页面无法正确反馈。
- 在未评估平台差异和宿主限制前，默认所有宿主平台行为一致。

## 提交前自检

- 是否为登录、支付、订阅消息、分享、定位、扫码、文件和客服能力定义了统一入口。
- 是否避免了页面层直接处理底层平台细节和异常对象。
- 是否对授权、失败结果、用户取消和平台差异做了清晰区分。
- 是否为关键平台能力补齐了日志、降级、灰度或回滚要求。
