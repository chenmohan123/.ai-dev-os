# 小程序通用规范

## 目标

- 为小程序与平台型轻应用提供一套通用边界，统一说明宿主运行时、页面组织、平台能力、状态组织和发布前基本要求。
- 避免把小程序简单当成浏览器页面或真正移动 App 工程处理，忽略宿主 API、审核提审、平台登录授权和运行时差异。
- 为后续微信小程序及其它同类平台应用的具体分册提供共性前置约束。

## 适用范围

- 适用于微信小程序及后续纳入的同类平台型轻应用。
- 适用于新项目初始化、已有项目治理、平台能力接入、架构评审和发布前检查。
- 不适用于浏览器页面、H5、活动页或 WebView 页面；这些场景应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。
- 不适用于真正移动 App 工程、安装包、商店分发或原生宿主问题；这些场景应回到 [../09-mobile/00_README.md](../09-mobile/00_README.md)。

## 总体原则

- 先明确交付宿主、运行时边界和平台能力，再讨论具体框架、工程组织和实现细节。
- 先复用项目现有小程序工程结构、分包策略、平台封装和发布链路，不为追求理想方案强行重构。
- 同一类基础能力只保留一套主方案，避免状态管理、请求层、登录授权、支付封装、埋点或平台 API 适配长期并存多套实现。
- 小程序规范不仅要覆盖页面和交互，还要覆盖宿主生命周期、授权路径、平台能力、审核提审和失败恢复。
- 新项目、样板工程和演示工程应优先采用当前公开稳定快照或宿主官方稳定工具链，不直接沿用明显过时的工程模板版本。
- 小程序的体验质量总口径应继续结合 [../01-general/09_多终端体验与高质量输出规范](../01-general/09_多终端体验与高质量输出规范.md) 理解，而不是把高质量输出误当成浏览器页面专属要求。

## 工程边界

- 小程序工程应区分应用入口、页面层、业务模块层、平台能力接入层和基础设施层，不把平台 API 调用散落到页面各处。
- 页面路由、全局状态、网络访问、埋点、日志、登录态恢复、支付流程和配置读取应通过统一入口收口。
- 平台能力如登录授权、手机号授权、支付、订阅消息、分享、定位、相机、文件、扫码和客服能力，应由受控封装层统一管理，不直接在业务代码中到处散用。
- 如项目内同时存在小程序页面、浏览器页面和移动 App，应明确各自边界，不把宿主 API、浏览器 API 和 App 原生能力混写成同一套规则。

## 结构建议

- 应至少区分应用入口、路由与页面层、业务模块层、平台能力层、数据访问层和基础设施层。
- 业务模块按稳定职责组织，不按单页名称零散堆放。
- 登录授权、支付、订阅消息、分享、埋点、环境配置和错误上报应沉淀为统一模块，而不是分散在多个页面生命周期中。
- 分包、公共组件、公共 hooks、公共 service 和跨页面状态应统一规划，不在项目增长后再被动拆包。

## 生命周期与状态要求

- 应明确应用启动、登录态恢复、页面进入离开、前后台切换、分享回流和异常恢复的处理路径。
- 应区分页面局部状态、跨页面共享状态、本地缓存状态和服务端同步状态，不把所有状态混进单一全局仓库。
- 对授权拒绝、网络不可用、用户中断支付、订阅消息失败、平台接口限流或审核回退等典型异常路径，应给出降级和恢复策略。
- 关键业务流程应考虑中断恢复能力，避免用户切出宿主或返回首页后流程状态直接丢失。

## 平台能力接入要求

- 能力接入前先确认使用场景、授权范围、失败处理和平台差异，不默认所有平台同能力同实现。
- 登录、手机号授权、支付、订阅消息、分享、定位、相机、文件、剪贴板、扫码和客服等能力，必须说明触发条件、用户授权时机和失败反馈。
- 涉及平台 API 时，应优先复用项目现有封装层，不并行引入第二套同类封装。
- 平台能力输出应有统一错误模型，不让页面层直接处理底层平台异常细节。

## 数据、网络与缓存要求

- 请求层应统一处理鉴权、错误拦截、重试、超时、日志和弱网场景。
- 本地缓存、草稿、临时数据和业务关键数据应区分对待，不把所有数据都当作永久可用缓存。
- openId、unionId、token、会话态、支付参数和敏感业务数据不得以不受控方式持久化，应遵循安全存储和后端签名要求并补读 [../07-security/00_README.md](../07-security/00_README.md)。
- 如应用依赖分享回流、订阅消息唤起或扫码进入，应明确数据一致性、过期策略和重复处理策略。

## 交互与体验要求

- 关键流程应优先保证反馈闭环，包括加载中、成功、失败、可重试和中断恢复状态。
- 不把桌面端高信息密度界面直接平移到小程序；应优先保证任务节奏、热区可点按性和宿主内导航的一致性。
- 页面返回、Tab 切换、分享进入、扫码进入和模板消息唤起后的落点应保持一致，不出现同一流程多套返回逻辑。
- 首屏加载、骨架屏和包体控制应服务于任务完成，不为视觉效果显著增加首屏负担或分包复杂度。
- 小程序界面除功能可用外，还应具备明确的信息主次、任务焦点和一致的设计语言，不交付只是能跑但体验平淡的宿主页面。

## 安全与发布要求

- 登录态、用户标识、手机号、支付、订阅消息、授权弹窗和数据上传应符合安全与隐私规范。
- 宿主配置、环境区分、提审版本、灰度策略和回退路径应在发布前明确。
- 应区分开发、测试、预发、生产环境的配置边界，不把测试环境能力残留到正式提审版本。
- 只要涉及账号体系、支付、企业数据或平台敏感能力，就应补读 [../07-security/00_README.md](../07-security/00_README.md)、[../05-tests/00_README.md](../05-tests/00_README.md) 和 [../06-devops/00_README.md](../06-devops/00_README.md)。

## 输出要求

- 输出小程序方案时，应说明宿主平台、交付形态、主要工程边界、平台能力依赖和发布影响。
- 如任务仍处于技术选型阶段，应继续结合 [12_小程序技术选型规范](12_小程序技术选型规范.md) 给出候选方案，而不是直接默认具体框架。
- 如任务属于具体页面或交互实现，也应说明其处于小程序工程中的哪一层，而不是只给页面示意。
- 如任务涉及版本治理或样板工程初始化，应继续结合 [20_小程序稳定版本基线清单](20_小程序稳定版本基线清单.md) 说明版本口径。

## 不建议的做法

- 把小程序工程等同于一套普通前端页面。
- 在未确认宿主平台和运行边界前，直接默认 H5 方案或 App 方案。
- 在同一项目里长期并存多套请求层、状态管理、支付封装或平台能力适配层。
- 忽略授权拒绝、支付中断、弱网、分享回流和提审失败等小程序高频异常路径。

## 提交前自检

- 是否明确这是小程序或平台型轻应用，而不是浏览器页面或真正移动 App。
- 是否明确了宿主平台、运行边界和主要平台能力依赖。
- 是否补齐了授权、失败恢复、提审发布和平台能力接入相关约束。
- 是否避免在用户未确认时默认唯一技术栈或唯一宿主方案。
