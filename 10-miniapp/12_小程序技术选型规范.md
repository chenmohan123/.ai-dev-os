# 小程序技术选型规范

## 目标

- 规范小程序与平台型轻应用在新项目和已有项目中的关键技术选型方式。
- 明确原生小程序开发、Taro、uni-app 及同类跨端方案分别适合什么场景，哪些结论必须先让用户确认。
- 避免在小程序方案设计时擅自拍板宿主平台、跨端方案、状态管理、分包策略和发布方式。

## 适用范围

- 适用于微信小程序及后续纳入的同类平台型轻应用的技术选型、方案确认和实现前决策。
- 涉及小程序工程共性边界时，应结合 [11_小程序通用规范](11_小程序通用规范.md) 使用。
- 涉及浏览器页面、H5、移动 Web 或 WebView 动态页时，应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。
- 涉及真正移动 App 工程、原生宿主或安装包分发时，应回到 [../09-mobile/00_README.md](../09-mobile/00_README.md)。

## 总体原则

- 新项目如未明确宿主平台、目标渠道、技术栈、分包策略、状态管理、平台能力依赖或测试方案，应先列出候选并让用户确认，不替用户拍板。
- 对真正以微信小程序为主交付形态的新项目，如未出现明确排除条件，默认应先把原生小程序方案作为优先评估对象，而不是从多个跨端方案中无差别平铺展开。
- 已有项目如已存在稳定小程序工程、分包结构、请求层、平台封装和提审链路，应优先复用项目现状。
- 选型优先考虑宿主平台约束、团队能力、交付速度、平台能力复杂度、长期维护成本和审核发布稳定性，而不是只看单点开发体验。
- 同一类基础能力只保留一套主方案，避免同一项目同时并存两套路由体系、两套状态管理或两套平台能力封装方式。
- 工程主方案选型与 UI 组件库选型应分层处理；原生小程序、Taro、uni-app 决定工程组织和运行方式，组件库只决定页面构建与设计语言，不应把两者混写成同一级结论。

## 新项目决策要求

- 在进入实现前，至少确认以下内容：目标宿主平台、是否必须跨多小程序平台复用代码、是否重度依赖平台能力、团队技术栈、分包策略、状态管理、平台 API 封装策略、测试策略和发布方式。
- 如用户未明确是否只面向微信小程序，应先确认是否存在支付宝、抖音或其它平台扩展目标，再讨论跨端方案。
- 如用户未明确是否需要深度平台能力、重支付链路、复杂分享回流或宿主限制较重，应先确认平台依赖强度，再讨论是否采用原生小程序方案。
- 如用户未明确提审方式、灰度策略、回退方式和环境切换，应先补问，而不是把发布流程当作后置细节。
- 如属于真正的新建小程序工程，应先从“是否优先评估原生小程序方案”开始判断；只有在出现明确转出条件时，才把 Taro、uni-app 或其它跨端方案提升为主候选。

## 已有项目决策要求

- 先检查当前项目的技术栈、目录结构、分包策略、状态管理、平台封装、发布脚本和测试链路。
- 如现有方案可满足需求，应直接复用，不新增第二套同类能力。
- 只有在现有方案明显无法满足多平台扩展、多人维护成本、平台能力接入或业务复杂度要求时，才讨论替换或升级。
- 讨论替换时，应明确迁移范围、平台影响、兼容成本、灰度方式和回退路径，不做隐式迁移。

## 关键方案选型边界

### 原生小程序方案

- 在真正以微信小程序为主交付形态的新项目中，原生小程序方案是默认优先评估方案，但不是无条件唯一答案。
- 适合宿主平台明确、平台能力依赖较强、需要贴合平台最佳实践且希望减少额外运行层的项目。
- 适合需要较低框架抽象成本、较强平台适配可控性和更直接提审发布链路的场景。
- 不适合一开始就明确要求多小程序平台高度复用，或团队完全不接受小程序原生栈维护成本的场景。
- 如项目高度依赖既有 React/Vue 生态，也要评估团队切换成本、平台 API 封装成本和长期治理成本。

### 原生小程序方案转出条件

- 如果项目明确要求跨多个小程序平台复用主要代码，应转向 Taro、uni-app 或其它跨端方案评估。
- 如果团队明确不接受原生小程序栈长期维护成本，而核心能力集中在 React 或 Vue 工程体系，可转向对应跨端方案。
- 如果项目已有稳定跨端小程序工程，仅做增量功能，不应强行切回原生方案，应优先复用现状。
- 如果平台能力依赖极重且跨端框架难以稳定封装，应回到原生小程序方案评估，不为“统一”牺牲可控性。

### Taro

- 适合团队已有 React/TypeScript 积累，希望在微信小程序及同类平台之间复用业务模型和工程组织的场景。
- 适合希望保持 React 开发体验，同时接受为平台差异补充适配层的项目。
- 不适合对平台最佳实践贴合度要求极高、平台能力复杂且团队缺少小程序底层调试能力的场景。
- 需重点评估运行差异、平台兼容性、升级成本和平台 API 二次封装复杂度。

### uni-app

- 适合团队已有 Vue 技术栈，希望在多个小程序平台或轻应用形态之间做较高复用的场景。
- 适合前端团队主导、业务逻辑复用优先且平台差异复杂度中低的项目。
- 不适合平台能力要求极深、定制化要求高且团队缺少宿主平台底层支撑的场景。
- 需重点评估平台兼容成本、调试链路、生态插件质量和后续治理成本。

### 其它跨端或宿主适配方案

- 如已有组织级统一框架或宿主适配层，应先复用现有体系，不新增第二套抽象层。
- 这类方案只适合存在明确组织沉淀、平台扩展目标和维护团队支撑的场景。
- 如无法说明长期维护收益，不应仅因“看起来更统一”而引入新的抽象层。

## UI 组件库选型边界

- 小程序 UI 组件库属于页面实现层决策，不替代工程主方案选型。
- 只有在用户已确认工程主方案，或至少已确认宿主平台、跨平台目标和团队技术栈后，才讨论是否引入特定组件库。
- 已有项目如果已存在稳定组件封装、主题 token、表单容器或页面模板，应优先复用现状，不因组件库偏好强行替换。
- 同一项目不应长期并存两套主 UI 组件体系，避免主题、交互语义、表单能力和页面模板割裂。

### Ant Design Mini

- 可作为小程序项目的候选 UI 组件库，但不应作为所有小程序项目的默认标准答案。
- 更适合支付宝小程序优先，或希望沿用 Ant Design 设计语言并接受其在微信小程序适配边界的项目。
- 适合表单、列表、结果页、轻量工作台等业务页面较多，且团队希望减少基础组件重复建设的场景。
- 可与原生小程序方案、uni-app 等主方案组合使用，但是否组合取决于宿主平台、适配成本和团队现状。
- 如果项目以微信小程序单平台为主且更强调宿主原生生态、平台最佳实践贴合度或已有稳定自研组件体系，不应默认引入 Ant Design Mini。
- 如果项目需要覆盖多个小程序平台，也应先验证组件兼容性、主题能力、构建链路和平台差异处理成本，再决定是否采用。

### Ant Design Mini 推荐条件

- 目标宿主明确包含支付宝小程序，或支付宝小程序是主交付平台之一。
- 团队已接受以 Ant Design 设计语言统一页面风格，并希望优先复用成熟业务组件。
- 页面类型以业务表单、结果页、列表页、轻工作台为主，而不是高度定制化营销视觉页面。
- 项目没有既有稳定的小程序组件体系，或现有组件体系维护成本已明显偏高。

### Ant Design Mini 不建议条件

- 微信小程序是唯一主平台，且团队更看重原生生态贴合度、微信社区沉淀或已有成熟微信侧组件方案。
- 页面需要大量高度定制化视觉表达，而不是以标准业务组件为主。
- 项目已经存在稳定组件库、主题体系和页面模板，替换收益不足以覆盖迁移成本。
- 团队尚未确认主工程方案，却先把组件库当成架构主决策推进。

## 常见决策维度

- 交付目标：是只做微信小程序，还是需要覆盖多个小程序平台。
- 团队能力：是以原生小程序团队为主，还是以 React/Vue 前端团队为主。
- 平台能力依赖：是否涉及登录授权、手机号授权、支付、订阅消息、分享、定位、相机、文件、客服或复杂平台接口。
- 发布形态：是否需要频繁提审、灰度发布、环境切换和版本回退。
- 生命周期：项目是短期验证、业务承载，还是长期演进的平台型产品。
- 治理复杂度：是否需要分包、公共组件体系、统一请求层、统一埋点和统一错误模型。
- 页面体系：是否需要统一设计语言、成熟业务组件库和主题能力，还是更适合基于原生组件或现有自研组件受控扩展。

## 常见组合建议

- 宿主贴合优先型：原生小程序方案 + 统一平台封装层 + 受控分包策略。适合以微信小程序为主、平台能力较重的团队。
- React 能力复用型：Taro + 统一 service 层 + 明确平台适配边界。适合 React 团队主导、存在多平台复用诉求的项目。
- Vue 能力复用型：uni-app + 统一能力封装 + 明确兼容策略。适合 Vue 团队主导、需要覆盖多平台的项目。
- 现状增强型：沿用已有小程序或跨端工程，只在当前基础上补缺能力。适合已有项目增量迭代。

## 快速决策矩阵

- 如果是新项目，且交付重点是微信小程序、平台能力较重、希望更贴合宿主最佳实践，应默认先评估原生小程序方案，并在没有明确转出条件时优先考虑原生方案。
- 如果是新项目，且团队以 React 为主、存在多平台复用诉求，同时平台能力复杂度中低，可优先考虑 Taro。
- 如果是新项目，且团队以 Vue 为主、存在多平台复用诉求，同时平台能力复杂度中低，可优先考虑 uni-app。
- 如果是已有项目做局部增量，不应直接默认整体迁移到另一套框架，应先评估沿用现状的边界。
- 如果主要需求只是浏览器活动页、移动 Web 页面或 WebView 承载的动态业务页，不应进入 10-miniapp，应回到 [../02-frontend/00_README.md](../02-frontend/00_README.md)。

## 默认优先评估与转出规则

- 默认优先评估只适用于真正以小程序为主交付形态的新项目，不适用于浏览器页面、移动 Web 页面或真正 App 工程。
- 默认优先评估不等于自动定案；仍需用户确认团队能力、平台扩展目标、平台能力复杂度和维护意愿。
- 一旦命中多平台复用要求明确、团队明确以 React/Vue 工程体系为核心、已有稳定跨端工程或组织已有统一方案等条件，应显式转出原生小程序方案，改评 Taro、uni-app 或其它可复用方案。

## 选型提问模板

- 这是新项目还是已有项目。
- 目标宿主平台是微信小程序单平台，还是多个小程序平台。
- 是否必须共享一套主要代码，还是可以接受宿主平台分别维护。
- 当前团队是以原生小程序为主、React 为主，还是 Vue 为主。
- 是否依赖登录授权、手机号授权、支付、订阅消息、分享、定位、文件、客服或其它复杂平台能力。
- 发布目标是频繁提审、灰度发布，还是以稳定低频提审为主。
- 本次是做增量功能、局部改造，还是准备新建一套平台型轻应用产品。
- 是否需要统一 Ant Design 风格的页面体系，或已有必须复用的组件库、主题 token 和页面模板。

## 选型结论输出要求

- 输出结论时，应明确区分推荐方案、备选方案、不建议方案三类信息。
- 每个候选方案至少说明适用场景、主要收益、主要限制、团队要求和迁移或维护成本。
- 如为新项目，应明确哪些点已由用户确认，哪些点仍待确认，不把待确认项写成既定事实。
- 如为已有项目，应明确哪些能力沿用现状，哪些能力只是局部增强，哪些能力确实需要架构调整。
- 如同时涉及 UI 组件库，应单独说明其与主工程方案的关系，不把组件库结论写成框架选型结论。
- 最终结论应能直接指导下一步进入工程初始化、目录规划或平台能力设计，而不是停留在泛泛比较。

## 选型结论模板示例

### 通用输出模板

- 项目类型：新项目或已有项目。
- 当前约束：目标宿主平台、现有技术栈、团队结构、平台能力依赖、发布要求。
- 推荐方案：本次建议采用的主方案。
- 推荐理由：为什么该方案更适合当前团队、业务复杂度和交付节奏。
- 备选方案：可行但不是首选的候选方案。
- 不建议方案：当前阶段不建议采用的方案及原因。
- 待确认项：仍需用户拍板的技术点。
- 下一步动作：进入工程初始化、目录规范、平台能力设计或发布方案设计。

### 新项目示例

- 项目类型：新项目。
- 当前约束：目标是微信小程序首发；团队以前端为主；存在登录授权、支付和订阅消息能力，但不要求一开始覆盖多个小程序平台。
- 推荐方案：原生小程序方案。
- 推荐理由：当前场景以单宿主平台交付为主，且平台能力较重，未出现明确转出条件，因此先按默认优先评估策略选择原生小程序方案。
- 备选方案：Taro。适合后续确认存在多平台复用需求且团队希望延续 React 工程体系的场景。
- 不建议方案：直接把该项目当成移动 Web 或真正移动 App 工程处理。当前交付宿主和发布链路都不匹配。
- 待确认项：是否存在多平台扩展计划；是否接受原生小程序栈长期维护；是否已有组织级跨端框架要求。
- 下一步动作：请用户确认待确认项后，进入小程序通用规范并继续补工程结构与平台能力分层方案。

### 已有项目示例

- 项目类型：已有项目。
- 当前约束：项目已是稳定 Taro 工程，已有统一登录、支付、埋点和提审链路；当前需求是新增一组业务模块。
- 推荐方案：沿用现有 Taro 工程，在现有架构上做增量扩展。
- 推荐理由：当前基础设施稳定，直接复用现状能降低迁移成本并保持提审链路可控。
- 备选方案：仅在现有工程明显无法满足平台能力或维护要求时，再单独评估回归原生小程序或更换跨端方案。
- 不建议方案：为单次业务增量整体迁移到另一套框架。当前收益不足以覆盖迁移风险。
- 待确认项：现有架构是否存在明确瓶颈；是否真的需要扩大平台覆盖范围。
- 下一步动作：先按现有工程结构落地需求，如出现明确基础设施缺口，再单独评估调整方案。

## 方案调整触发条件

- 当现有工程无法支撑平台扩展、平台能力接入或提审发布要求时，可以讨论升级或替换主技术栈。
- 当现有跨端框架兼容成本、平台 API 封装成本或性能问题已经明显影响交付时，可以讨论切换或分层重构。
- 当团队结构已经变化，例如从 React 主导转向原生小程序主导，或从 Vue 主导转向组织级统一方案主导，可以重新评估技术栈选择。
- 只要现有方案仍可满足需求，就不应仅因“社区更流行”或“看起来更统一”而发起迁移。

## 推荐决策顺序

- 先判断是新项目还是已有项目。
- 再确认目标宿主平台、交付形态和是否必须共享代码。
- 再确认平台能力复杂度、团队能力和发布要求。
- 只有在缺口明确存在时，才列出候选方案让用户选择。
- 方案确认后，再进入工程规范和具体实现。

## 禁止事项

- 禁止在用户未确认的情况下，直接假定原生小程序方案、Taro、uni-app 或其它跨端方案为唯一答案。
- 禁止在已有项目中无视现有工程和提审链路，强行整体迁移到另一套技术体系。
- 禁止同一项目中长期并存两套路由体系、两套主状态管理或两套平台能力封装体系。
- 禁止只给出选型结论，不说明适用场景、收益、限制和成本。

## 提交前自检

- 是否明确区分了新项目选型和已有项目复用两种场景。
- 是否先检查了宿主平台、团队能力、平台依赖和发布要求，而不是直接代替用户选框架。
- 是否对候选方案说明了收益、限制和维护成本。
- 是否避免把浏览器页面方案或真正移动 App 方案误判为小程序选型问题。
