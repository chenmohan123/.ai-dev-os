# 小程序稳定版本基线清单

## 目标

- 为小程序和平台型轻应用提供版本治理口径，避免模板工程和演示工程长期停留在明显落后的框架或宿主链路上。
- 明确小程序的“最新稳定版”不应等于盲目追 latest，而应等于“当前公开稳定快照或官方稳定工具链 + 目标宿主验证后锁定”。

## 使用原则

- 新项目、样板工程和演示工程默认优先采用当前公开稳定快照或宿主官方稳定链路。
- 已有项目默认优先复用现状，只有在现有版本已经影响平台兼容、维护效率或提审发布时才讨论升级。
- 小程序版本治理必须区分工程主方案与 UI 组件库，不把 Taro、uni-app、原生小程序和组件库混成一条结论。

## 快照说明

- 快照日期：2026-04-27
- 快照来源：npm view 查询结果与宿主官方稳定工具链原则

## 当前推荐快照

### Taro

- @tarojs/cli：4.2.0

适用场景：

- React 团队主导
- 存在多小程序平台复用诉求的小程序项目

### uni-app

- @dcloudio/uni-app：2.0.2-5000720260410001

适用场景：

- Vue 团队主导
- 需要覆盖多个平台型轻应用宿主的项目

### 原生小程序方案

- 默认口径：采用目标宿主官方稳定开发工具链和团队已验证的工程基线，不把单次网络查询结果直接当作团队固定版本。

适用场景：

- 微信小程序单宿主优先
- 平台能力依赖重、需要更贴宿主最佳实践的项目

## UI 组件库说明

- 小程序 UI 组件库属于页面实现层，不替代工程主方案基线。
- 如需引入 Ant Design Mini 或其他 UI 组件库，应在主方案确认后，再单独评估与宿主平台、构建链路和主题体系的兼容性。

## 不建议的做法

- 只升级跨端框架，而不验证宿主能力、提审流程和包体影响。
- 把 latest 直接当成小程序工程模板默认版本。
- 在原生小程序和跨端方案之间来回切换，却不固定团队已验证的版本组合。
