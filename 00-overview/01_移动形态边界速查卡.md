# 移动形态边界速查卡

## 目标

- 用一页内容快速判断当前需求到底属于移动 Web、微信小程序，还是移动 App 工程。
- 避免在 02-frontend、09-mobile 和 10-miniapp 之间来回跳转后仍无法确定入口。
- 把最常见的误判场景压缩成可直接执行的分流规则。

## 什么时候先读本页

- 你只知道“这是手机上用的东西”，但还不知道应该归到哪个目录。
- 需求里同时出现 H5、WebView、小程序、Flutter、React Native、原生 iOS、原生 Android 这类关键词。
- 你需要先判断交付形态，再决定后续该进入哪个目录的 00_README。

## 一句话判断法

- 如果主体运行在浏览器或 WebView 里，归 [../02-frontend/00_README.md](../02-frontend/00_README.md)。
- 如果主体运行在微信小程序或同类宿主平台里，归 [../10-miniapp/00_README.md](../10-miniapp/00_README.md)。
- 如果主体是安装包、原生宿主、商店分发或真正 App 生命周期，归 [../09-mobile/00_README.md](../09-mobile/00_README.md)。

## 三类形态的核心区别

### 1. 移动 Web

- 运行宿主：手机浏览器或 App 内 WebView。
- 典型关键词：H5、活动页、外链页、扫码页、移动表单页、WebView 页面。
- 典型能力边界：浏览器能力、前端页面路由、响应式布局、移动 Web 组件。
- 应进入：[../02-frontend/00_README.md](../02-frontend/00_README.md)。

### 2. 小程序 / 平台型轻应用

- 运行宿主：微信小程序或同类平台宿主。
- 典型关键词：微信小程序、订阅消息、提审、体验版、支付、分享回流、扫码进入、宿主 API。
- 典型能力边界：平台登录授权、支付、订阅消息、分享、平台审核发布、分包加载。
- 应进入：[../10-miniapp/00_README.md](../10-miniapp/00_README.md)。

### 3. 移动 App 工程

- 运行宿主：原生 App、跨端 App 宿主、安装包运行环境。
- 典型关键词：Flutter、React Native、iOS、Android、安装包、商店上架、签名、深链、推送、原生桥接。
- 典型能力边界：App 生命周期、原生能力、安装包构建、商店分发、原生或跨端工程结构。
- 应进入：[../09-mobile/00_README.md](../09-mobile/00_README.md)。

## 高风险误判场景

### 场景 1：手机上打开的页面

- 错误判断：只要在手机上显示，就归 09-mobile。
- 正确判断：先看运行宿主。浏览器或 WebView 里的页面归 02-frontend；宿主平台内运行的小程序归 10-miniapp；安装包内的真正 App 页面才归 09-mobile。

### 场景 2：企业微信或微信里打开的业务入口

- 错误判断：在微信里打开就一定是小程序。
- 正确判断：如果是链接页、H5 页、WebView 页，仍归 02-frontend；只有真正走小程序运行时和小程序 API 的才归 10-miniapp。

### 场景 3：需要支付、扫码或分享

- 错误判断：有支付和扫码就一定是 App 工程。
- 正确判断：先看能力来自哪个宿主。浏览器侧能力仍归 02-frontend；小程序支付和分享归 10-miniapp；原生支付和原生分享才归 09-mobile。

### 场景 4：想复用前端团队能力

- 错误判断：只要团队是前端，就应该归 02-frontend。
- 正确判断：团队背景不决定目录归属，交付宿主和运行时才决定。

## 不建议的做法

- 只看“运行设备是不是手机”，不看运行宿主和交付形态。
- 只看团队背景，不看产品交付方式和平台 API 来源。
- 在还没判断清楚入口前，就先开始写页面规范、App 规范或小程序规范。
