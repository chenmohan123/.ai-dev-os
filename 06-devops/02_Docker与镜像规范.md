# Docker与镜像规范

## 适用范围

- 适用于 Dockerfile、镜像构建、基础镜像选择、运行时配置和镜像发布流程。

## 目标

- 让镜像构建具备可复现、可追溯、最小运行面和稳定交付边界。
- 避免把开发环境状态、调试残留、密钥或无关依赖一起打进正式镜像。

## 基本要求

- 镜像以可复现、可追溯、可最小化运行面为目标，不把开发机状态直接打包进镜像。
- 基础镜像优先选择稳定、受维护、来源明确的官方或组织认可镜像。
- 构建与运行职责分离，尽量通过多阶段构建减少无关依赖进入最终镜像。
- 配置、密钥和环境差异通过运行时注入，不写死在镜像层。

## 构建设计要求

- 先明确构建输入、运行时依赖、启动方式和最终产物边界，再编写 Dockerfile。
- 构建阶段只承担编译、安装和打包职责，运行阶段只保留执行所需最小内容。
- 对语言运行时、系统包和构建工具版本应保持可追溯，不依赖隐式最新版本。
- 构建结果应能映射到源码版本、构建时间和流水线记录。

## 实现约束

- Dockerfile 保持层次清晰，避免一次 RUN 堆积过多隐式行为。
- 构建上下文应最小化，避免把无关源码、缓存和凭据打进镜像。
- 最终镜像只保留运行必需文件，不混入测试数据、调试工具和构建缓存。
- 镜像标签应能表达版本、来源或构建批次，避免长期只用 latest。

## 运行时要求

- 容器启动参数、配置、密钥、证书和环境差异通过环境变量、挂载或平台配置注入。
- 日志输出优先走标准输出或标准错误，不把容器内临时文件当主日志出口。
- 容器内进程模型应清晰，避免用启动脚本串联大量隐式预处理逻辑。
- 运行用户、文件权限、暴露端口和工作目录应最小化配置，避免默认高权限运行。

## 标签与制品要求

- 镜像标签至少能区分版本号、提交来源、构建批次或环境用途。
- latest 仅可作为辅助标签，不应成为唯一可用发布标识。
- 同一次构建产出的镜像应保留可追溯摘要，便于回滚和审计。
- 基础镜像更新应记录来源、更新时间和影响范围。

## 常见风险信号

- Dockerfile 中直接复制整个仓库、缓存目录或本地配置文件，说明构建边界失控。
- 构建阶段和运行阶段使用同一重量级镜像，说明没有做最小化收敛。
- 镜像里保留测试工具、包管理缓存或调试脚本，说明最终产物未清理。
- 发布时只能依赖 latest 找镜像，说明版本追溯能力不足。

## 验收清单

- 是否明确基础镜像来源和更新策略。
- 是否把配置、凭据和环境差异排除在镜像层之外。
- 是否控制了镜像体积、标签策略和制品可追溯性。

## 输出模板建议

- 构建输入：说明源码、依赖、基础镜像和构建工具来源。
- Dockerfile 结构：说明构建阶段、运行阶段和最终保留内容。
- 运行方式：说明启动命令、配置注入、日志输出和权限策略。
- 制品策略：说明标签规则、摘要记录和回滚定位方式。
- 风险与限制：说明当前体积、基础镜像风险或待清理项。

## 不建议的做法

- 把开发机生成的产物、凭据或本地配置直接 COPY 进镜像。
- 用一个大而全的基础镜像长期承载构建和运行。
- 在容器启动脚本里堆大量环境判断和隐式初始化逻辑。
- 正式发布只依赖 latest，而没有稳定版本标签和摘要记录。
