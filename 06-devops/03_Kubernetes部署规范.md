# Kubernetes部署规范

## 适用范围

- 适用于 Kubernetes 资源清单、Deployment、Service、ConfigMap、Secret、Ingress 和环境发布配置。

## 目标

- 让 Kubernetes 部署具备可重复交付、环境差异可控、发布可回滚和运行可观测的基础能力。
- 避免通过线上手改资源、复制多份 YAML 或拍脑袋探针参数维持集群运行。

## 基本要求

- 资源清单以环境可重复部署、可审查和可回滚为前提，不依赖线上手工修改。
- 应明确工作负载、副本策略、探针、资源请求限制和配置来源。
- 环境差异应通过明确的配置层表达，而不是复制多份近似 YAML 人工维护。
- 服务暴露、网络入口和敏感配置需与安全目录要求保持一致。

## 资源组织要求

- 资源按应用、环境或模块稳定组织，便于审查、差异比较和回滚。
- Deployment、Service、Ingress、ConfigMap、Secret、HPA、Job 等资源应围绕同一工作负载清晰归组。
- 环境差异通过 values、overlay、kustomize、模板变量或等价配置层表达，不通过复制粘贴整套 YAML 维护。
- 命名空间、标签、注解和选择器应有统一约定，便于筛选、观测和发布操作。

## 实现约束

- 部署清单应按应用、环境或模块组织，避免所有资源散落在单文件中。
- 就绪探针、存活探针和启动探针要与实际启动行为一致，不能机械照抄默认值。
- 资源 request 和 limit 需基于应用特征设定，避免完全缺失或随意拍值。
- 发布策略要说明滚动更新、回滚条件和异常时的止血方式。

## 配置与密钥要求

- 配置项与密钥分开管理，密钥不得以内联明文方式长期留在正式清单中。
- ConfigMap 和 Secret 应只承载当前工作负载真正需要的配置，不把整份环境变量全集全部挂入。
- 环境差异应通过明确键值和配置来源表达，不在容器启动命令里塞大量环境判断逻辑。

## 发布策略要求

- 明确滚动发布、蓝绿、金丝雀或等价策略，不默认所有应用都使用同一发布方式。
- 发布前确认副本数、最小可用实例、PodDisruptionBudget 或等价可用性策略是否满足要求。
- 变更涉及 ConfigMap、Secret、镜像、探针或资源限制时，应明确是否触发重建、滚动更新或灰度观察。
- 回滚路径应能定位到具体资源版本、镜像摘要和配置版本，而不是只依赖手工记忆。

## 运行观测要求

- 探针参数应基于实际启动时间、预热行为和依赖特征配置，避免误杀或长时间假健康。
- 关键工作负载应明确日志出口、监控指标、事件查看方式和告警接入点。
- 对 Job、CronJob、Consumer 等非 HTTP 工作负载，应定义完成条件、失败重试和积压观察方式。
- 资源 request/limit 应结合历史使用情况持续调整，而不是一经设置长期不复查。

## 常见风险信号

- 同一应用在不同环境维护多份高度相似但长期漂移的 YAML。
- 探针全部照搬默认值，启动慢的服务频繁被误杀。
- Secret 以明文或临时占位长期保留在版本库中。
- 发布后只能靠 kubectl 临时查看现状，却无法从仓库明确知道上一版本配置。

## 验收清单

- 是否能从仓库直接复现部署，而不是依赖集群内手工状态。
- 是否定义了探针、资源限制和配置来源。
- 是否考虑了环境差异、滚动发布和回滚路径。

## 输出模板建议

- 资源结构：说明工作负载、服务暴露、配置资源和环境分层方式。
- 发布策略：说明发布方式、回滚入口和异常止血策略。
- 配置来源：说明 ConfigMap、Secret、镜像版本和环境差异管理方式。
- 运行约束：说明探针、资源配额、扩缩容和观察指标。
- 风险与限制：说明当前集群依赖、人工步骤和待治理项。

## 不建议的做法

- 线上直接 kubectl edit 修资源，再事后补仓库。
- 为每个环境复制一套完整 YAML，靠人工维护差异。
- 使用默认探针和资源限制，却不结合应用真实启动与负载特征。
- 发布策略和回滚路径只存在于口头经验中，没有体现在配置和文档里。
