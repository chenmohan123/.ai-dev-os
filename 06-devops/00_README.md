# 目录索引

## 目标

- 为 DevOps 目录提供统一入口，先说明容器化、流水线、发布执行和脚本治理应该从哪里进入。
- 避免把 DevOps 仅理解为部署脚本，而忽略镜像治理、Kubernetes 资源、门禁设计和变更执行约束。

## 使用说明

- 进入 DevOps 规范目录后，默认先阅读本文件，再按脚本、容器、Kubernetes 或 CI/CD 主题进入对应分册。
- 如任务属于长时间运行脚本、批量处理、补跑恢复或执行审计，优先先读 [01_批量脚本编写规范](01_批量脚本编写规范.md)。
- 如任务涉及容器构建、镜像治理、部署清单或流水线门禁，继续阅读后续专题分册。
- 如任务跨发布、安全、数据库迁移或测试门禁，先读本目录对应分册，再补读相关目录。

## 什么时候先读本页

- 还不确定当前任务属于脚本治理、镜像构建、集群部署、流水线门禁还是发布执行。
- 需要先判断上线相关工作是落在 DevOps 目录，还是应先去数据库、安全或测试目录补齐前置约束。
- 当前任务涉及交付流程，但还没明确是“构建问题”“部署问题”还是“发布执行问题”。

## 按任务类型快速入口

- 做长时间运行脚本、并发执行、断点续跑和运行审计：先读 [01_批量脚本编写规范](01_批量脚本编写规范.md)。
- 做 Dockerfile、镜像构建、运行时配置和镜像发布：先读 [02_Docker与镜像规范](02_Docker与镜像规范.md)。
- 做 Kubernetes 资源编排、部署清单、环境隔离和滚动发布：先读 [03_Kubernetes部署规范](03_Kubernetes部署规范.md)。
- 做 CI/CD 流水线、门禁、制品归档和自动化发布：先读 [04_CI_CD规范](04_CI_CD规范.md)。
- 做发布执行清单、变更前检查、回滚准备和交付观察：先读 [05_发布与变更执行规范](05_发布与变更执行规范.md)。

## 与其它目录的配合方式

- DevOps 目录负责回答“如何构建、如何部署、如何放行、如何执行发布”。
- 通用交付门槛仍以 [../01-general/08_交付质量规范.md](../01-general/08_交付质量规范.md) 为基础。
- 涉及数据库变更时补读 [../04-database/04_数据库迁移规范.md](../04-database/04_数据库迁移规范.md)，涉及安全放行时补读 [../07-security/08_安全评审与发布检查清单.md](../07-security/08_安全评审与发布检查清单.md)。
- 涉及发布前验证、回归门禁和自动化校验时，再补读 [../05-tests/00_README.md](../05-tests/00_README.md)，不要让流水线门禁替代真实测试策略。

## DevOps 分册

- [01_批量脚本编写规范](01_批量脚本编写规范.md)：长时间运行、可恢复、可审计的批量脚本设计与验收要求。
- [02_Docker与镜像规范](02_Docker与镜像规范.md)：容器镜像构建、基础镜像选择、配置注入和发布治理要求。
- [03_Kubernetes部署规范](03_Kubernetes部署规范.md)：Kubernetes 资源组织、部署方式、环境隔离和运行约束。
- [04_CI_CD规范](04_CI_CD规范.md)：流水线阶段划分、门禁、制品管理和自动化发布要求。
- [05_发布与变更执行规范](05_发布与变更执行规范.md)：发布前检查、执行顺序、回滚准备和上线观察要求。

## 阅读顺序建议

- 做单次脚本或批量处理：先读 [01_批量脚本编写规范](01_批量脚本编写规范.md)。
- 做容器化交付：先读 [02_Docker与镜像规范](02_Docker与镜像规范.md)，再读 [04_CI_CD规范](04_CI_CD规范.md)。
- 做 Kubernetes 部署：先读 [03_Kubernetes部署规范](03_Kubernetes部署规范.md)，再按需要补读 Docker 与发布分册。
- 做上线交付：先读 [05_发布与变更执行规范](05_发布与变更执行规范.md)，必要时补读 [../07-security/08_安全评审与发布检查清单.md](../07-security/08_安全评审与发布检查清单.md) 和 [../05-tests/01_测试设计与分层规范.md](../05-tests/01_测试设计与分层规范.md)。

## 不建议的做法

- 把 DevOps 工作收缩成“写个部署脚本”，忽略镜像、资源、门禁和回滚准备。
- 还没区分脚本、镜像、Kubernetes、流水线和发布执行问题，就在同一轮里一起改。
- 只依赖 CI/CD 通过作为放行依据，不补读测试、安全和数据库迁移约束。
