# 批量脚本编写规范

## 目标

- 保证批量脚本在长时间运行、异常中断、失败重试场景下仍具备可恢复性和可追溯性。
- 保证运行结果能稳定落盘，而不是只存在内存或终端日志中。
- 保证结果、失败过程、运行参数都有结构化记录，便于复盘、补跑和验收。

## 适用范围

- 适用于批量处理类脚本，例如批量调用外部接口、批量数据清洗、批量转换、批量导入导出、批量识别与分析任务。
- 只要脚本存在并发执行、长时间运行、失败重试、结果回写、补跑恢复等需求，默认都应满足本文档要求。
- 如有特殊场景无法满足，需要在脚本说明或配套文档中明确写出例外原因。

## 核心原则

### 1. 结果必须可落盘

- 禁止只在全部任务完成后一次性输出最终结果。
- 每完成一条记录或一个处理单元后，应尽快把状态和结果写回磁盘。
- 即使脚本中途中断，已经完成的结果也必须保留。
- 断点恢复依据必须来自已落盘状态，不能只依赖终端日志或内存变量。

建议实现方式：

- 每条结果完成后立即写结果文件并保存。
- 先写入中间状态文件，例如 JSONL、CSV、SQLite，再同步导出最终结果文件。
- 同时保留恢复状态文件和结果导出文件，其中恢复状态文件作为断点续跑依据。

### 2. 必须支持断点续跑

- 脚本再次启动时，应自动识别已完成记录并跳过。
- 对于一条记录中部分步骤已完成、部分步骤未完成的情况，只补跑缺失部分。
- 断点判断逻辑必须稳定、可重复，不能靠人工判断日志。
- 恢复逻辑应优先依赖结构化状态字段，而不是模糊文本备注。

建议最少保留以下状态字段：

- 记录唯一标识。
- 当前处理状态。
- 已完成步骤或子任务状态。
- 最近更新时间。
- 累计尝试次数。

### 3. 必须结构化记录尝试次数和错误信息

- 不能只在备注字段中用自然语言描述执行过程。
- 尝试次数、最终状态、最后错误必须作为稳定字段输出。
- 如果一个记录包含多个处理步骤、多个外部服务或多个阶段，应分别记录各自状态。
- 文本备注可以保留，但不能替代结构化列。

建议最少保留以下字段：

- 最终状态。
- 总尝试次数。
- 首次失败原因。
- 最近一次失败原因。
- 最近一次成功时间或完成时间。

### 4. 必须保留失败历史

- 不能只保留最后一次错误。
- 每次失败都要记录时间、步骤、尝试序号、错误摘要。
- 最终成功的记录，也必须保留此前失败历史。
- 最终结果中至少应能看出该记录是否经历过失败，以及失败过多少次。

建议表现形式：

- 结果文件中保留失败历史摘要列。
- 完整失败历史写入 JSONL、日志文件或独立状态文件。

建议失败历史格式：

- 第 1 次：超时
- 第 2 次：接口限流
- 第 3 次：成功

### 5. 必须有明确的最小重试策略

- 单次瞬时失败不应直接作为最终失败。
- 在正式批量场景下，每条记录在判定最终失败前，应至少具备明确的重试次数配置。
- 默认值应采用保守且可解释的设置，不能静默降低重试次数。
- 如果因成本、接口限制、业务时效要求等原因降低默认重试次数，必须在说明中写明原因。

记录要求：

- 必须能看到总共尝试了几次。
- 必须能看到每一次失败的错误摘要。
- 必须能区分一次成功和重试后成功。

### 6. 必须区分不同完成结果

- 结果汇总时不能只区分成功和失败。
- 至少应区分：一次成功、重试后成功、达到最大重试次数后失败。
- 如果存在跳过、取消、部分完成等状态，也应显式定义并结构化输出。

### 7. 并发必须可控，不能瞬时打满

- 并发类批量脚本不能在启动瞬间把所有任务同时打到外部系统。
- 必须支持渐进启动或分批启动机制。
- 必须允许配置任务启动间隔，避免瞬时请求洪峰。
- 文档中要明确区分目标并发数和启动爬坡策略。

建议要求：

- 启动节流作为显式参数存在，例如启动间隔、分批大小、爬坡周期。
- 运行元数据中记录实际并发配置和启动策略。
- 如有需要，应支持按步骤或按外部服务分别设置不同并发与启动间隔。

### 8. 必须保留运行级别的审计信息

- 每次正式跑批都应产出一份运行元数据。
- 运行元数据不能只散落在终端输出中。
- 运行结束后应能复盘本次执行使用了什么输入、什么参数、跑出了什么结果。

建议最少记录以下信息：

- 启动时间。
- 输入来源。
- 输出路径。
- 并发数。
- 启动爬坡参数。
- 最大重试次数。
- 重试退避参数。
- 总记录数。
- 成功数。
- 失败数。
- 跳过数。

建议输出位置：

- 单独的 run summary JSON 文件。
- 结果文件中的运行信息工作表或附加区域。

### 9. 必须保证重跑幂等性

- 同一条记录的最终结果不能因重复执行而被重复追加，导致结果冲突。
- 补跑时应覆盖未完成状态，或通过明确版本机制替换旧结果。
- 不能因为脚本多次执行，就在同一条记录上不断叠加冲突文本。
- 幂等策略应在数据结构和写回逻辑中显式体现，不能只靠人工约束。

## 输出结果规范

- 最终结果文件应同时适合人工查看和程序继续处理。
- 核心状态字段应保持稳定命名，不要频繁变化。
- 结构化列应优先满足筛选、统计、补跑、审计需求。
- 备注字段只用于补充说明，不承担核心状态表达职责。

## 推荐实现顺序

建议按下面顺序落地：

1. 增量落盘。
2. 断点续跑。
3. 结构化记录状态、尝试次数和最后错误。
4. 保留完整失败历史。
5. 固化并发爬坡与重试策略。
6. 增加运行级别审计信息。
7. 完善幂等补跑机制。

## 验收标准

后续批量脚本至少应满足以下验收条件：

1. 人为中断脚本后，已完成结果仍能在磁盘上看到。
2. 再次启动时，脚本可以跳过已完成部分，只继续未完成部分。
3. 结果文件中可以直接看到每条记录的状态、尝试次数和最后错误。
4. 即使最终成功，也能追溯该记录之前是否失败过。
5. 高并发运行时，任务启动不是瞬时全部打满，而是按设定策略逐步抬升。
6. 每条记录在最终失败前都经历了预设的重试流程，且过程可追溯。
7. 跑批结束后，能够产出一份独立的运行汇总。
8. 失败重跑不会产生重复冲突结果。

## 提交前自检

- 是否支持中途中断后的结果保留。
- 是否支持依据落盘状态做断点续跑。
- 是否把状态、尝试次数、错误信息做了结构化输出。
- 是否保留了完整失败历史，而不是只保留最后一次错误。
- 是否显式记录了并发、重试、爬坡等关键参数。
- 是否保证重跑不会破坏已有结果。
- 是否能在没有终端日志的情况下完成复盘。

## 补充建议

- 长时间运行脚本优先采用状态文件 + 导出文件双轨设计。
- 对外部接口依赖较强的脚本，优先把失败分类标准、重试条件、不可重试错误单独定义清楚。
- 如果结果最终要交给业务方查看，导出层与执行状态层建议分离，避免为了展示牺牲恢复能力。
- 编写脚本时，先设计状态模型、恢复机制和结果结构，再补充执行逻辑，不要先把并发和调用流程写满再回头补状态管理。
