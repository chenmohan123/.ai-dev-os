# 目录索引

## 目标

- 为测试目录提供统一入口，先说明测试规范负责什么、什么时候进入本目录、如何与通用规范和交付流程配合。
- 避免把“测试”只理解为写测试代码，而忽略测试分层、回归范围、工具治理和发布前验证边界。

## 使用说明

- 进入测试规范目录后，默认先阅读本文件，再按测试策略或测试工具进入对应分册。
- 如任务属于需求验收、回归范围、测试分层或发布前验证，优先先读 [01_测试设计与分层规范](01_测试设计与分层规范.md)。
- 如任务明确涉及测试框架、Mock、夹具、数据准备或自动化执行方式，再读 [02_测试工具与实现规范](02_测试工具与实现规范.md)。

## 什么时候先读本页

- 还不确定当前任务应该补哪些测试层级，或测试边界该落到哪里。
- 需要先判断是补测试设计、回归范围，还是补工具、夹具和自动化实现。
- 当前任务已进入发布、验收或回归阶段，需要明确测试目录与交付质量规范的配合方式。

## 按任务类型快速入口

- 做单元测试、集成测试、端到端测试或回归范围设计：先读 [01_测试设计与分层规范](01_测试设计与分层规范.md)。
- 做测试框架选择、测试数据、Mock、夹具和自动化落地：先读 [02_测试工具与实现规范](02_测试工具与实现规范.md)。

## 与其它目录的配合方式

- 测试目录负责回答“测什么、测到哪一层、如何落地执行”。
- 通用测试口径和交付门槛仍以 [../01-general/07_通用测试规范.md](../01-general/07_通用测试规范.md) 和 [../01-general/08_交付质量规范.md](../01-general/08_交付质量规范.md) 为基础。
- 具体业务场景仍回到对应专题目录，例如前端测试补读 [../02-frontend/00_README.md](../02-frontend/00_README.md)，后端测试补读 [../03-backend/00_README.md](../03-backend/00_README.md)。
- 涉及数据库迁移、安全发布或流水线门禁时，再补读数据库、安全和 DevOps 目录，而不是让测试目录替代发布规范。

## 测试分册

- [01_测试设计与分层规范](01_测试设计与分层规范.md)：测试目标、分层边界、覆盖策略和验收要求。
- [02_测试工具与实现规范](02_测试工具与实现规范.md)：测试工具、夹具、Mock、数据准备和执行约束。

## 阅读顺序建议

- 先判断测什么、测到哪一层：先读 [01_测试设计与分层规范](01_测试设计与分层规范.md)。
- 再决定怎么搭测试和如何落地自动化：读 [02_测试工具与实现规范](02_测试工具与实现规范.md)。
- 涉及发布门禁或交付标准时，可补读 [../01-general/08_交付质量规范.md](../01-general/08_交付质量规范.md)。

## 不建议的做法

- 把测试目录等同于“单元测试代码目录”，忽略回归范围、验收边界和发布前验证。
- 还没判断测试目标和层级，就直接先搭框架、写 Mock 或堆自动化脚本。
- 只看测试目录，不回到前端、后端、数据库或发布目录补齐场景约束。
