# 目录索引

## 目标

- 为数据库目录提供统一入口，先说明建模、命名、索引和迁移分别该从哪里进入。
- 避免把数据库工作只理解成写 SQL 或改表结构，而忽略命名协同、索引设计、迁移风险和发布配合。

## 使用说明

- 进入数据库规范目录后，默认先阅读本文件，再按设计、命名、索引或迁移主题进入对应分册。
- 如任务涉及表结构设计或字段建模，优先从 [01_数据库设计规范](01_数据库设计规范.md) 开始。
- 如任务是上线变更、字段调整或历史表演进，必须同时阅读 [04_数据库迁移规范](04_数据库迁移规范.md)。

## 什么时候先读本页

- 还不确定当前任务属于库表设计、命名治理、索引优化还是上线迁移。
- 需要先判断数据库改动是设计问题、查询问题，还是发布变更问题。
- 当前任务虽然由后端功能触发，但还没明确要不要补读数据库规范来约束数据模型和迁移方式。

## 按任务类型快速入口

- 做库表建模、主外键、字段约束和范式设计：先读 [01_数据库设计规范](01_数据库设计规范.md)。
- 做库名、表名、字段名、索引名和关联表命名统一：先读 [02_数据库命名规范](02_数据库命名规范.md)。
- 做查询优化、索引规划、唯一约束或慢查询治理：先读 [03_数据库索引规范](03_数据库索引规范.md)。
- 做表结构变更、数据修复、上线脚本和回滚准备：先读 [04_数据库迁移规范](04_数据库迁移规范.md)。

## 与其它目录的配合方式

- 数据库目录负责回答“数据模型怎么设计、命名怎么统一、查询怎么支撑、迁移怎么执行”。
- 涉及接口模型、服务边界和数据访问分层时，补读 [../03-backend/00_README.md](../03-backend/00_README.md)。
- 涉及数据脱敏、敏感字段、审计或发布安全时，补读 [../07-security/00_README.md](../07-security/00_README.md)。
- 涉及回归验证、发布前确认和上线门禁时，补读 [../05-tests/00_README.md](../05-tests/00_README.md) 和 [../06-devops/00_README.md](../06-devops/00_README.md)。
- 通用命名和交付门槛仍以 [../01-general/05_通用命名规范.md](../01-general/05_通用命名规范.md) 和 [../01-general/08_交付质量规范.md](../01-general/08_交付质量规范.md) 为基础。

## 数据库分册

- [01_数据库设计规范](01_数据库设计规范.md)：实体建模、字段约束、主键策略和关联关系设计要求。
- [02_数据库命名规范](02_数据库命名规范.md)：数据库对象统一命名和语义一致性要求。
- [03_数据库索引规范](03_数据库索引规范.md)：索引设计、查询支撑、唯一性和写入成本平衡要求。
- [04_数据库迁移规范](04_数据库迁移规范.md)：DDL、DML、灰度迁移、回滚和上线验证要求。

## 阅读顺序建议

- 新表设计：先读 [01_数据库设计规范](01_数据库设计规范.md)，再读 [02_数据库命名规范](02_数据库命名规范.md)。
- 查询优化：先读 [03_数据库索引规范](03_数据库索引规范.md)，必要时回看设计分册。
- 上线变更：先读 [04_数据库迁移规范](04_数据库迁移规范.md)，再回到设计或索引分册补齐细节。

## 不建议的做法

- 把数据库问题简单收缩成“改几列字段”或“补一个索引”，不先判断设计和迁移影响。
- 只从后端代码角度调整数据结构，不回到数据库目录确认命名、索引和迁移约束。
- 在没有回滚、验证和发布协同方案前，直接执行数据库变更。
