# 数据库设计规范

## 适用范围

- 适用于关系型数据库的库表建模、字段设计、主键策略和关联关系设计。

## 目标

- 让表结构能够稳定表达业务实体、状态语义和长期演进边界。
- 避免为了短期页面展示或一次性流程，把库表设计成难维护的临时拼装结构。

## 基本要求

- 表设计以业务实体和稳定关系为中心，不以页面字段临时拼接表结构。
- 每张核心表应明确主键、唯一约束、状态字段、审计字段和删除策略。
- 字段类型应按真实语义选择，避免用字符串兜底所有结构化数据。
- 关联关系优先保证可读性和一致性，再考虑局部冗余优化。

## 设计输出最低要求

- 实体边界：这张表表达什么业务对象，不表达什么。
- 核心字段：主键、业务唯一键、状态字段、时间字段和审计字段。
- 关系设计：与其它表是一对一、一对多还是多对多，关联约束如何表达。
- 生命周期：创建、变更、归档、删除和恢复策略。
- 查询预期：高频查询条件、分页方式、排序方式和归档需求。
- 演进约束：未来扩展点、兼容要求和迁移影响。

## 设计约束

- 一张表只表达一个清晰主题，不混杂多种无关业务语义。
- 状态字段、枚举字段和标记位需有明确取值范围与说明。
- 可空字段必须能解释业务原因，不能用可空掩盖流程不确定性。
- 审计字段如创建时间、更新时间、创建人、更新人应保持统一策略。

## 字段设计要求

- 字段名应能直接表达业务语义，不用 pageData、ext1、remark2 这类兜底字段掩盖设计问题。
- 数值、金额、数量、状态、时间、布尔标记应使用匹配语义的类型，不以字符串统一兜底。
- 对外部标识、业务单号、幂等键、租户标识等稳定标识字段，应明确唯一性和使用范围。
- 默认值必须符合业务含义，不用默认值掩盖未初始化或流程缺口。

## 关联与冗余要求

- 关联关系优先保持清晰可追踪，避免通过文本拼接、逗号分隔或 JSON 字段隐式表达结构关系。
- 冗余字段应有明确收益，例如提高查询效率、降低联表成本或保留快照语义。
- 冗余数据一旦存在，必须明确更新时机、一致性要求和修复路径。
- 多对多关系优先使用中间表表达，而不是把多值关系硬塞进单字段。

## 生命周期设计要求

- 明确是物理删除、逻辑删除、归档迁移还是禁止删除。
- 如使用逻辑删除，应明确默认查询过滤策略、唯一键冲突处理和恢复路径。
- 状态流转应能映射到明确字段，不用多个分散标记位隐式组合业务状态。
- 对历史记录、审计记录和快照数据，应明确保留时长和清理方式。

## 常见风险信号

- 一张表同时承载主记录、配置、流程状态和统计汇总，说明主题过杂。
- 字段大量可空且没有业务解释，说明流程或约束没有收敛。
- 为了兼容前端展示直接把结构化关系打平成大量重复字段，说明建模被页面驱动。
- 同一业务概念在多张表中重复出现但没有统一约束，说明实体边界不稳定。

## 验收清单

- 是否清楚定义了主键、唯一性、状态语义和删除策略。
- 是否避免了过宽字段、语义不明字段和页面驱动建模。
- 是否为后续查询、归档和迁移预留了稳定结构。

## 输出模板建议

- 表目标：说明表的业务主题和边界。
- 字段说明：列出主键、唯一键、状态字段、时间字段和关键业务字段。
- 关系说明：列出与其它表的关系和约束。
- 生命周期：说明删除、归档、恢复和审计策略。
- 风险与限制：说明当前冗余、兼容要求和未来迁移注意事项。

## 不建议的做法

- 为了赶进度先加通用扩展字段，后续长期不清理。
- 使用字符串或 JSON 字段承载本应稳定建模的结构化数据。
- 同时用多个布尔位和状态位表达一个核心业务状态。
- 在没有生命周期设计的前提下直接引入逻辑删除。
