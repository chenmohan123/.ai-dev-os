# 数据库索引规范

## 适用范围

- 适用于关系型数据库的主键索引、唯一索引、联合索引和查询优化设计。

## 目标

- 让索引设计基于真实查询模式、约束需求和写入成本，而不是凭经验堆积。
- 避免把索引当成 SQL 质量差、表结构不清或分页方式不合理的通用补丁。

## 基本要求

- 索引为查询模式服务，不为“可能会用到”预先堆积索引。
- 建索引前先明确高频查询条件、排序条件和返回规模。
- 唯一索引优先承载数据一致性约束，不仅作为查询优化手段。
- 联合索引应按过滤性、排序和最左匹配原则设计。

## 索引设计步骤

- 先确认高频 SQL、筛选条件、排序条件、分页方式和返回列。
- 再判断现有主键、唯一键和已有联合索引是否已经覆盖目标查询。
- 再决定是优化 SQL、调整查询路径还是新增索引，不默认直接建索引。
- 新增索引前明确收益场景、命中条件和可能增加的写入成本。

## 设计约束

- 低区分度字段不应单独建立普通索引，除非有明确场景支撑。
- 重复、重叠和长期不命中的索引要定期清理。
- 新增索引时需同步评估写入放大、锁竞争和存储成本。
- 查询优化优先从 SQL、筛选条件和分页方式入手，不把索引当万能补丁。

## 联合索引要求

- 字段顺序应围绕主要筛选条件、排序条件和最左匹配规则设计。
- 不要为多个轻微差异查询随意复制多组相似联合索引。
- 如查询依赖范围条件、排序和分页组合，应重点评估字段顺序对扫描范围的影响。
- 覆盖索引仅在收益明确时使用，不为减少一次回表而大幅增加索引宽度。

## 唯一索引与约束要求

- 唯一索引优先用于表达业务唯一性，而不是只在应用层口头约定。
- 对逻辑删除场景，应评估唯一索引与删除标记、恢复路径和历史数据兼容的关系。
- 对租户隔离、多环境、多版本并存等场景，应明确唯一性的实际作用域。

## 查询优化边界

- 慢查询不应只从加索引入手，还应回看 SQL 写法、筛选条件、分页方式和返回列规模。
- 如果查询本身不稳定，例如条件组合过多、临时排序频繁变化，应优先收敛查询模式。
- 对统计类、大范围扫描或报表类查询，应优先评估归档、汇总表或离线方案，而不是直接堆索引。

## 发布前后验证要求

- 发布前确认目标 SQL 的执行计划、扫描行数、排序行为和回表情况。
- 发布前评估新增索引对写入、更新、删除、锁等待和存储占用的影响。
- 发布后观察慢查询、写入延迟、索引命中率和未预期的执行计划变化。
- 对效果不明显或副作用过大的索引，应准备回退或清理方案。

## 常见风险信号

- 同一张表存在多组字段顺序略有不同的相似联合索引。
- 为低区分度状态字段单独建索引，却没有高频查询支撑。
- 每次出现慢查询都新增索引，但不复查旧索引和 SQL 本身。
- 分页查询大量依赖深分页 offset，却试图只靠索引解决性能问题。

## 验收清单

- 是否为核心查询建立了明确支撑索引。
- 是否存在重复索引、顺序不合理索引或无效索引。
- 是否在建索引前后对查询计划和写入成本做了验证。

## 输出模板建议

- 目标查询：列出要支撑的 SQL 或查询模式。
- 现状分析：说明当前执行计划、瓶颈点和已有索引情况。
- 索引方案：说明新增、调整或清理哪些索引，以及字段顺序原因。
- 风险评估：说明写入放大、存储占用、锁竞争和兼容影响。
- 验证结果：说明发布前后执行计划和性能变化。

## 不建议的做法

- 未看执行计划就直接加索引。
- 为每个查询条件排列组合建一套索引。
- 在没有业务唯一性要求时滥用唯一索引。
- 只关注读性能，不评估写入和维护成本。
