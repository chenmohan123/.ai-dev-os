# Git 与 PR 规范

## 目标

让代码协作过程具备可追踪性、可审阅性和可回滚性。

## 适用范围

- 适用于功能开发、缺陷修复、重构、文档调整、配置变更和发布前协作流程。
- 只要改动会进入分支、提交、PR、评审、合并或回滚链路，就应遵循本规范。

## 分支策略

- 主分支保持可发布或至少可验证状态，不直接在主分支上堆积未完成改动。
- 功能开发、缺陷修复、重构、专项治理和发布准备应使用独立分支。
- 一个分支只承载一个明确主题，不把多个无关任务混在同一分支中推进。
- 长周期任务应拆成多个阶段性分支或阶段性 PR，而不是长期悬挂超大分支。

## 分支命名建议

- 功能分支：feature/主题名。
- 缺陷修复：fix/问题名。
- 重构整理：refactor/主题名。
- 文档调整：docs/主题名。
- 紧急修复：hotfix/问题名。
- 主题名应直接表达业务或问题语义，不使用 temp、test、misc 这类低信息量名称。

## 分支原则

- 功能、修复、重构使用独立分支。
- 一个分支聚焦一个主题，避免混入无关变更。
- 长周期任务按阶段拆分，而不是积累超大分支。

## 提交前整理要求

- 先清理无关改动、调试输出、注释掉的旧代码和临时文件，再进行提交。
- 重命名、纯格式化和逻辑修改尽量拆开，不混成难以回看的单次提交。
- 提交前至少完成一项与改动范围匹配的可执行验证；若无法验证，应明确记录原因。
- 如改动涉及接口、数据库、配置、权限、迁移或发布流程，应在提交或 PR 描述中显式标记。

## 提交原则

- 提交要小而完整，保持语义清晰。
- 提交消息说明变更意图，而不是只写 update 或 fix。
- 避免把重命名、格式化和逻辑修改混在同一提交。

## 提交消息建议

- 标题直接说明本次变更意图，优先使用“动作 + 对象 + 目的”的表达方式。
- 标题不必追求花哨，但应让评审者在不打开 diff 时也能大致理解改动主题。
- 如需要补充背景、风险或迁移信息，可在正文继续说明，而不是把所有信息塞进标题。

## 提交消息示例

- 新增用户导出任务失败重试记录。
- 修复审批回调重复入库问题。
- 重构订单同步任务的状态落盘逻辑。
- 补充数据库迁移发布前检查说明。

## PR 要求

- PR 描述必须包含目标、范围、风险、验证方式。
- 若涉及接口、数据库、配置、权限，必须显式标出。
- 若存在已知限制或后续步骤，要在 PR 中写明。
- 评审前先自行检查明显问题和无关改动。

## PR 描述建议模板

- 目标：这次改动要解决什么问题。
- 范围：改了哪些模块、文件或流程。
- 风险：可能影响哪些接口、配置、数据、权限或发布路径。
- 验证：执行了哪些测试、命令、人工验证或发布前检查。
- 已知限制：当前未覆盖、待后续处理或有前提条件的部分。

## 评审前自检

- 改动目标是否能在 PR 标题和描述中被快速理解。
- 是否已经清理无关 diff、调试代码和偶发性提交残留。
- 是否把接口、数据库、配置、权限、迁移或发布影响显式写出。
- 是否补齐了与改动范围匹配的验证记录。
- 是否能明确说清这次改动怎么回滚、怎么继续跟进。

## 审阅协作

- 回复 review 意见时要说明处理结果或保留原因。
- 对有争议的设计点，要记录权衡而不是只改代码。
- 合并前确认 CI、验证、迁移与回滚条件。

## 合并前检查

- CI 和必要验证已通过，或已明确说明未执行原因与风险。
- 评审意见已处理完毕，未处理项有保留理由。
- 如涉及迁移、配置或发布步骤，相关说明已补齐。
- 如涉及数据库、脚本、权限或自动化任务，已确认回滚或止血路径。

## 不建议的做法

- 一个分支里同时混入多个主题，只靠 PR 备注解释。
- 提交消息长期使用 fix、update、调整一下 这类无信息量表述。
- PR 描述只贴截图或只写“见代码”。
- 明知有数据库、配置、权限或发布影响，却不在 PR 中标识。
- 评审意见只回复“已处理”，但不说明改动点或保留原因。
