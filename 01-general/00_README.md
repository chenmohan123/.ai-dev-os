# 目录索引

## 目标

- 为通用规范目录提供统一入口，先说明需求、架构、编码、命名、评审、测试和交付这些跨技术栈规则该如何进入。
- 避免在已经落入具体技术目录后，忽略仍然适用的通用边界、命名口径和交付要求。

## 使用说明

- 进入通用规范目录后，默认先阅读本文件，再按任务阶段进入需求、架构、编码、评审或交付相关分册。
- 本目录负责跨技术栈都适用的共性规则，不替代前端、后端、数据库、测试、DevOps、安全或 Electron 的专题规范。
- 如任务已经明确落在某个技术目录，本目录优先用来补齐通用边界，而不是替代专题正文。
- 如任务需要落到具体语言实现，再进入 [languages/00_README.md](languages/00_README.md)。

## 什么时候先读本页

- 还不确定当前任务先该做需求澄清、架构判断、编码约束、命名统一、评审还是交付检查。
- 当前任务虽然落在某个技术目录，但明显还缺通用规则，例如命名、评审、测试边界或交付口径。
- 需要先建立跨技术栈都一致的工作方式，再进入前端、后端、数据库或 DevOps 目录。

## 按任务类型快速入口

- 做需求澄清、缺口识别、反问清单和实施前阻塞检查：先读 [01_需求分析与澄清规范](01_需求分析与澄清规范.md)。
- 做系统分层、依赖方向、模块边界和通用架构约束：先读 [02_通用架构规范](02_通用架构规范.md)。
- 做编码风格、目录治理、结构分层和冗余清理：先读 [03_通用编码规范](03_通用编码规范.md)。
- 做提交规范、分支协作和 PR 流程：先读 [04_Git与PR规范](04_Git与PR规范.md)。
- 做目录、文件、类型、函数和变量命名统一：先读 [05_通用命名规范](05_通用命名规范.md)。
- 做代码评审、结构性问题识别和评审结论输出：先读 [06_代码评审规范](06_代码评审规范.md)。
- 做通用测试边界、测试覆盖和验证要求：先读 [07_通用测试规范](07_通用测试规范.md)。
- 做交付质量门槛、上线前检查和阶段验收：先读 [08_交付质量规范](08_交付质量规范.md)。
- 做多终端体验质量、视觉层级和跨端高质量界面要求：先读 [09_多终端体验与高质量输出规范](09_多终端体验与高质量输出规范.md)。
- 做语言层面的实现约束：进入 [languages/00_README.md](languages/00_README.md)。

## 与其它目录的配合方式

- 通用目录负责回答“做事的方法和共性边界是什么”，例如需求澄清、架构约束、编码习惯、命名、评审和交付口径。
- 具体实现与场景规则仍回到前端、后端、数据库、测试、DevOps、安全或 Electron 目录。
- 语言层实现约束进入 [languages/00_README.md](languages/00_README.md)，不要让通用分册替代具体语言规范。
- 当任务同时涉及专题规则和通用规则时，优先用本目录补齐共性边界，再回到专题目录落地。

## 阅读顺序建议

- 新任务启动：先读 [01_需求分析与澄清规范](01_需求分析与澄清规范.md)，再根据需要进入架构或编码分册。
- 方案设计与实现：优先读 [02_通用架构规范](02_通用架构规范.md) 和 [03_通用编码规范](03_通用编码规范.md)。
- 如果任务直接交付可见界面：补读 [09_多终端体验与高质量输出规范](09_多终端体验与高质量输出规范.md)。
- 提交与评审阶段：补读 [04_Git与PR规范](04_Git与PR规范.md)、[05_通用命名规范](05_通用命名规范.md) 和 [06_代码评审规范](06_代码评审规范.md)。
- 验证与交付阶段：补读 [07_通用测试规范](07_通用测试规范.md) 和 [08_交付质量规范](08_交付质量规范.md)。
- 已明确具体实现语言时：进入 [languages/00_README.md](languages/00_README.md) 补齐语言层约束。

## 不建议的做法

- 只因为任务落在某个技术目录，就完全跳过通用需求、命名、评审和交付规则。
- 把通用目录当成万能替代目录，用来覆盖前端、后端、数据库或安全的专题规范。
- 在没有先补齐通用工作方式前，直接进入具体实现细节，导致跨目录口径不一致。
