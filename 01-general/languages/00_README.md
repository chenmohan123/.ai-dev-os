# 目录索引

## 目标

- 为 Java、.NET、Python、JavaScript/TypeScript 提供统一的语言层实现入口。
- 帮使用者快速判断何时该补读语言规范，以及语言规范与前端、后端、测试等专题规范如何配合。

## 使用说明

- 进入语言分册目录后，先根据当前实现语言进入对应分册，再回到所属技术目录结合场景规范一起使用。
- 本目录只定义语言层面的实现约束，不替代前端、后端、测试或 DevOps 的专题规范。
- 如任务同时涉及架构、目录结构、测试或发布，先读语言分册，再补读上层目录对应规范。

## 什么时候先读本页

- 已明确当前改动落在某种语言实现上，但还不确定应该读哪篇语言规范。
- 任务同时涉及通用编码规范和具体语言实现约束，需要先选主语言入口。
- 代码库存在多语言协作，需要先判断主实现语言和辅助语言的阅读优先级。

## 语言选择规则

- Java 服务、Java 工具和 Java 批处理任务：进入 [01_Java语言规范](01_Java语言规范.md)。
- .NET Web API、后台服务、桌面端 C# 逻辑和 .NET 工具：进入 [02_.NET语言规范](02_.NET语言规范.md)。
- Python 服务、脚本、任务处理和自动化工具：进入 [03_Python语言规范](03_Python语言规范.md)。
- JavaScript/TypeScript 前端、Node.js 程序和共享前端工具：进入 [04_JavaScript_TypeScript语言规范](04_JavaScript_TypeScript语言规范.md)。
- 跨多语言任务优先选主实现语言为主读分册，其它语言只补读相关约束，不并行通读全部分册。

## 按语言快速入口

- 做 Java 实现规范、代码组织和常见工程约束：先读 [01_Java语言规范](01_Java语言规范.md)。
- 做 .NET/C# 实现规范、项目结构和通用编码约束：先读 [02_.NET语言规范](02_.NET语言规范.md)。
- 做 Python 实现规范、模块组织和运行约束：先读 [03_Python语言规范](03_Python语言规范.md)。
- 做 JavaScript 或 TypeScript 实现规范、模块边界和前端通用代码约束：先读 [04_JavaScript_TypeScript语言规范](04_JavaScript_TypeScript语言规范.md)。

## 语言分册

- [01_Java语言规范](01_Java语言规范.md)：Java 代码组织、类型设计、异常处理和工程实现约束。
- [02_.NET语言规范](02_.NET语言规范.md)：C#/.NET 代码结构、命名、异常模型和依赖组织约束。
- [03_Python语言规范](03_Python语言规范.md)：Python 模块组织、类型提示、异常处理和运行边界约束。
- [04_JavaScript_TypeScript语言规范](04_JavaScript_TypeScript语言规范.md)：JavaScript/TypeScript 的模块边界、类型表达和前端通用实现约束。

## 与其它目录的配合方式

- 语言分册负责具体语言的实现习惯、类型表达、异常处理和常见结构约束。
- 通用规则仍以 [../03_通用编码规范.md](../03_通用编码规范.md)、[../05_通用命名规范.md](../05_通用命名规范.md) 和 [../06_代码评审规范.md](../06_代码评审规范.md) 为基础。
- 场景规则仍以对应技术目录为主，例如前端任务回到 [../../02-frontend/00_README.md](../../02-frontend/00_README.md)，后端任务回到 [../../03-backend/00_README.md](../../03-backend/00_README.md)。
- 涉及安全、测试、数据库或发布时，再补读相应专题目录，而不是让语言规范替代场景规范。

## 阅读顺序建议

- 先按实现语言选择对应分册。
- 再回到 [../00_README.md](../00_README.md) 或相关技术目录 00_README，补齐架构、场景和交付约束。
- 如任务跨多语言，优先以主实现语言为准，其它语言分册只补读相关段落。

## 不建议的做法

- 把语言规范当成完整项目规范使用，忽略前端、后端或测试专题目录。
- 因为仓库里存在多种语言，就机械地把所有语言分册都读一遍。
- 在没有明确主实现语言时，同时按多套语言规范改写同一段实现。
