# Java语言规范

## 适用范围

- 适用于 Java 服务、工具程序和通用库代码的实现层规范。

## 目标

- 让 Java 代码在类型设计、集合与空值语义、异常边界和公共抽象上保持一致可维护性。
- 避免因为工具类膨胀、继承层级失控、泛型滥用或空值语义混乱，逐步削弱实现层可读性。

## 基本要求

- 类型命名、包结构和职责边界应直接表达业务语义，避免缩写堆叠和工具类泛滥。
- 面向对象建模以职责清晰为先，不用继承层级掩盖真正的依赖关系。
- 公共能力抽取基于稳定复用场景，不把尚未稳定的业务逻辑下沉为通用组件。
- 异常、日志和配置使用统一约定，避免每个模块各自定义一套处理方式。

## 包与项目组织要求

- 包结构优先围绕业务域、职责层和部署边界组织，而不是只按 controller、service、util 这类技术分类机械拆分。
- 包名、模块名和公共库命名应能反映真实业务语义，避免 common、base、misc 长期吸收无关实现。
- 共享模块只承载稳定复用能力，不把业务专用逻辑提前下沉为跨项目公共依赖。

## 类型与集合语义要求

- 类、枚举、接口和值对象应按真实业务语义建模，不用一个大对象承载多层职责。
- Optional、集合、空字符串和 null 语义保持明确，不让同一字段同时表达多种状态。
- 泛型只在能明显提升复用性和可读性时使用，不为了抽象而抽象。
- DTO、VO、Entity、Command、Query 等不同职责模型应分开维护，不混用同一类型贯穿全链路。

## 契约与弱类型边界要求

- 对外契约、事件消息、配置模型和公共接口优先使用清晰可演进的显式类型，不长期依赖 Map、Object 或字符串键值对透传。
- 枚举、常量、状态值和错误码应集中管理，避免在多个包和模块里散落硬编码。
- 集合返回值优先表达稳定语义，不用 null 代替空集合，也不靠默认空字符串表达复杂状态。

## 实现约束

- 一个类只承担单一稳定职责，避免超级 Service、超级 Util 或超级 DTO。
- Optional、集合和空值语义保持明确，不用 null 同时表达“未知、无值、未加载”多种含义。
- Stream、函数式写法和泛型只在能提升可读性时使用，不为炫技牺牲可维护性。
- 公共常量、枚举和配置键保持收敛管理，不散落在各层代码中。

## 配置与框架边界要求

- 配置绑定、配置校验和环境差异处理应集中在启动或明确边界层，不在业务方法中零散读取系统属性、环境变量或配置中心值。
- Servlet、Spring 上下文、框架注解能力和基础设施适配应停留在边界层，不反向侵入核心业务模型。
- AOP、过滤器、拦截器和框架扩展点应用于横切关注点，不把核心业务规则隐式塞进框架生命周期。 

## 抽象与异常要求

- 继承用于稳定 is-a 关系，组合优先于为了复用代码而引入的脆弱继承层次。
- 工具类、静态方法和通用组件使用受控，不把业务规则不断沉积到 util 包。
- 业务异常、参数异常和系统异常应有明确边界，不依赖 message 文本临时区分。
- 日志记录应围绕关键上下文和业务动作，不把调试信息长期沉积为正式日志模式。

## 常见风险信号

- util、common、helper 包持续吸收业务逻辑，说明抽象边界失控。
- Stream 链路过长、嵌套过深，导致可读性反而下降。
- 同一概念在多个 DTO、Entity、Map 结构里重复表达，却没有统一语义来源。
- null、Optional 和空集合混用，调用方必须靠猜测处理分支。
- 关键契约长期退化成 Map、Object 或字符串键拼装，说明类型语义没有真正收敛。

## 验收清单

- 是否存在职责过大的类、难懂缩写或层级过深的继承结构。
- 是否统一了异常、日志和配置读取方式。
- 是否避免为局部方便引入难维护的泛型和工具类堆叠。
- 是否明确了包边界、弱类型契约和空值/集合语义。

## 输出模板建议

- 包与类型结构：说明包边界、核心类型和模型分层方式。
- 语义策略：说明 Optional、集合、异常和日志处理约定。
- 抽象策略：说明接口、继承、组合和公共组件抽取边界。
- 配置边界：说明配置绑定、框架上下文与业务模型的隔离方式。
- 风险与限制：说明当前工具类、泛型抽象或空值语义治理点。

## 不建议的做法

- 用 util、common、base 等泛化包长期吸收业务实现。
- 在没有稳定收益时堆叠泛型、Stream 和函数式写法。
- 用 null、空字符串和默认值混合表达多种业务状态。
- 通过继承层级复用实现细节，而不是围绕职责做稳定建模。
