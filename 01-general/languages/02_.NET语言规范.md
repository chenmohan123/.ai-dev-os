# .NET语言规范

## 适用范围

- 适用于 C#/.NET 服务、库、后台任务和通用组件的实现层规范。

## 目标

- 让 C#/.NET 代码在命名、类型表达、异步边界和依赖组织上保持一致的可维护性。
- 避免因为接口泛滥、扩展方法失控、静态工具堆叠或异步滥用，逐步把实现层变成难演进的隐式耦合结构。

## 基本要求

- 命名、命名空间和项目结构应与业务边界一致，不靠文件夹堆叠表达架构。
- 类型、接口和扩展方法的抽象要有稳定边界，避免为了测试或复用过度接口化。
- 异步、异常和资源释放行为保持一致，不能每个模块各自实现一套风格。
- 配置、选项和依赖注入使用框架推荐方式集中管理。

## 项目与命名空间组织要求

- 项目、程序集和命名空间优先围绕业务边界、部署边界和职责层组织，而不是只按技术类型拆目录。
- 命名空间应能反映业务模块和层级关系，避免出现 Shared、Common、Utils 长期吸收无关实现。
- 公共库只承载稳定复用能力，不把业务专用逻辑提前下沉到跨项目共享层。

## 类型与抽象要求

- class、record、struct、enum、interface 的选择应以语义稳定性和变更成本为先。
- 只有存在明确替换边界、测试隔离收益或多实现需求时，才引入接口抽象。
- 领域模型、请求模型、响应模型和配置模型分开表达，不让单一类型承担多层职责。
- 扩展方法只用于稳定、通用且不反转业务依赖的能力，不把核心业务规则塞进扩展类。

## 空值、集合与类型语义要求

- Nullable 引用类型、可空值类型和默认值语义应明确，不用 null 同时表达未设置、空结果和异常状态。
- 集合返回值优先表达稳定语义，避免用 null 代替空集合或靠魔法值表达状态。
- 枚举、常量和错误码应集中管理，不在多个模块中散落硬编码字符串和整数状态值。
- 对外公开接口、事件模型和配置模型优先使用清晰可演进的类型，而不是 Dictionary<string, object> 长期透传。

## 实现约束

- 类和方法名优先表达业务动作或领域概念，不用模糊后缀泛化一切职责。
- async/await 用于真实异步边界，不为了形式统一把同步逻辑机械包装成异步。
- record、class、struct、interface 的选择以语义和演进稳定性为先。
- 扩展方法、静态工具和全局帮助类使用受控，避免污染核心领域代码。

## 配置与框架边界要求

- Options、配置绑定、配置校验和环境差异处理应集中在宿主初始化或明确边界层，不在业务方法里零散读取。
- 不把框架上下文、容器解析、HttpContext 或 IConfiguration 直接传入核心业务类型。
- 中间件、过滤器、管道行为和宿主扩展方法应承载横切关注点，不反向吸收业务规则。

## 异步与资源管理要求

- 异步方法应只覆盖真实 I/O、等待边界或并发场景，不制造伪异步链路。
- CancellationToken、超时、重试和资源释放策略应在关键边界统一处理。
- IDisposable、IAsyncDisposable 和 using 范围要清晰，避免资源释放依赖隐式约定。
- 不用 fire-and-forget 掩盖失败路径；如必须异步派发，应保留观测与补偿机制。

## 常见风险信号

- 为了可测试性给几乎所有类都加接口，但没有真实抽象边界。
- 扩展方法和静态工具持续吸收业务逻辑，导致依赖方向难以判断。
- 同一仓库中 async/await、Task、异常传播和资源释放风格混乱。
- 命名空间和项目边界与业务结构脱节，只剩技术层堆叠。
- 关键契约大量退化成 object、Dictionary 或弱类型配置，说明类型语义没有真正收敛。

## 验收清单

- 是否存在无意义接口、难以替换的静态帮助类或滥用扩展方法。
- 是否统一了异步、异常和资源释放方式。
- 是否让命名空间和项目边界能直接映射业务结构。
- 是否明确了 null、空集合、配置模型和公开契约的类型语义。

## 输出模板建议

- 结构边界：说明项目、命名空间、类型分层和依赖方向。
- 类型策略：说明 interface、record、class、struct 等选择依据。
- 运行语义：说明异步、异常、资源释放和配置接入方式。
- 配置边界：说明 Options、宿主初始化和框架上下文的隔离方式。
- 风险与限制：说明当前静态工具、扩展方法或接口抽象的治理点。

## 不建议的做法

- 为了“以后可能替换”给所有类先套一层接口。
- 把扩展方法和静态帮助类当成默认承载业务逻辑的地方。
- 用 Task.Run 或伪异步包装隐藏同步阻塞问题。
- 让配置读取、服务定位和资源释放散落在业务实现细节中。
