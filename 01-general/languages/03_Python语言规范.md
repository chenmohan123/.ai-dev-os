# Python语言规范

## 适用范围

- 适用于 Python 服务、脚本、任务处理程序和通用模块的实现层规范。

## 目标

- 让 Python 代码在模块组织、数据表达、异常边界和运行行为上保持清晰与可预测。
- 避免因动态特性使用失控、脚本式演进和导入期副作用，把实现层变成难测试、难排查的隐式结构。

## 基本要求

- 模块组织要清晰，避免单文件堆积路由、业务、数据访问和脚本入口。
- 变量、函数和模块命名优先可读性，不用过度缩写和隐式约定表达业务语义。
- 类型提示、数据模型和异常边界尽量明确，减少运行期靠猜测拼装数据结构。
- 导入、副作用和初始化逻辑保持可预测，不在模块导入阶段执行重操作。

## 数据与类型要求

- 关键输入输出优先使用 dataclass、TypedDict、Pydantic 模型或等价明确结构表达，而不是多层 dict 透传。
- 类型提示优先覆盖公共接口、核心服务边界和复杂返回值，避免只在局部零散补充。
- None、空列表、空字符串和缺失字段语义应区分清楚，不用单一空值表达多种状态。
- 魔法键名、字符串状态值和隐式字段约定应集中定义，而不是散落在多个模块里。

## 实现约束

- 函数应保持单一职责，避免长函数通过布尔参数切换多套流程。
- 数据结构优先使用明确模型或命名结构，而不是层层嵌套字典透传。
- 日志、异常和重试不应散落在每个调用点临时拼接。
- 脚本入口用明确的 main 或命令入口收敛，不在模块顶层直接运行主流程。

## 模块与运行要求

- 模块导入应轻量、可预测，数据库连接、网络请求和配置加载集中在明确初始化阶段。
- 公共工具模块只承载稳定通用能力，不把业务判断长期塞进 utils、helpers 或 common。
- 长任务、批处理和 CLI 入口应显式记录输入参数、执行进度、失败样本和退出结果。
- 异常边界应能区分业务错误、依赖错误和系统错误，不用裸 except 吞掉上下文。

## 常见风险信号

- 关键业务流依赖多层 dict 读写和字符串键名约定，说明结构表达不足。
- 导入模块就触发连接初始化、配置重载或主流程执行，说明运行边界失控。
- 大量逻辑散落在 utils、helpers 或脚本顶层，缺少稳定模块边界。
- 通过 monkey patch、动态属性或隐式全局状态掩盖本应显式设计的依赖。

## 验收清单

- 是否存在导入期副作用、脚本式全局状态或长函数堆叠。
- 是否让类型、数据结构和异常边界保持清晰。
- 是否避免用动态技巧掩盖本应显式表达的结构。

## 输出模板建议

- 模块结构：说明入口、服务、数据模型和基础依赖的落点。
- 数据表达：说明关键输入输出模型、类型提示和状态语义策略。
- 运行边界：说明初始化、异常处理、日志和任务执行方式。
- 风险与限制：说明当前动态特性使用、历史脚本残留和待治理点。

## 不建议的做法

- 用多层嵌套 dict 和隐式键名长期承载核心业务模型。
- 在 import 阶段执行连接初始化、参数解析或主任务。
- 用裸 except、print 调试和全局变量串联正式业务流程。
- 用动态补丁或隐式属性访问替代稳定的数据结构设计。
