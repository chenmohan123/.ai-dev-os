# 通用后端规范

## 适用范围

- 适用于 API 服务、管理后台后端、任务处理服务、消息消费服务和第三方集成适配器。

## 目标

- 让后端服务在分层边界、接口契约、依赖治理和运行治理上保持一致约束。
- 避免随着业务增长，把协议处理、业务编排、数据访问、外部集成和环境配置混成不可维护的大泥团。

## 基本要求

- 服务边界应围绕业务能力划分，禁止把控制器、领域逻辑、数据访问和集成逻辑混在同一层。
- 接口契约应稳定、可读、可审计，禁止把内部对象直接透出为外部响应结构。
- 配置、密钥、连接和环境差异必须外置管理，禁止依赖硬编码环境分支。
- 错误模型需要统一，外部响应只返回必要信息，内部细节写入日志与监控。
- 可观测性默认具备，请求日志、关键指标、异常追踪和健康检查应随服务一起设计。

## 服务边界要求

- 服务划分优先围绕业务能力和职责边界，而不是围绕数据库表或框架目录堆叠。
- API 服务、异步任务、消息消费和集成适配器可以部署形态不同，但应遵循一致的分层和治理规则。
- 共享能力只有在语义稳定、跨多个服务复用且边界清晰时才抽取，避免过早建设模糊的公共层。

## 设计约束

- 控制器只负责协议适配、参数校验和结果转换，不承载复杂业务编排。
- 业务逻辑应收敛在明确的服务层或用例层，避免跨层直接互调。
- 数据访问层只负责持久化交互，不混入业务决策和接口协议。
- 对外部系统的调用应通过适配器或客户端封装，避免 SDK 侵入业务层。

## 接口契约与错误模型要求

- 请求模型、响应模型、持久化模型和内部领域对象应分层使用，不用同一对象贯穿全链路。
- 接口字段、错误码、分页结构和筛选排序能力应有稳定约定，避免每个接口各自定义风格。
- 对外错误响应只暴露必要信息，堆栈、SQL、组件细节和第三方原始报错进入日志系统。
- 对重试、幂等、并发冲突、部分失败和超时应定义明确处理方式，而不是留给调用方猜测。

## 配置与依赖治理要求

- 配置、密钥、连接串、第三方地址和环境差异统一外置管理，并在启动阶段完成校验。
- 外部依赖调用应统一定义超时、重试、熔断、降级和失败告警策略。
- 数据库、缓存、消息、对象存储和第三方服务等依赖应在边界层统一接入，避免业务层直接散落 SDK 代码。
- 本地开发、测试、预发和生产环境的差异应靠配置治理解决，而不是靠代码分支硬切换。

## 运行与可观测性要求

- 请求入口、关键业务节点、外部依赖调用和失败路径应具备可追踪日志。
- 至少为成功率、错误率、耗时、吞吐和关键资源使用情况建立基础指标。
- 健康检查应区分进程存活、依赖可用和核心链路可用，避免只有一个“服务启动成功”。
- 异步任务、消息消费和批处理服务也应具备和 API 服务一致的日志、指标和告警要求。

## 常见风险信号

- Controller 或 Handler 同时处理协议转换、业务决策、事务、外部调用和持久化。
- 接口直接返回数据库实体、ORM 对象或第三方 SDK 原始结构。
- 配置项、密钥和环境分支散落在业务代码中，靠 if else 切换环境行为。
- 错误处理完全依赖 try catch 临时兜底，没有统一错误码、异常映射和日志策略。
- 异步任务、消息消费或定时任务脱离统一工程治理，被当成“脚本”独立堆放。

## 验收清单

- 是否清楚定义了接口边界、错误模型和依赖方向。
- 是否避免了硬编码配置、共享可变状态和跨层直接访问。
- 是否为关键链路补齐了日志、指标和健康检查。

## 输出模板建议

- 服务边界：说明接口层、应用层、领域逻辑、数据访问和外部集成的职责边界。
- 接口契约：说明请求响应模型、错误模型、分页筛选和幂等约定。
- 关键依赖：说明数据库、缓存、消息和第三方服务的接入及治理方式。
- 运行治理：说明日志、指标、追踪、健康检查和告警策略。
- 风险与限制：说明当前遗留结构、兼容负担和后续治理项。

## 不建议的做法

- 用一个控制器、服务类或工具类承接大部分业务编排和依赖调用。
- 直接把数据库实体、ORM 模型或第三方返回结构暴露给接口消费者。
- 在业务代码中散落读取配置、判断环境和拼接外部调用地址。
- 只给 API 服务做治理，而忽略消息消费、定时任务和批处理服务的一致性要求。
