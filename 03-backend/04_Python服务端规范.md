# Python服务端规范

## 适用范围

- 适用于 Python API 服务、任务处理程序、自动化服务和轻量集成适配器。

## 目标

- 让 Python 服务从脚本式实现收敛为可维护、可测试、可观测的工程化结构。
- 避免入口函数、模块导入、副作用初始化和字典透传逐步演变成难以排查的隐式耦合。

## 基本要求

- 项目结构应区分入口层、业务层、数据访问层和基础设施层，避免脚本式堆叠。
- 配置、密钥、连接和运行模式统一由配置层管理，不在模块导入阶段执行副作用逻辑。
- 数据校验、序列化和业务规则分离，避免字典透传和隐式字段约定。
- 日志、异常和重试策略要统一，不能依靠 print 或局部 try/except 拼接流程。

## 项目组织要求

- 路由、CLI 入口、任务入口和调度入口应各自清楚，但共享同一套业务层和基础设施抽象。
- 配置对象、客户端适配器、仓储层和服务层应有稳定落点，不在 utils 或 scripts 中长期混放。
- 一次性脚本与正式服务代码分开管理，不让临时脚本反向侵入正式模块结构。

## 运行与依赖要求

- 外部 API、数据库、缓存和消息客户端应集中封装，统一超时、重试、异常转换和连接管理。
- 模块导入应保持轻量和可预测，不在 import 时执行数据库建连、网络请求、任务调度或配置重载。
- 长任务、批处理和消费程序应显式记录进度、失败样本、重试情况和最终结果。
- 参数解析、数据校验、序列化和业务编排分开实现，不用裸 dict 在多层之间长期透传。

## 实现约束

- 路由或入口函数只做参数接收、校验和结果封装。
- 业务逻辑放在明确的 service 或 use case 层，避免散落在路由、脚本 main 或工具模块。
- 外部依赖调用要封装成客户端或适配器，统一超时、重试和错误转换。
- 模块初始化应可预测，不在 import 时执行数据库连接、网络请求或重配置。

## 常见风险信号

- 入口文件既做参数解析，又做业务编排、数据访问和外部调用。
- 多个模块依赖共享的全局变量、全局 client 或运行时动态注入状态。
- 为了方便开发，在 import 阶段直接初始化连接池、读取远程配置或触发任务。
- 大量业务规则散落在 utils、helpers 或脚本 main 函数中，缺少稳定服务层。

## 验收清单

- 是否存在脚本式全局状态、循环依赖或导入期副作用。
- 是否把数据访问、外部调用和业务编排隔离清楚。
- 是否为关键入口补齐了参数校验、日志和异常处理。

## 输出模板建议

- 项目结构：说明入口层、业务层、数据访问层和基础设施层的落点。
- 运行方式：说明 API、任务、批处理或集成适配器的启动和调度方式。
- 关键依赖：说明配置、数据库、缓存、消息和第三方客户端的管理方式。
- 风险与限制：说明导入期副作用、历史脚本残留和后续治理计划。

## 不建议的做法

- 把正式服务长期写成单文件脚本，靠注释分区代替结构分层。
- 在 import 阶段做连接初始化、远程调用或环境切换。
- 用 print、裸 except 或全局变量串起核心流程。
- 让一次性脚本直接复用并修改正式服务内部模块，却不纳入工程治理。
