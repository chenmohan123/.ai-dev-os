# Java服务端规范

## 适用范围

- 适用于 Java API 服务、后台管理服务、消息处理服务和定时任务服务。

## 目标

- 让 Java 服务在工程结构、分层职责、运行治理和外部依赖处理上保持一致约束。
- 避免随着业务增长，把控制器、实体、工具类和配置逻辑堆成难维护的大泥团。

## 基本要求

- 工程结构应清晰区分启动层、接口层、应用层、领域层和基础设施层。
- 依赖注入、配置绑定、事务和持久化边界应集中定义，避免散落在控制器中。
- DTO、命令对象、查询对象和持久化对象分层使用，不混用同一模型贯穿全链路。
- 异常处理、日志规范和统一响应应在框架层集中处理。

## 工程组织要求

- 包结构优先围绕业务域和分层边界组织，不按个人习惯随意拆散。
- 启动类、配置类、接口层、应用服务、领域对象和基础设施适配器应有稳定落点。
- 公共组件只有在语义稳定、被多个业务复用时再抽到共享层，避免过早堆入 common。

## 接口契约要求

- Controller、Facade、RPC 入口和消息消费入口的请求模型、响应模型、命令对象和查询对象应分层表达，不混用 Entity 或持久化对象直出。
- 统一响应、分页结构、错误码和校验失败结果应有稳定约定，不让每个接口各自拼装风格。
- 对外契约优先使用显式类型和稳定字段，不长期依赖 Map、Object 或弱类型扩展字段透传。

## 运行与依赖要求

- 外部调用应统一超时、重试、熔断或降级策略，不在业务代码里零散手写。
- 事务边界应收敛在应用层或明确的业务边界，不在控制器和工具类中随意开启。
- 配置项通过集中绑定和校验接入，不在各处直接读取系统属性或环境变量。
- 消息消费、异步任务和定时任务应与 API 服务复用同样的日志、异常和可观测性要求。

## 启动与框架边界要求

- Spring Boot 启动类、自动配置接入、配置绑定和 Bean 装配应集中在宿主或基础设施边界，不反向侵入核心业务模型。
- Servlet、Filter、Interceptor、AOP 和框架上下文应用于横切关注点，不把核心业务规则长期埋入框架生命周期。
- 数据库、缓存、消息和第三方客户端的接入配置应集中组织，避免在业务方法里直接 new 或零散读取上下文。 

## 实现约束

- 控制器不直接访问 Repository 或 Mapper。
- Service 不应感知 HTTP、Servlet、网关上下文等协议细节。
- 公共组件按稳定性与复用价值抽取，避免把业务代码过早下沉到 common。
- 异步任务、消息消费和定时任务也应复用相同的分层与配置治理原则。

## 多入口一致性要求

- API、消息消费、定时任务和后台批处理虽入口不同，但应共享一致的应用层、异常映射、日志和监控策略。
- 不同入口对同一业务规则应复用同一主实现，而不是在 Controller、Consumer 和 Job 中各写一份近似逻辑。
- 幂等、重试、失败补偿和审计留痕应按入口类型明确定义，不靠调用方自行猜测。 

## 常见风险信号

- Controller 同时处理参数转换、事务、外部调用和持久化，说明职责严重混杂。
- Entity 被直接当成请求模型、响应模型和业务模型复用，说明分层失效。
- common 包持续吸收业务逻辑，导致公共层失去边界。
- 为了赶进度在静态工具类里堆业务规则、上下文读取和外部依赖调用。
- 同一业务流程在 API、消息消费和定时任务中分别复制一套实现，说明主实现没有收敛。

## 验收清单

- 是否存在胖 Controller、胖 Entity 或胖工具类。
- 是否把事务、重试、幂等和异常映射放在合适层级。
- 是否对外部依赖调用建立了超时、重试和失败监控。
- 是否明确了接口契约、启动边界和多入口复用方式。

## 输出模板建议

- 工程结构：说明启动层、接口层、应用层、领域层和基础设施层的落点。
- 契约设计：说明请求响应模型、错误码和校验失败结果的统一方式。
- 关键依赖：说明数据库、缓存、消息、第三方接口和配置接入方式。
- 运行约束：说明事务、重试、幂等、异常处理和监控策略。
- 风险与限制：说明当前遗留结构、兼容负担和后续治理点。

## 不建议的做法

- 用一个 Service 或 Util 承接大部分业务，靠方法名区分职责。
- 在 Controller 中直接拼装 Repository 查询和事务逻辑。
- 在代码中散落读取配置、切换环境分支和手写异常响应。
- 将消息消费和定时任务当成“脚本”处理，不纳入统一工程治理。
