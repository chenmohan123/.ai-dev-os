# 目录索引

## 目标

- 为后端目录提供统一入口，先说明服务端规范解决什么问题、什么时候进入本目录、如何与数据库、安全、测试和交付目录配合。
- 避免把后端规范只理解为某种语言框架约定，而忽略通用服务边界、接口契约和运行治理约束。

## 使用说明

- 进入服务端规范目录后，默认先阅读本文件，再按技术栈或任务类型进入对应分册。
- 如任务属于服务划分、接口边界、异常处理、配置治理或可观测性，优先先读 [01_通用后端规范](01_通用后端规范.md)。
- 如任务明确为 Java、.NET 或 Python 服务，再进入对应语言分册。
- 如任务跨接口、安全、数据库或发布流程，先读本目录对应分册，再补读安全、数据库和 DevOps 目录。

## 什么时候先读本页

- 还不确定当前改动属于服务边界问题、语言实现问题，还是数据库、安全、发布协同问题。
- 需要先判断接口开发、服务重构、任务处理、消息消费或第三方集成该从哪篇后端规范进入。
- 当前任务虽然是后端改动，但仍不清楚后续应补读安全、数据库还是测试目录。

## 按任务类型快速入口

- 做服务边界、分层设计、接口风格、错误模型或配置治理：先读 [01_通用后端规范](01_通用后端规范.md)。
- 做 Java 服务开发、目录组织或工程约束：先读 [02_Java服务端规范](02_Java服务端规范.md)。
- 做 .NET Web API、后台服务或工程治理：先读 [03_.NET服务端规范](03_.NET服务端规范.md)。
- 做 Python API、脚本型服务或任务处理：先读 [04_Python服务端规范](04_Python服务端规范.md)。

## 与其它目录的配合方式

- 后端目录负责回答“服务怎么分层、接口怎么定义、依赖怎么治理、运行怎么观测”。
- 涉及输入校验、鉴权、密钥和审计时，补读 [../07-security/00_README.md](../07-security/00_README.md)。
- 涉及库表设计、索引优化和迁移发布时，补读 [../04-database/00_README.md](../04-database/00_README.md)。
- 涉及测试设计、回归范围和发布门禁时，补读 [../05-tests/00_README.md](../05-tests/00_README.md) 和 [../06-devops/00_README.md](../06-devops/00_README.md)。
- 通用编码、命名和评审口径仍以 [../01-general/00_README.md](../01-general/00_README.md) 为基础。

## 服务端分册

- [01_通用后端规范](01_通用后端规范.md)：服务边界、接口设计、异常模型、配置治理和可观测性的通用要求。
- [02_Java服务端规范](02_Java服务端规范.md)：Java 服务端项目的分层、工程结构、依赖管理和交付要求。
- [03_.NET服务端规范](03_.NET服务端规范.md)：.NET 服务端项目的目录组织、接口实现、配置与依赖约束。
- [04_Python服务端规范](04_Python服务端规范.md)：Python 服务、任务脚本和 API 工程的结构、依赖和运行约束。

## 阅读顺序建议

- 先做通用边界判断时：先读 [01_通用后端规范](01_通用后端规范.md)。
- 已明确语言栈时：读通用分册后直接进入对应语言分册。
- 涉及数据落库、索引或迁移时：补读 [../04-database/00_README.md](../04-database/00_README.md)。
- 涉及鉴权、密钥、输入校验或审计时：补读 [../07-security/00_README.md](../07-security/00_README.md)。

## 不建议的做法

- 还没判断服务边界和接口契约，就直接按语言或框架习惯开始堆实现。
- 把数据库、安全、测试和发布问题都挤在后端分册里解决，不回到对应专题目录。
- 只看语言分册，不先经过通用后端规范确认服务边界、错误模型和依赖方向。
