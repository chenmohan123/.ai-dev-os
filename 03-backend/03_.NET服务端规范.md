# .NET服务端规范

## 适用范围

- 适用于 .NET Web API、后台服务、Worker、消息消费程序和第三方集成服务。

## 目标

- 让 .NET 服务在目录组织、分层职责、宿主初始化和运行治理上保持一致约束。
- 避免随着业务叠加，把 Controller、Endpoint、Options、任务逻辑和基础设施适配混成难维护的耦合结构。

## 基本要求

- 目录结构应明确区分 API、Application、Domain、Infrastructure 等职责层。
- 依赖注入注册、配置绑定、中间件和宿主初始化应集中放在启动入口附近。
- 请求模型、响应模型和持久化模型分离，避免直接暴露实体对象。
- 统一处理验证失败、业务异常和系统异常，避免每个接口手工拼装错误响应。

## 工程组织要求

- API、Application、Domain、Infrastructure 和 Hosting 相关代码应有稳定落点，不按个人习惯分散放置。
- 控制器、最小 API Endpoint、后台任务和消费者虽入口不同，但应共享一致的应用层和基础设施治理方式。
- 通用组件只有在复用稳定、职责清晰时再提到共享层，避免把业务逻辑沉淀到 Common 或 Extensions 杂物目录。

## 运行与依赖要求

- Options、配置绑定、配置校验和环境差异处理应集中管理，不在业务代码里零散读取配置。
- 中间件、过滤器、管道行为和异常映射应统一处理，不在每个接口重复拼装。
- 外部调用应统一超时、重试、熔断或等价失败治理策略，并保留必要观测信息。
- Worker、消息消费者和定时任务应与 API 服务复用同样的日志、监控、幂等和失败处理规则。

## 实现约束

- Controller 或 Endpoint 只负责协议适配，不直接承载复杂业务与数据访问。
- Application 层组织用例，Domain 层表达业务规则，Infrastructure 层承接外部依赖。
- 配置项按 Options 或等价方式集中管理，不在业务代码中直接读环境变量。
- 后台任务、队列消费者和定时任务需复用同样的日志、监控和失败处理规则。

## 常见风险信号

- Controller、Endpoint 或 Handler 中同时出现参数转换、事务、仓储调用和外部接口编排。
- Domain 层开始依赖配置、日志框架、HTTP 上下文或基础设施细节，说明边界反转。
- Common、Shared 或 Extensions 目录持续吸收不相关业务代码，说明共享层失控。
- 为了快速落地，把 Options 读取、环境判断和容器解析散落到多个业务模块。

## 验收清单

- 是否存在跨层直接引用和职责混杂。
- 是否明确了验证、异常、中间件和依赖注入边界。
- 是否为关键任务链路补齐了幂等、重试和可观测性。

## 输出模板建议

- 工程结构：说明 API、Application、Domain、Infrastructure 和宿主初始化的落点。
- 关键依赖：说明数据库、缓存、消息、第三方接口和配置接入方式。
- 运行约束：说明 Options、异常处理中间件、重试、幂等和监控策略。
- 风险与限制：说明历史结构债务、共享层污染或待治理点。

## 不建议的做法

- 让 Controller 或 Endpoint 直接依赖 DbContext、Repository 和第三方客户端完成整条流程。
- 在业务模块中散落读取 IConfiguration、Environment 或容器实例。
- 把后台任务和消费者当成独立脚本处理，不纳入统一宿主和可观测性治理。
- 用 Shared、Common 兜底一切复用诉求，最终变成难以维护的杂物层。
